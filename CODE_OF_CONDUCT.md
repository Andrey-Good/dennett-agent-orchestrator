# Code Of Conduct

## Expected Behavior

Contributors, maintainers, and users are expected to:

- communicate respectfully and professionally;
- focus criticism on ideas, code, documentation, and evidence;
- assume public issues and pull requests may be read by future contributors;
- avoid posting secrets, credentials, private prompts, private memory records, proprietary outputs, or unredacted logs;
- respect the repository's current launch boundary and avoid presenting unapproved capabilities as ready.

## Unacceptable Behavior

The following behavior is not acceptable in this project:

- harassment, threats, insults, or personal attacks;
- discriminatory language or conduct;
- deliberate disruption of issues, pull requests, reviews, or discussions;
- publishing another person's private information without permission;
- posting exploit details, credentials, private data, or sensitive operational details in public channels;
- knowingly misrepresenting project readiness, support commitments, or security status.

## Reporting

For conduct concerns, contact the maintainers through the repository's normal private maintainer contact path when available. If no private path is available, open a minimal public issue requesting maintainer contact without naming people, publishing private details, or including sensitive evidence. If the concern includes a vulnerability, exploit detail, credential, private prompt, memory record, provider configuration, or other sensitive data, follow [SECURITY.md](./SECURITY.md) instead of opening a public issue.

Maintainers should handle reports with discretion, minimize unnecessary sharing of private information, and take proportionate action. Possible actions include clarifying expectations, moderating public threads, closing or locking issues, removing sensitive content where platform tools allow it, or limiting participation for repeated or severe violations.

## Scope

This code of conduct applies to repository-hosted project spaces, including issues, pull requests, discussions, reviews, and documentation contributions. It does not create a hosted service support commitment, SLA, incident-response promise, or public launch approval.
