## Полная логика долговременной, проектной, агентной, персонализированной, федеративной и мультимодальной памяти

**Версия:** 1.2

**Статус:** прагматическая совместимая ревизия версии 1.1 и документ бизнес-логики и системных инвариантов. # 0. Итоговый вердикт

## 0.1. Что исправляет версия 1.2

Версия 1.2 сохраняет доказательное и событийное ядро Memory Fabric 1.1, но отменяет ошибочную презумпцию, что смысловая память должна постепенно превращаться в постоянно расширяющуюся систему строгих типов и миграций.

Повторная проверка показала, что критика этой части была в основном справедливой:

1. строгие структуры действительно необходимы для операционного состояния, прав, идентичности, времени, provenance, удаления, задач, событий и других данных, где ошибка поля способна изменить действие системы;
2. для вкусов, социальных наблюдений, опыта, идей, исследовательских выводов и большинства знаний о пользователе строгая DDL-подобная схема часто не даёт достаточного выигрыша, чтобы оправдать taxonomy churn, backfill и миграции;
3. современные сильные модели хорошо работают с evidence-grounded Markdown, topic documents, claims и semi-structured facets, если retrieval возвращает правильный контекст и сохраняет источники;
4. полностью свободный текст всё же недостаточен там, где нужны точный current state, отрицательные запросы, агрегация, автоматическое действие или проверяемый контракт;
5. поэтому правильна не крайность «всё типизировать» и не крайность «всё оставить заметками», а **выборочное структурирование только по доказанной операционной необходимости**.

Версия 1.2 вводит принцип **Selective Semantic Structuring**:

> **Канонический control envelope всегда строг. Смысловая память по умолчанию остаётся открытой, evidence-grounded и человекочитаемой. Типизированная semantic projection создаётся только тогда, когда конкретный запрос, действие или повторяющаяся ошибка действительно требуют точной структуры.**

Это означает:

- неожиданный смысл всегда можно сохранить как Open Observation или Human Note;
- topic dossiers и claims остаются основным semantic layer;
- гибкие namespaced facets допускаются без создания нового глобального типа;
- LLM может извлекать временную структуру под конкретный запрос;
- reusable semantic view создаётся только после повторяющейся практической потребности;
- глобальная миграция старой памяти не является default;
- backfill выполняется лениво или только для hot scope;
- структура не получает новых прав влияния автоматически;
- при отсутствии измеримого выигрыша система остаётся на freeform/topic-document представлении.

## 0.2. Что остаётся неизменным

Не изменяются фундаментальные решения 1.0–1.1:

- Event Ledger и Evidence Objects являются каноническим основанием;
- history отделена от current state;
- valid time отделено от transaction time;
- semantic claims обязаны сохранять provenance;
- Causal Trace фиксирует опыт действий;
- Prospective Intent хранит будущие условия;
- retrieval остаётся query-planned и multi-lane;
- similarity не равна допустимости влияния;
- Memory Spaces остаются федеративными;
- Portable Project Memory Pack и Instruction Interoperability сохраняются;
- любые derived representations пересобираемы;
- удаление, poisoning protection и evaluation встроены в lifecycle.

## 0.3. Обновлённая формула

> **Dennett Memory Fabric 1.2 = Evidence Objects + Event Ledger + Open Semantic Memory + Selective Semantic Structuring + Topic Dossiers + Temporal/Current State + User Alignment Models + Causal Experience + Prospective Memory + Federated Memory Spaces + Portable Project Memory + Poly-Retrieval + Trust/Influence Gate + Indexed Working Context + Utility-Driven Consolidation + Continuous Evaluation.**

Главное изменение касается только semantic layer:

- строгие operational records существуют там, где системе надо надёжно действовать;
- человекоподобное знание о мире и пользователе хранится прежде всего как evidence, claims, episodes, topic documents и свободные observations;
- reusable typed view является оптимизацией для конкретного workload, а не стадией «созревания» любой памяти;
- архитектура обязана доказывать пользу структуры end-task метриками, а не эстетикой онтологии.

## 0.4. Практическое правило

Перед созданием нового semantic type система должна задать пять вопросов:

1. Какой реальный запрос или действие сейчас невозможно надёжно выполнить через evidence + topic documents + facets?
2. Нужна ли точная агрегация, current-state resolution, validation или automation?
3. Повторяется ли эта потребность достаточно часто?
4. Будет ли структура дешевле и надёжнее on-demand извлечения?
5. Можно ли удалить эту projection без потери исходного знания?

Если ответы не подтверждают пользу, новый тип не создаётся.

# 1. Что память Dennett должна уметь

## 1.1. Основная цель

Память Dennett должна позволять системе годами накапливать контекст пользователя, проектов и собственной работы, не превращая этот объём в бесконечный prompt и не заставляя пользователя вручную обслуживать архив.

Практически «неограниченная память» означает не бесконечное внимание модели, а следующее:

1. логическое хранилище может расти настолько долго, насколько позволяет доступное физическое хранение;
2. стоимость обычного запроса не должна линейно зависеть от общего объёма памяти;
3. рабочий контекст агента всегда ограничен и собирается под конкретную задачу;
4. любой важный вывод можно проследить до исходного доказательства;
5. старые данные могут переходить в более дешёвые уровни хранения, не исчезая из логического пространства поиска;
6. индексы и summaries могут заменяться без потери первичного знания;
7. система должна уметь признавать, что доказательств недостаточно, вместо уверенной подстановки похожего фрагмента.

Исследования длинного контекста показывают, что простое расширение окна не решает задачу: модели хуже используют информацию в середине больших контекстов, а дополнительный нерелевантный материал способен снижать качество. Поэтому долговременная память должна быть системой отбора и реконструкции, а не складом, который целиком подставляется модели. [[S35]]

## 1.2. Классы задач памяти

Память должна одинаково серьёзно обслуживать не только разговоры, но и следующие классы работы:

- многомесячные кодовые проекты;
- интерактивные project chats в логике Codex/Claude Code;
- автономные поручения оркестратору;
- multi-agent и параллельные ветки;
- исследования с источниками и меняющимися гипотезами;
- дизайн с визуальными референсами и пользовательскими реакциями;
- computer-use и web navigation;
- встречи, голосовые обсуждения и фоновые разговоры;
- постоянное наблюдение экрана и событий устройств;
- социальный контекст и обещания;
- персональные предпочтения и их изменения;
- навыки, ошибки и опыт агентов;
- будущие условия: «когда произойдёт X, вспомнить и сделать Y»;
- точное удаление, секретность и разграничение доступа.

Свежие benchmark-наборы показывают, что текущие memory-системы особенно слабы вне обычного chat recall: на реальных project-oriented сценариях, длинных agent trajectories, инструментальных действиях, coding и computer-use. [[S01]] [[S02]] [[S03]] [[S04]]

## 1.3. Не-цели

Память Dennett не должна:

- имитировать человеческий мозг ради самой метафоры;
- хранить каждый внутренний chain-of-thought агента;
- считать любой summary правдой;
- доверять любому найденному воспоминанию;
- заставлять пользователя одобрять каждую запись;
- создавать отдельную жёсткую подсистему под каждый частный тип знания;
- превращать всё в knowledge graph;
- превращать всё в embeddings;
- пытаться удерживать всю историю в контексте модели;
- удалять исторические данные только из-за возраста;
- автоматически улучшать собственные правила без eval, версионирования и отката.

## 1.4. Дополнительные обязательные сценарии версии 1.1

Память обязана обслуживать следующие end-to-end сценарии, а не только хранить относящиеся к ним факты:

- построить ответ на входящее сообщение с учётом отношений, текущего предметного контекста, прошлой манеры общения пользователя, допустимого раскрытия информации и актуального состояния проекта;
- отдельно решить, следует ли сообщение игнорировать, сохранить, показать пользователю, превратить в черновик или отправить автономно;
- при обнаружении незапрошенного события определить, хочет ли пользователь вообще вмешательства, и рассмотреть бездействие, наблюдение, откладывание и подготовку как полноценные альтернативы действию;
- хранить новости, статьи, tweets, release notes, продукты, методы и инструменты так, чтобы отличать сигнал от подтверждённого знания и перепроверять быстро меняющиеся свойства перед применением;
- вести изолированное исследование, не позволяя его гипотезам преждевременно загрязнять глобальное знание;
- передать репозиторий другому пользователю вместе с полезной памятью проекта, но без личной памяти, секретов, runtime-кэшей и машинно-зависимых индексов;
- открыть чужой репозиторий с несколькими `AGENTS.md`, `CLAUDE.md`, `MEMORY.md` и rules-файлами, корректно воспроизвести нативную для каждого провайдера область и приоритет инструкций, ничего не удаляя;
- после изменения таких файлов в новом commit обновить эффективную память проекта, а при обратном экспорте не затереть изменения человека или другого инструмента;
- сохранить неожиданное наблюдение, даже если разработчики Dennett заранее не предусмотрели соответствующий тип памяти.

## 1.5. Дополнительные не-цели версии 1.1

Память не должна:

- выбирать между полностью фиксированной онтологией и бесформенной кучей текста;
- создавать новый жёсткий тип данных для каждого нового интереса пользователя;
- позволять свободно сгенерированному типу сразу влиять на разрешения, safety или автоматические действия;
- считать сходство стиля достаточным основанием для отправки сообщения;
- превращать персонализацию в угождение, навязчивое упоминание личных фактов или подмену объективного вывода;
- хранить всю глобальную Memory Fabric внутри каждого Git-репозитория;
- использовать Git как хранилище embeddings, ANN-индексов, секретов и постоянно меняющихся runtime projections;
- считать `AGENTS.md` канонической памятью Dennett;
- слепо синхронизировать два текстовых файла в обе стороны без ownership, base version и conflict detection;
- удалять или переписывать чужие provider-native файлы только ради унификации.

# 2. Исследовательский протокол и критерии отказа

Этот раздел фиксирует метод, по которому получена схема. Он должен использоваться и при дальнейшем проектировании архитектуры: любое красивое решение обязано пройти проверку на реальных задачах Dennett.

## 2.1. Этап A: зафиксировать требования и угрозы

### Действия

- выделить пользовательские, проектные, агентные, сенсорные и системные сценарии;
- разделить retrospective memory, prospective memory, working memory и procedural experience;
- определить, где память только помогает ответить, а где влияет на реальное действие;
- определить виды источников, доверия, секретности, прав и удаления;
- определить ожидаемые масштабы: количество событий, длительность использования, объём media, число проектов и параллельных агентов.

### Критерий принятия этапа

Для каждого важного сценария должен существовать как минимум один end-to-end тест: от записи события до ответа или действия.

### Критерий отмены/возврата

Если очередной компонент памяти нельзя связать ни с одним пользовательским сценарием, benchmark или системным инвариантом, он не добавляется.

## 2.2. Этап B: построить карту benchmark-компетенций

Память оценивается не одним общим score, а минимум по следующим компетенциям:

- точное извлечение;
- temporal reasoning;
- knowledge updates;
- abstention;
- multi-session и multi-hop;
- project state tracking;
- causality;
- использование памяти в tool calls;
- опыт среды и recurring gotchas;
- cognitive/latent constraints;
- prospective triggers;
- multimodal memory;
- selective forgetting;
- access control;
- poisoning resistance;
- масштабирование и latency.

### Критерий принятия

Новая схема должна иметь понятный механизм для каждой компетенции и тест, который способен выявить регрессию.

### Критерий отмены

Если одна универсальная метрика маскирует провал отдельного класса, решение не считается доказанным.

## 2.3. Этап C: сравнить кандидаты на фундамент

Сравниваются:

- notes-only;
- append-only raw log;
- profile/fact store;
- vector-first memory;
- graph-first memory;
- topic-document memory;
- event-sourced evidence memory;
- neural/parametric memory;
- гибриды.

### Критерии

Фундамент должен обеспечивать:

- восстановимость;
- provenance;
- временную историю;
- параллельные записи;
- correction/rollback;
- ACL и deletion;
- мультимодальность;
- отсутствие зависимости от конкретной LLM;
- возможность полностью заменить индексы и semantic extractors.

### Решение

Единственный кандидат, удовлетворяющий всем критериям без превращения производных данных в необратимую истину, — event-sourced evidence layer. Notes, graph, embeddings и topic documents принимаются только как projections и retrieval structures.

## 2.4. Этап D: спроектировать write path

### Проверяемые вопросы

- что подтверждается пользователю до завершения тяжёлой обработки;
- как обеспечивается idempotency;
- что происходит при падении extractor, indexer или graph builder;
- как быстро новая запись становится доступной;
- как отделяется raw evidence от inference;
- как запись получает scope, trust, sensitivity и retention;
- как работают offline clients и повторная доставка.

### Критерий принятия

После успешного ответа `record_event` событие не должно потеряться даже при полном отказе всех derived-подсистем. Новая информация должна быть доступна через recent-event overlay до завершения индексации.

### Критерий отмены

Если запись признаётся успешной только после серии LLM-вызовов или зависит от успешного graph extraction, схема отклоняется.

## 2.5. Этап E: спроектировать retrieval

### Обязательные baselines

- full context, если corpus помещается;
- BM25/lexical only;
- dense only;
- hybrid lexical+dense;
- graph only;
- notes-only предыдущей версии;
- новый query-planned poly-retrieval.

### Критерий принятия отдельного retrieval lane

Lane сохраняется, только если:

- улучшает end-task quality или evidence recall на целевом домене;
- либо обеспечивает обязательное свойство, например exact lookup, temporal correctness или access control;
- его стоимость и latency оправданы;
- его отказ не уничтожает всю систему.

### Критерий отмены

Если lane дублирует другой без измеримого выигрыша, он удаляется. Если reranker снижает recall или выбрасывает важные кандидаты, он отключается для этого домена. Практические issue-трекеры Graphiti показывают, что подобные потери кандидатов реальны, поэтому reranking не может считаться безусловно полезным. [[S37]]

## 2.6. Этап F: проверить temporal, conflicts и concurrency

### Критерий принятия

Система обязана различать:

- когда событие произошло;
- когда система о нём узнала;
- что считалось текущим на выбранный момент;
- что считается текущим сейчас;
- какие источники спорят;
- является ли конфликт разрешённым или лишь скрытым.

### Критерий отмены

Любая схема с silent last-write-wins для значимых semantic claims отклоняется. Исключение — строго заданный predicate resolver, где порядок обновлений действительно определяет состояние.

## 2.7. Этап G: проверить безопасность по всему lifecycle

Проверка ведётся по фазам Write → Store → Retrieve → Execute → Share/Propagate → Forget/Rollback. [[S17]]

### Критерий принятия

- память не может сама повысить права;
- внешний текст не становится инструкцией;
- секреты не попадают в обычный context;
- удалённые данные не возвращаются из кэша, индекса или summary;
- cross-project и cross-principal leakage тестируются;
- происхождение любой влиятельной памяти прослеживается.

### Критерий отмены

Если безопасность добавляется только после retrieval или полагается на просьбу к LLM «игнорировать плохую память», схема отклоняется.

## 2.8. Этап H: проверить масштабирование

### Критерий принятия

Рост общего архива не должен линейно увеличивать:

- размер обычного контекста;
- online ingestion latency;
- стоимость стандартного запроса;
- время восстановления после сбоя отдельного индекса.

Все производные индексы должны иметь версию и возможность rebuild. При смене embedding-модели должна поддерживаться параллельная миграция без остановки памяти.

### Критерий отмены

Если заменить embedding-модель или graph schema можно только миграцией самой «истины», фундамент выбран неправильно.

## 2.9. Этап I: провести ablation и self-critique

Каждый крупный механизм должен иметь ablation:

- без graph;
- без topic dossiers;
- без reranker;
- без learned gate;
- без causal trace;
- без prospective layer;
- без late interaction;
- без deep consolidation;
- с notes-only baseline;
- с предыдущей v0.2.

Новая версия считается лучше только при выигрыше по заранее определённому набору качества, стоимости, безопасности или управляемости. Сложность сама по себе не является достоинством.

---
## 2.10. Этап J: исследовать степень semantic structuring

### Сравниваемые варианты

- полностью свободные evidence-grounded notes и topic documents;
- semi-structured claims с открытыми facets;
- on-demand extraction под конкретный запрос;
- reusable semantic views для повторяющихся запросов;
- строгие domain records для current state и автоматических действий;
- unrestricted dynamic semantic schema evolution как контрольный сложный вариант, который принимается только при доказанном выигрыше.

### Критерий принятия структуры

Структурирование принимается только если оно:

1. устраняет наблюдаемую ошибку retrieval, state resolution или action;
2. улучшает end-task quality, latency либо cost-of-success по сравнению с freeform baseline;
3. требуется неоднократно, а не для одного эпизода;
4. имеет чёткий scope и не заставляет заполнять неизвестные поля догадками;
5. может быть пересобрано из evidence;
6. не требует массовой миграции без доказанной пользы;
7. не получает control-plane authority автоматически.

### Критерий отказа

Если topic document, open facets или on-demand extraction дают сопоставимый результат дешевле, schema/view не создаётся. Если структура порождает пустые поля, taxonomy churn, ложную определённость или дорогой backfill, она отключается и остаётся только как экспериментальная projection.


## 2.11. Этап K: исследовать user-aligned communication и proactive action

### Проверяемые вопросы

- какие факты необходимы, чтобы сформировать ответ «как пользователь», а какие создают over-personalization;
- как разделить содержание ответа, стиль, решение об отправке и авторизацию;
- как учитывать preference drift, исключения и конкретного собеседника;
- как оценивать вмешательство, если пользователь заранее не создавал trigger;
- где должны находиться «ничего не делать», «понаблюдать» и «подготовить черновик»;
- когда нужна pre-action clarification, а когда она только раздражает;
- как превращать post-action feedback в обновление памяти без переобучения всей системы.

### Критерий принятия

Компонент принимается, если одновременно растут acceptance rate черновиков, action success и user satisfaction, а disclosure errors, unauthorized sends, over-personalization и false interventions не ухудшаются.

### Критерий отказа

Если персонализированная модель повышает sycophancy, навязчивость или риск действия, её влияние ограничивается `style/personalization/advisory`, а решения возвращаются к clarification или пользователю.

## 2.12. Этап L: исследовать переносимую проектную память

### Сравниваемые варианты

- только глобальная серверная память;
- полная копия Memory Fabric в Git;
- один `AGENTS.md`/`MEMORY.md`;
- обычная папка Markdown-заметок;
- portable event/evidence package с human views и внешними ссылками;
- стандартный packaging profile поверх нативного формата.

### Критерий принятия

После clone или передачи архива другой установке Dennett должны восстанавливаться project scope, история решений, переносимые evidence, исследования, инструкции и provenance без обязательного переноса личной памяти и без сохранённых embeddings/индексов.

### Критерий отказа

Если формат требует постоянно коммитить большие производные индексы, создаёт частые merge conflicts или не позволяет проверить утечку private data, он не используется как native project memory.

## 2.13. Этап M: исследовать Instruction Interoperability

### Проверяемые экосистемы

- `AGENTS.md` и `AGENTS.override.md`;
- `CLAUDE.md`, `CLAUDE.local.md`, imports и `.claude/rules/**`;
- `GEMINI.md` и настраиваемые context filenames;
- `.cursorrules`, `.cursor/rules/**`, `.windsurfrules`, `.devin/rules/**` и другие обнаруживаемые provider configs;
- `MEMORY.md`, `MEMORIES/` и произвольные project-local memory conventions.

### Критерий принятия

Dennett должен воспроизводить provider-native порядок, scope, lazy loading и override semantics; сохранять source file; отслеживать hash/version/owner; импортировать изменения; экспортировать только управляемые им logical records; предотвращать циклы и silent overwrites.

### Критерий отказа

Если полноценная двусторонняя синхронизация конкретного формата не может быть сделана без неоднозначных перезаписей, baseline для этого формата остаётся `import-only + suggested patch`.

# 3. Что показало повторное исследование

## 3.1. Современные benchmark-и опровергают chat-centric подход

Предыдущая версия исследования в основном опиралась на conversational memory и общие RAG-подходы. Этого недостаточно для Dennett.

В 2026 году появились benchmark-и, которые измеряют память в более реалистичных агентных условиях:

- **MemGym** объединяет tool-use dialogue, deep research, coding и computer-use и отдельно измеряет вклад самой памяти, чтобы не смешивать его со способностями модели. Авторы прямо отмечают, что chat-memory benchmark-и плохо переносятся на coding и web navigation. [[S01]]
- **RealMem** моделирует долгосрочные проекты с меняющимися целями и состоянием; существующие memory systems испытывают трудности именно с dynamic project context. [[S02]]
- **AMA-Bench** использует реальные и синтетические agent trajectories произвольной длины; его анализ связывает провалы с отсутствием причинности и потерями similarity retrieval. [[S04]]
- **Mem2ActBench** проверяет не ответ на вопрос, а способность применить прошлую память к выбору tools и их параметров. Текущие системы делают это плохо. [[S03]]
- **LongMINT** создаёт до 1.8 млн токенов interference-heavy histories и показывает особенно слабое reasoning по нескольким обновляющимся фактам. [[S05]]
- **LongMemEval-V2** проверяет опыт работы в специализированной среде: dynamic state, workflows, recurring failures и premise awareness на histories до 115 млн токенов. Сильный результат там достигается не простым top-k RAG, а активным gathering evidence coding-агентом, хотя это дороже. [[S38]]
- **TriggerBench** показывает, что spontaneous recall будущего условия существенно сложнее обычного вопроса о прошлом и обладает отдельным precision-recall trade-off. [[S08]]
- **GateMem** показывает, что retrieval-система может хорошо отвечать, но всё равно протекать между пользователями, ролями или после удаления. [[S09]]
- **Mem-Gallery** показывает, что мультимодальная память требует явного сохранения и организации visual evidence, а не только текстового описания. [[S10]]

Следствие: Dennett нельзя оптимизировать только на «нашёл ли нужную заметку». Главные единицы качества — корректное состояние проекта, причинно обоснованное действие, применение памяти к tool-use, своевременное срабатывание намерений, безопасное использование и способность показать доказательство.

## 3.2. Retrieval важнее красивой ingestion-магии

**MemMachine** сохраняет полные эпизоды и показывает в ablation, что улучшения retrieval depth, форматирования контекста, search prompt и query correction дали больше, чем изменения ingestion/chunking. Это важное предостережение: нельзя чрезмерно сжимать и «умно понимать» всё в момент записи, надеясь, что retrieval потом станет простым. [[S11]]

**Infini Memory** подтверждает полезность topic-structured documents, staging и iterative evidence inspection, но её topic documents лучше воспринимать как поддерживаемое представление, а не единственную истину. [[S12]]

**Memex(RL)** показывает сильную идею: в рабочем контексте можно держать compact summary и стабильные индексы, а полные взаимодействия сохранять снаружи и открывать только по необходимости. Это существенно менее lossy, чем summary-only. [[S13]]

**HiGMem** использует high-level event summaries как указатели на конкретные turns и извлекает меньше материала при хорошем качестве. [[S26]]

Следствие: запись должна сохранять достаточно исходной информации, а read path должен активно выбирать масштаб и глубину доказательств.

## 3.3. Текущая истина — это отдельная вычислительная задача

Temporal graph сам по себе не гарантирует, что система правильно выберет актуальное значение. Исследование deterministic conflict resolution показывает, что даже Graphiti/Zep и другие системы могут проигрывать простой операции `max(timestamp)` там, где задача действительно определяется временем обновления. При этом авторы отдельно предупреждают, что такой resolver не универсален и должен выбираться по типу вопроса. [[S15]]

**StateFuse** показывает более фундаментальное правило: конфликты полезнее сохранять в явном виде и разрешать при построении projection, чем необратимо схлопывать. Это не обязательно повышает обычную accuracy, но повышает способность безопасно abstain, исправлять и аудировать. [[S14]]

Следствие: Dennett должен хранить history и candidates отдельно от current-state projection. LLM не должна всякий раз заново «угадывать, что новее» из случайного набора chunks.

## 3.4. Память должна хранить причинность и опыт действий

AMA-Bench связывает слабость существующих систем с отсутствием causality и objective information. Простая пара «текст запроса — ответ» не позволяет понять, почему агент принял решение, какое действие реально сработало и при каких условиях. [[S04]]

Для Dennett это особенно важно: проектные агенты меняют файлы, запускают команды, сталкиваются с ошибками, получают feedback пользователя, переключаются в subagents и работают с внешними приложениями. Поэтому память должна сохранять не только «агент сделал X», а структурную цепочку:

`intent → goal → plan/decision → action/tool call → observation → artifact/state change → verification → outcome → user/system assessment`.

Именно из таких traces можно безопаснее строить procedural memory и понимать причины провалов.

## 3.5. Prospective memory — самостоятельный класс

Память Dennett должна не только отвечать на запросы о прошлом, но и сама вспоминать нужное условие в будущем. TriggerBench показывает, что это не «обычный retrieval по таймеру»: implicit и overloaded triggers резко снижают качество, а попытка повысить recall часто создаёт слишком много ложных напоминаний. [[S08]]

Следствие: future intents и semantic triggers должны иметь собственные сущности, budget внимания, стоимость false positive/false negative, cooldown, expiry и механизм верификации. Нельзя просто надеяться, что обычная заметка случайно появится в vector search в нужную секунду.

## 3.6. Мультимодальность требует общей временной оси

MIRIX, Mem-Gallery и современные multimodal memory approaches показывают, что хранить screenshot только как текстовое описание недостаточно. Нужно сохранять visual evidence, OCR, объекты, temporal context и связи с другими модальностями. [[S10]] [[S25]]

Для Dennett ключевой единицей должна стать не «фотография отдельно» и «аудио отдельно», а synchronized event: что было на экране, что звучало, какое приложение было активно, что делал пользователь, какой проект был открыт и какое действие последовало.

## 3.7. Learned memory managers перспективны, но не могут быть фундаментом первой версии

AgeMem, Fine-Mem, DeltaMem и Mem-π показывают, что memory operations можно обучать как действия: когда писать, что удалять, когда извлекать и какую guidance генерировать. [[S30]] [[S31]]

Но для Dennett нельзя начинать с полностью learned controller:

- нужен большой собственный набор trajectories;
- reward легко оптимизировать не туда;
- ошибка контроллера способна молча потерять память;
- трудно объяснить и откатить решение;
- безопасность и права не должны зависеть от policy model.

Следствие: первая сильная версия использует детерминированные инварианты и эвристически/LLM-управляемые предложения. Learned policies добавляются позднее в shadow mode и не получают власть над каноническими доказательствами, правами и физическим удалением.

## 3.8. Neural/parametric memory — cache, а не правда

MemOS предлагает единообразно работать с plaintext, activation и parametric memory. Это полезная системная абстракция. Однако neural memory связана с конкретной моделью, непрозрачна, трудна для удаления, provenance и точечной коррекции. [[S23]]

Следствие: Dennett может позднее использовать model-side activation/neural memory как ускоритель, learned guidance или cache. Но она должна быть:

- воспроизводимой из внешней памяти;
- сбрасываемой;
- неавторитетной;
- не единственным местом хранения факта;
- запрещённой для секретов и юридически значимых удалений без специальных гарантий.

## 3.9. Практические issue-трекеры показывают необходимость rebuild и fallback

В production-проектах регулярно встречаются ситуации, когда:

- фильтр сохраняет память в scope, из которого её невозможно вернуть;
- vector backend перестаёт возвращать записи;
- reranker выбрасывает кандидаты;
- singleton facts не попадают в graph query;
- temporal query возвращает эпизоды после reference time;
- persistent core memory протекает между сессиями;
- entity extraction теряет case-sensitive names;
- metadata или vector pipeline повреждаются. [[S37]]

Это не доказывает, что конкретные библиотеки плохи. Это доказывает, что ни graph, ни vector store, ни mutable core memory нельзя делать единственным каноническим слоем. Dennett обязан уметь диагностировать, пересобрать и временно обойти каждый derived-компонент.

## 3.10. Memory correctness — свойство траектории состояния, а не отдельной записи

Работа **Is Agent Memory a Database?** формулирует важную проблему: обычные records, vectors и graph edges локализуют корректность на отдельных объектах, тогда как долговременная память должна корректно эволюционировать как целое. Авторы выделяют ingestion, revision, forgetting и retrieval как операции над состоянием и связывают слабость существующих подходов с нерегулируемым ростом, отсутствием semantic revision, capacity-driven forgetting и read-only retrieval. [[S40]]

Для Dennett это означает:

- нельзя оценивать запись только вопросом «правильный ли этот fact»;
- нужно проверять, как она меняет current state, history, права, будущие действия и связанные projections;
- revision и forgetting — не CRUD над одной строкой, а контролируемые state transitions;
- retrieval должен быть способен изменить рабочее состояние агента только через допустимый influence path;
- качество памяти оценивается по тому, сохраняет ли вся траектория инварианты после тысяч обновлений, конфликтов и удалений.

Event Ledger не противоречит этой идее: ledger является каноническим журналом операций, а Memory State Machine определяет допустимые преобразования состояния поверх него.

## 3.11. Typed representation защищает от смешения источника, факта и инструкции

**MemIR** описывает provenance-role collapse: плоский текст заставляет модель путать исходное свидетельство, поисковую подсказку и утверждение, которому разрешено считаться фактом. Предложенное разделение raw evidence, retrieval cues и truth-bearing claims совпадает с ключевым направлением новой схемы Dennett. [[S41]]

Следствие:

- raw webpage остаётся внешним evidence, а не «воспоминанием Dennett»;
- summary является retrieval cue или projection, но не повышает авторитет источника;
- claim получает factual authority только через provenance и admission;
- instruction authority хранится отдельно от semantic similarity;
- result assembler получает нормализованный evidence bundle, а не плоскую смесь текста неизвестного происхождения.

## 3.12. Системная стоимость памяти распределяется между write, read и generation

Системное исследование десяти agent-memory подходов показывает, что разные дизайны не устраняют стоимость, а перемещают её между construction, retrieval и generation; важны scheduling, amortization по числу запросов, freshness-latency trade-off и capability floor используемой модели. [[S39]]

Следствие для Dennett:

- дорогая consolidation оправдана для часто используемой темы, но бессмысленна для одноразового эпизода;
- дешёвый write path должен быть отделён от optional semantic enrichment;
- retrieval strategy зависит от ожидаемого числа повторных запросов;
- нельзя сравнивать memory systems только по token cost одного ответа, игнорируя стоимость ingestion и maintenance;
- scheduler сна должен учитывать ожидаемую будущую полезность, а не просто возраст данных.

---

## 3.13. Фиксированная схема и свободный текст решают разные классы проблем

SCG-MEM показывает пользу динамической Cognitive Schema: новые данные сначала ассимилируются в существующую структуру, а при недостаточности структуры запускают accommodation — расширение схемы новыми концептами. Одновременно работа подчёркивает риск полностью свободной генерации ключей: модель может придумать структурно правдоподобный, но несуществующий путь. [[S42]]

DIAL-KG приходит к совместимому выводу с другой стороны: заранее заданные schemas ограничивают динамическое знание, однако schema-free extraction без governance недостаточен; новые schemas следует выводить из уже подтверждённого знания и применять инкрементально. [[S43]]

Следствие для Dennett: **необходим open-world semantic layer внутри closed-world operational envelope**. Неизвестное заранее содержание должно сохраняться свободно, но ID, provenance, principal, scope, time, trust, sensitivity, lifecycle и influence rights остаются строго типизированными.

## 3.14. Персонализация является траекторией, а не профилем

PERMA показывает, что предпочтения формируются в последовательности связанных событий, могут менять форму и idiolect и страдают от cross-domain interference. Простое извлечение похожих фраз хуже связывания событий, из которых предпочтение эволюционировало. [[S45]]

PAHF показывает полезность замкнутого цикла: clarification перед неоднозначным действием, применение извлечённых предпочтений и обновление памяти после feedback. Особенно важно, что этот цикл нужен для preference shift, а не только для первичного обучения. [[S46]]

OP-Bench показывает обратную сторону: подключение памяти часто вызывает irrelevance, repetition и sycophancy — агент использует личную информацию даже тогда, когда она не нужна. [[S47]]

Следствие: Dennett не должен иметь одну плоскую карточку «пользователь любит X». Ему нужны evidence-backed Preference Trajectories с контекстом применимости, исключениями, временем и допустимыми каналами влияния.

## 3.15. Proactive action требует модели бездействия и времени вмешательства

Исследование proactive assistants на 1000 симулированных personas и 30 участниках выделяет повторяющиеся измерения предпочтений: timing, autonomy и communication style; индивидуальная адаптация повышала воспринимаемое качество и доверие. [[S48]]

Полевое исследование IDE-помощника показало сильную зависимость от момента: вмешательства на границах workflow принимались заметно чаще, тогда как вмешательства посреди задачи чаще отклонялись. [[S49]]

Следствие: semantic event detector не должен сразу создавать действие. Он создаёт **Opportunity Candidate**, после чего Action Deliberation сравнивает `act`, `ask`, `draft`, `watch`, `defer`, `remember` и `do nothing`.

## 3.16. Исполняемая память полезна, но не должна становиться произвольным кодом

User as Code демонстрирует сильную идею: append-only history можно периодически компилировать в типизированное исполняемое состояние, что улучшает aggregate queries и позволяет детерминированно обнаруживать важные сочетания фактов. [[S50]]

Однако для Dennett произвольный Python, сгенерированный моделью, не может быть authority: он создаёт новый канал исполнения, migration burden и возможность poisoning. Принимается более безопасный вариант — **Safe Executable Projection**: versioned declarative rules, materialized queries или ограниченный DSL, построенные из evidence, протестированные и не способные расширять permissions.

## 3.17. Project-local event memory жизнеспособна, но self-study недостаточно для принятия формата целиком

PROJECTMEM демонстрирует практически полезный паттерн: local-first append-only typed events, детерминированные Markdown projections и pre-action предупреждения о повторении неудачных решений. [[S51]]

Это поддерживает переносимый проектный event log, но опубликованная оценка является двухмесячным self-study на десяти проектах. Поэтому Dennett заимствует принцип, а не готовую схему: формат обязан пройти clone/mount/concurrency/privacy/eval tests на собственных workloads.

## 3.18. Project memory лучше передавать как profile-based package

RO-Crate предлагает полезную модель переносимого directory package: root metadata file описывает локальные и внешние объекты как связанный JSON-LD graph, а profiles позволяют вводить только необходимые ограничения, сохраняя open-ended расширяемость. [[S52]]

W3C PROV даёт общий язык происхождения через entities, activities и agents, включая versioning, derivation и links между provenance bundles. [[S53]] PROV-AGENT показывает, как эту основу можно расширять для agent-centric workflows. [[S57]]

Следствие: нативный формат Dennett остаётся оптимизированным под Memory Fabric, но может генерировать RO-Crate/PROV-compatible interoperability projection. Внешний стандарт не должен становиться внутренним hot-path storage.

## 3.18A. Перенос памяти требует проверки целостности и ограниченного раскрытия

Portable Agent Memory предлагает content-addressed entries, Merkle-DAG provenance, capability-based selective disclosure и injection-resistant rehydration при передаче памяти между разными агентными средами. [[S58]]

Для Dennett это подтверждает четыре требования:

- importируемая project memory должна иметь проверяемый content root и lineage, а не только доверять именам файлов;
- получатель должен иметь возможность монтировать лишь разрешённые subsets, не раскрывая весь pack;
- rehydration в prompt должна проходить через тот же Trust and Influence Gate, что и обычный retrieval;
- integrity доказательство не превращает содержимое в epistemically trusted instruction: подписанный вредоносный pack остаётся вредоносным.

Полный формат этой свежей работы не принимается как обязательный стандарт Dennett без сравнительного eval. Заимствуются свойства tamper evidence, scoped disclosure и безопасного rehydration; native event/state model Memory Fabric сохраняется.

## 3.19. Provider instruction files нельзя сводить к одному универсальному файлу

Открытый формат `AGENTS.md` использует root и вложенные файлы, где ближайший файл имеет приоритет; Codex строит цепочку root → current working directory и имеет ограничение общего объёма по умолчанию. Claude Code использует другую модель: `CLAUDE.md`/`CLAUDE.local.md` конкатенируются, вложенные файлы могут загружаться по мере чтения поддиректории, а `AGENTS.md` поддерживается через import или symlink. [[S54]]

Исследование 100 популярных репозиториев обнаружило widespread configuration smells: context bloat, skill leakage и conflicting instructions. [[S55]]

Следствие: Dennett строит **Instruction Graph**, сохраняющий нативную семантику каждого источника, а не копирует содержимое всех файлов в одну project note.

## 3.20. Двусторонняя синхронизация требует ownership, а не last-write-wins

Server-Side Apply в Kubernetes является полезной инженерной аналогией: разные managers заявляют ownership отдельных полей, а конфликт блокирует случайное перезаписывание чужого значения; ownership может быть передан явно. [[S56]]

Dennett применяет этот принцип на уровне logical instruction records и managed sections. Это не означает зависимость от Kubernetes; принимается только модель `manager + base version + owned fields + conflict object + explicit transfer`.

# 4. Аудит предыдущей версии v0.2

## 4.1. Что было сильным и сохраняется

Предыдущая версия правильно определила:

- необходимость raw evidence;
- отличие дневника от изменяемых выводов;
- hybrid retrieval;
- temporal reasoning;
- project context;
- sensory memory;
- retrieval gate;
- sleep и consolidation;
- eval/replay;
- секретную память;
- human-readable views;
- отказ от pure vector и pure graph;
- необходимость адаптировать memory под проект.

Новая версия не отменяет эти идеи, а делает их строго совместимыми.

## 4.2. Проблема нескольких источников истины

В v0.2 одновременно существовали:

- append-only diary;
- living topic notes;
- graph facts;
- index notes;
- technical indexes.

Не было полного правила, что делать, если living note говорит одно, graph fact — другое, а diary содержит третье. Формулировка «обновить всё» не решает проблему: одна операция может завершиться частично, модель извлечения может ошибиться, два агента могут обновить разные слои параллельно.

### Исправление

Только Event Ledger и Evidence Objects каноничны. Все semantic layers содержат `derived_from` и `projection_version`. Их можно пересчитать.

## 4.3. Staging Buffer был слишком неопределённой сущностью

Staging в v0.2 одновременно означал:

- временную память;
- очередь на consolidation;
- место для сомнительных записей;
- recent context.

Это создаёт риск, что важная запись «застряла в staging», истекла или была обработана дважды.

### Исправление

В новой схеме нет staging как отдельного вида истины. Есть:

- каноническое событие;
- processing state и очереди derived processing;
- task/session working memory;
- quarantine для недоверенных semantic proposals;
- retention class для временных raw objects.

Событие уже записано и не потеряется; «не обработано» — статус pipeline, а не полу-память.

## 4.4. Прямое редактирование living notes агентами было опасно

Инструмент `memory_update_living_note` позволял агенту непосредственно изменять представление, которое потом могло трактоваться как актуальная память. Это ведёт к:

- silent overwrite;
- потере источника;
- одновременному конфликту;
- poisoning через скомпрометированного агента;
- невозможности понять, что именно изменилось в истине.

### Исправление

Агент подаёт proposal или correction event. Memory service транзакционно добавляет его в ledger, запускает resolver и перестраивает projections. Даже ручное редактирование Markdown создаёт semantic patch event.

## 4.5. Человеко-читаемые метаданные только внутри текста недостаточны

v0.2 справедливо требовала, чтобы источник, дата и уверенность были видны агенту в тексте. Но если они существуют только как свободный текст, невозможно надёжно:

- фильтровать ACL;
- вычислять `as_of`;
- делать deterministic resolver;
- отслеживать deletion;
- шардировать;
- проводить range query;
- проверять provenance;
- менять language/format;
- масштабировать индексирование.

### Исправление

У каждой сущности есть структурный machine-readable envelope. Человекочитаемое представление обязано отображать его смысл. Это не скрытый JSON вместо памяти, а dual representation: строгие поля для системы и нормальный Markdown для человека и агента.

## 4.6. Граф был недостаточно чётко ограничен

Temporal graph был полезным, но не было ответа, какие факты действительно должны быть nodes/edges, а какие лучше оставить эпизодами, документами или state records. Попытка графифицировать всё приводит к дорогому extraction, entity-resolution errors, огромному числу слабых edges и сложной синхронизации.

### Исправление

Graph — selective projection для:

- устойчивых entities;
- typed relations;
- temporal facts;
- causal chains;
- explicit references;
- project/dependency topology.

Слабые, одноразовые и плохо типизируемые фрагменты остаются observations/episodes и доступны другим retrieval lanes.

## 4.7. Retrieval scoring был слишком монолитным

v0.2 описывала hybrid scoring, но не отделяла калибровку результатов разных lanes. Cosine score, BM25 score, graph distance, freshness и importance нельзя честно складывать как одинаковые числа.

### Исправление

Используется rank fusion или learned calibration поверх per-lane ranks/features. Каждая lane может иметь собственный top-k и threshold. Финальная релевантность отделена от admissibility и evidence sufficiency.

## 4.8. Reranking считался почти обязательным

Исследования Anthropic показывают сильные выигрыши contextual BM25+dense+rerank на их наборах. Но более свежие исследования и production issues показывают, что off-the-shelf reranker может ухудшить отдельный домен или отбросить важные candidates. [[S20]] [[S37]]

### Исправление

Reranker — optional stage с per-domain eval, fallback и мониторингом candidate survival. Для exact/state query он может вообще не использоваться.

## 4.9. Не хватало причинной памяти

В v0.2 procedural memory существовала, но не было канонического action trace. Поэтому система могла запомнить совет без точной связи с environment version, инструментом, ошибкой, outcome и user feedback.

### Исправление

Causal Trace становится first-class projection и основанием для процедурных lessons.

## 4.10. Не было полноценной prospective memory

События Dennett упоминались в общем концепте, но memory document не моделировал future intent как отдельный жизненный цикл с false alarms, expiration и permission check.

### Исправление

Prospective Intent входит в Memory Fabric и имеет собственный retrieval/trigger path.

## 4.11. Недостаточно про concurrent multi-agent writes

v0.2 почти не определяла:

- branch-local memory;
- simultaneous corrections;
- offline writes;
- order of events;
- idempotency;
- conflict-preserving merge;
- read-your-writes.

### Исправление

Канонические операции append-only; у них есть causality/correlation и ordered identifiers. Derived current state строится поверх OpSet/CRDT-подобной модели и никогда не удаляет конфликт из истории.

## 4.12. Недостаточно про масштабирование и смену индексов

Фраза «гибридные индексы» не отвечала на:

- как добавлять миллионы/миллиарды vectors;
- как обновлять записи без полной перестройки;
- как менять embedding model;
- как хранить media;
- как искать при частичном отказе;
- как обслуживать hot и cold data.

### Исправление

Вводятся tiered storage, independently rebuildable indexes, embedding/index versioning, dual-read migration, federated query planning и backpressure.

## 4.13. Eval был слишком мал и chat-oriented

Десять сценариев v0.2 были полезны, но недостаточны для project, tool-use, causality, prospective triggers, concurrency, multimodality, deletion и poisoning.

### Исправление

Новая eval matrix объединяет public benchmarks и Dennett-specific end-to-end suites; сложный компонент остаётся только после ablation.

# 4A. Аудит версии 1.0 по новым сценариям

## 4A.1. Typed semantic objects могли стать скрытой фиксированной онтологией

Версия 1.0 правильно типизировала Claims, Current State, Prospective Intent и Causal Trace, но не определяла универсальный путь для знания, которое не укладывается ни в один известный профиль.

### Исправление

Observation получает open semantic payload. Любой extractor может сохранить `Freeform Insight` с evidence refs и arbitrary facets. Формализация в новый Claim subtype или schema extension происходит позже и никогда не является условием сохранения исходного смысла.

## 4A.2. Personal/Social Profile описывал, что известно, но не полностью — как выбирать действие

История отношений, предпочтения и обещания позволяли создать контекст, но не гарантировали корректный выбор между draft, send, ignore, ask и no-op.

### Исправление

Добавляются Preference Trajectory, Communication Model, Decision Precedent, Delegation Policy, Opportunity Candidate и Action Deliberation. Они остаются derived projections и не заменяют authorization.

## 4A.3. Prospective Intent не покрывал незапрошенные возможности

Prospective Intent хорошо работает, когда существует явное или выведенное будущее условие. Но система может заметить полезное событие, для которого trigger никогда не создавался.

### Исправление

Semantic observation сначала создаёт Opportunity Candidate. Оркестратор оценивает его через user goals, anti-goals, precedents, intervention preferences, interruption cost, reversibility и risk. Создание постоянного intent является одним из возможных результатов, а не обязательным первым шагом.

## 4A.4. Не было отдельного lifecycle для World Intelligence

Research Profile хранил источники и claims, но долгоживущая технологическая разведка требует постоянного dedupe, entity resolution, volatility, current verification и связи с опытом применения.

### Исправление

Добавляется World Intelligence Memory Space/Profile со стадиями signal → candidate claim → corroborated/current state → stale/deprecated, а также Technology Radar и Tool Experience Matrix.

## 4A.5. Project scope не был физически переносимым

Logical project scope можно было экспортировать, однако не было определено, что именно находится рядом с проектом, что монтируется по ссылке, что коммитится в Git и как избегать conflicts.

### Исправление

Добавляются Memory Space и Portable Project Memory Pack. Native project events хранятся в immutable writer/session segments; human notes и shareable evidence могут коммититься; indexes, embeddings, personal overlays и secrets — нет.

## 4A.6. Импорт project instructions был однонаправленным и слишком абстрактным

Версия 1.0 называла `Agents.md/project instructions` mandatory anchor, но не различала provider semantics, ownership и изменения upstream.

### Исправление

Mandatory anchor становится Effective Instruction Set из Instruction Graph. Внешние файлы остаются источниками со своими scope/precedence. Двусторонний export разрешён только для managed records/blocks с three-way merge.

## 4A.7. Полный перенос памяти между пользователями создаёт trust и privacy проблему

Даже корректный project pack может содержать conclusions, procedures или instructions, созданные чужими агентами.

### Исправление

Импортированный pack монтируется в отдельном trust domain `external-shared`. Он доступен для поиска как project evidence, но его procedures и instructions проходят workspace trust/admission; он не сливается автоматически с Global Personal Memory.

## 4A.8. Совместимость могла начать диктовать внутренний дизайн

Попытка сделать всю память читаемой обычным coding-agent привела бы к упрощению структуры до Markdown и потере provenance/state/governance.

### Исправление

Dennett-native package является каноническим. Markdown views, MCP и provider files — adapters/projections. Совместимость максимизируется только там, где она не снижает качество собственной памяти.

# 5. Неподвижные инварианты памяти

Инварианты важнее конкретных алгоритмов. Архитектура может меняться, но нарушение этих правил считается дефектом системы.

## 5.1. Каноничность

1. Каноничны только подтверждённые операции Event Ledger и связанные Evidence Objects.
2. Summary, note, graph edge, embedding, topic dossier и current state — derived projections.
3. Любая projection содержит version и provenance.
4. Любую projection можно удалить и пересобрать без потери исходного события.

## 5.2. История и изменение

1. Новое утверждение не переписывает прошлое.
2. Current state строится отдельно от history.
3. Коррекция создаёт новое событие и stable correction handle.
4. Конфликт сохраняется до явного resolver decision или остаётся unresolved.
5. Historical query использует `as_of`, а current query — текущую projection с freshness policy.

## 5.3. Доказательность

1. Semantic claim должен ссылаться на evidence либо быть явно помечен как unsupported inference/hypothesis.
2. Summary не может повысить epistemic status исходного материала.
3. Если raw evidence удалено по policy, projection обязана показывать, что grounding больше недоступен.
4. Для consequential action память должна возвращать evidence package, а не только красивый вывод.

## 5.4. Разделение данных и власти

1. Найденная память является данными, а не системной инструкцией.
2. External content никогда не становится trusted procedure автоматически.
3. Память не может выдать агенту новые permissions.
4. Пользовательское предпочтение не может менять safety gate, authorization или объективный факт.
5. Learned procedure не становится активной до validation/quarantine exit.

## 5.5. Scope и доступ

1. Каждое событие имеет principal, scope и policy.
2. Project scope — сильный prior, но не абсолютная стена: cross-project retrieval возможен только намеренно и после gate.
3. Cross-principal memory запрещена без явной policy.
4. Secret data никогда не попадает в обычные индексы в открытом виде.
5. Access decision принимается до формирования prompt context.

## 5.6. Надёжность write path

1. Запись не зависит от успешного LLM extraction.
2. Повторная доставка не создаёт дубль благодаря idempotency key/content hash.
3. После acknowledgement каноническая запись durable.
4. Derived processing допускает eventual consistency, но task должен видеть собственные недавние записи через overlay.
5. Ошибка любого indexer видима и повторяема.

## 5.7. Поиск

1. Ни одна retrieval lane не считается универсальной.
2. Similarity не равна admissibility.
3. Exact lookup, current state, history и causal query не должны решаться одним и тем же top-k vector pipeline.
4. Context budget ограничен независимо от общего объёма памяти.
5. Retrieval может остановиться с abstention.
6. Для каждого важного результата можно объяснить, какие memories повлияли и почему.

## 5.8. Масштабирование

1. Online write path не сканирует весь архив.
2. Обычный query не зависит линейно от числа событий.
3. Media objects дедуплицируются и tiered.
4. Индексы версионируются и мигрируют без остановки canonical writes.
5. Система имеет degraded mode при отказе graph, vector, reranker или LLM extractor.

## 5.9. Удаление

1. Attention decay, supersession, archival и physical deletion — разные операции.
2. User forget request имеет traceable deletion plan.
3. Удаление распространяется на objects, projections, indexes, caches, replicas и backups согласно policy.
4. После удаления система не должна «восстановить» содержимое из summary или procedural lesson.
5. Tombstone может сохранить факт операции удаления без сохранения удалённого содержания.

## 5.10. Эволюция

1. Memory policy не меняется бесконтрольно на live traffic.
2. Новая версия проходит replay, shadow или canary.
3. Prompt/model/index/schema version сохраняется в provenance.
4. Улучшение доказывается end-task метриками, а не субъективным впечатлением.
5. Откат derived logic не требует отката canonical evidence.

---
## 5.11. Open-world semantics, closed-world control

1. Система не обязана заранее знать все виды смысловой информации.
2. Любой новый смысл можно сохранить как evidence-grounded Open Observation.
3. Нельзя оставлять свободными ID, principal, scope, provenance, time, sensitivity, lifecycle, trust и influence policy.
4. Неизвестный semantic facet не получает автоматически новых прав доступа, исполнения или авторизации.
5. Свободная semantic memory является полноценным конечным состоянием, а не временным сырьём, которое обязано стать schema.

## 5.12. Selective semantic structuring

1. Topic Dossier, Claim, Human Note и Open Observation являются default semantic representations.
2. Flexible facet или on-demand extraction предпочтительнее нового global type.
3. Semantic View создаётся только для повторяющегося retrieval/aggregation/action use case.
4. Strict operational schema применяется только к настоящему системному контракту.
5. View имеет evidence, owner, scope, tests и возможность отключения, но не требует массовой миграции исходной памяти.
6. Taxonomy churn, пустые поля и ложная определённость считаются дефектом.


## 5.13. User alignment не является истиной или разрешением

1. Preference Trajectory, Communication Model и User Decision Model являются projections.
2. Они обязаны сохранять scope, exceptions, uncertainty и evidence.
3. Они могут влиять только на разрешённые channels: style, personalization, planning, intervention timing и подобные.
4. Они не могут подменить current facts, safety policy или permission token.
5. В objective mode персональная память отделяется от factual judgment.
6. Вариант `do nothing` обязателен для proactive deliberation.

## 5.14. Федерация без неявного слияния

1. Каждая Memory Space сохраняет stable identity, owner, trust domain и provenance.
2. Mount не означает copy и не означает merge.
3. Imported space не становится Global Personal Memory автоматически.
4. Promotion знания между spaces является отдельным событием с source handles.
5. Один и тот же evidence object может быть доступен нескольким spaces по ссылке, но policy каждого пространства проверяется отдельно.

## 5.15. Переносимая память не равна полной памяти

1. Project Memory Pack содержит только portable/shareable subset.
2. Secret, personal, machine-local и runtime-only data не попадают в pack по умолчанию.
3. Embeddings, vector indexes, caches и graph databases являются rebuildable local derivatives.
4. Pack должен быть полезен после clone без исходной установки Dennett, но полная функциональность разрешена только совместимому consumer.
5. Export проходит privacy, license и dependency audit.

## 5.16. Нативные instruction sources сохраняют семантику

1. `AGENTS.md`, `CLAUDE.md`, rules и другие файлы не превращаются в один плоский prompt.
2. Dennett сохраняет provider, path scope, precedence, lazy-load semantics и source version.
3. Existing files не удаляются и не перезаписываются без ownership.
4. Bidirectional sync работает по logical records или managed blocks, а не по полному last-write-wins файлу.
5. Conflict останавливает автоматическую перезапись.

## 5.17. Совместимость является projection

1. Dennett-native memory model не ограничивается возможностями Markdown или конкретного coding-agent.
2. `AGENTS.md`, `CLAUDE.md`, Markdown index, MCP и RO-Crate являются compatibility/interoperability views.
3. Потеря совместимого view не должна приводить к потере канонической памяти.
4. Любой view можно пересобрать из canonical space и mapping records.

# 6. Общая логическая модель Memory Fabric

Memory Fabric состоит из шести плоскостей. Это логическое разделение; позже несколько плоскостей могут быть реализованы одним сервисом или, наоборот, одна плоскость может быть разделена на несколько компонентов.

## 6.1. Canonical Evidence Plane

Хранит то, что система действительно получила или сделала:

- immutable Memory Events;
- raw и normalized Evidence Objects;
- source metadata;
- access/retention policy;
- causal ordering;
- hashes/signatures;
- deletion tombstones.

Это основание для восстановления всего остального.

## 6.2. Semantic Projection Plane

Содержит вычисляемые структуры:

- Observations;
- Episodes и Scenes;
- Claims/Beliefs;
- Entities и relations;
- Current State Projections;
- Causal Traces;
- Prospective Intents;
- Procedures/Lessons;
- Topic Dossiers;
- Human Notes.

Они ускоряют понимание и поиск, но не заменяют evidence.

## 6.3. Retrieval and Context Plane

Содержит:

- query planner;

- temporal indexes;
- graph/causal search;
- multimodal search;
- candidate fusion;
- trust/admission gate;
- evidence sufficiency loop;
- context composer;
- indexed handles.

## 6.4. Working Memory Plane

Содержит краткоживущие рабочие состояния:

- session workspace;
- project-chat workspace;
- task workspace;
- branch-local agent workspace;
- goals, constraints, decisions, open questions;
- stable handles на долгосрочную память;
- recent-event overlay;
- bounded scratch summaries.

Working memory не является долговременной истиной. После завершения задачи из неё сохраняются значимые события, outcomes и decisions, а не весь скрытый scratchpad.

## 6.5. Governance Plane

Содержит:

# 7. Канонический слой

## 7.1. Memory Event

Memory Event — минимальная неизменяемая запись о том, что произошло с точки зрения Memory Fabric.

Примеры:

- пользователь отправил сообщение;
- агент вызвал tool;
- tool вернул результат;
- пользователь сделал скриншот;
- ASR распознал фразу;
- git commit создан;
- файл изменён;
- пользователь исправил память;
- событие было удалено по privacy request;
- resolver выбрал текущую версию;
- агент завершил задачу;
- пользователь оценил результат;
- prospective trigger сработал или был подавлен.

### Логический контракт Memory Event

```yaml
memory_event:
  event_id: globally_unique_orderable_id
  schema_version: string

  principal_id: owner_or_security_principal
  actor:
    type: user | agent | tool | device | system | external_person | external_service
    id: stable_actor_id

  event_kind: typed_event_name
  source_channel: chat | voice | screen | camera | repo | filesystem | tool | web | email | messenger | calendar | system

  observed_at: event_time_or_interval
  ingested_at: canonical_commit_time
  source_clock: optional_device_sequence_or_vector_clock

  scope:
    user_id: optional
    project_id: optional
    conversation_id: optional
    session_id: optional
    task_id: optional
    agent_id: optional
    device_id: optional

  causality:
    correlation_id: optional
    parent_event_ids: []
    caused_by_event_ids: []
    goal_id: optional
    action_id: optional
    branch_id: optional

  evidence_refs: []
  content_digest: optional_hash
  idempotency_key: optional

  epistemic_origin: direct_observation | user_assertion | agent_assertion | tool_result | external_content | system_inference
  trust_domain: typed_domain
  sensitivity: typed_policy
  influence_policy: typed_policy
  access_policy_ref: policy_id
  retention_policy_ref: policy_id

  processing:
    capture_status: committed
    projection_status: pending | partial | complete | failed
    latest_processing_version: optional

  integrity:
    source_signature: optional
    canonical_hash: hash
```

Это логическая схема, а не обязательный конкретный YAML/JSON формат. Однако поля времени, scope, provenance, trust, sensitivity, causality, version и idempotency должны существовать структурно, а не только внутри текста.

## 7.2. Почему событие неизменяемо

Неизменяемость даёт:

- возможность аудита;
- восстановление после плохой модели consolidation;
- параллельное обновление без потери одной ветки;
- точную коррекцию;
- воспроизводимый rebuild graph/indexes;
- сравнение алгоритмов на одной истории;
- безопасный rollback;
- объяснение «почему Dennett так решил»;
- корректное offline sync.

Неизменяемость не означает, что персональные данные нельзя удалить. При delete request создаётся tombstone/forget operation, а физические bytes удаляются или cryptographically erased согласно policy. Неизменяемым остаётся допустимый audit-факт, что операция произошла, но не обязательно само удалённое содержание.

## 7.3. Evidence Object

Evidence Object хранит первичный или производный media-объект.

Примеры:

- сообщение целиком;
- фрагмент transcript;
- audio segment;
- screenshot;
- accessibility tree;
- video fragment/keyframe;
- PDF/document;
- tool stdout/stderr;
- HTTP response;
- git diff;
- test report;
- generated artifact;
- user annotation.

### Логический контракт Evidence Object

```yaml
evidence_object:
  object_id: content_address_or_unique_id
  media_type: mime_or_typed_media
  byte_length: integer
  content_hash: cryptographic_hash

  created_at: timestamp
  observed_interval: optional_interval
  source_event_ids: []

  renditions:
    original: storage_ref
    normalized_text: optional_storage_ref
    transcript: optional_storage_ref
    ocr: optional_storage_ref
    accessibility_tree: optional_storage_ref
    keyframes: []
    thumbnails: []
    redacted_variant: optional_storage_ref

  derivation:
    parent_object_ids: []
    transform_name: optional
    transform_version: optional
    model_version: optional

  security:
    sensitivity: typed_policy
    encryption_key_ref: optional
    access_policy_ref: policy_id
    taint_labels: []

  lifecycle:
    storage_tier: hot | warm | cold | offline
    retention_policy_ref: policy_id
    deletion_state: active | pending_delete | tombstoned | erased
```

## 7.4. Content-addressed storage и дедупликация

Если два события ссылаются на один и тот же файл, screenshot или media content, bytes не обязаны храниться дважды. Content hash позволяет:

- дедуплицировать;
- проверять integrity;
- строить stable references;
- безопасно мигрировать storage;
- обнаруживать повторные captures;
- не индексировать одинаковый screenshot сотни раз.

При этом одинаковые bytes в разных contexts могут иметь разные Memory Events и разные policies. Dedup объекта не означает слияние смысла или прав.

## 7.5. Append before interpret

Критическое правило write path:

> Сначала канонически зафиксировать то, что произошло; затем интерпретировать.

Для текстового сообщения это означает:

1. сохранить исходное сообщение и event envelope;
2. подтвердить приём;
3. затем выделять claims, preferences, entities, project links и intents.

Для tool call:

1. сохранить structured input/output;
2. связать с goal/action;
3. затем делать summary и lesson.

Для сенсорного события:

1. локальная privacy policy решает, можно ли сохранить;
2. сохранить selected raw segment и event metadata;
3. затем делать OCR/ASR/entity extraction.

Это предотвращает потерю факта из-за неудачного extractor call.

## 7.6. Processing state вместо staging memory

Каждое событие проходит derived processing с явно наблюдаемым состоянием:

```text
COMMITTED
  → CONTEXTUALIZED
  → SEGMENTED
  → SEMANTIC_PROPOSALS_CREATED
  → PROPOSALS_ADMITTED_OR_REJECTED
  → STATE_RESOLVED
  → INDEXED
  → HUMAN_VIEWS_UPDATED
```

Переходы могут выполняться асинхронно, повторяться и идти частично. Ошибка на этапе graph update не откатывает событие. Queue item содержит idempotency key и processing version.

## 7.7. Read-your-writes

Derived indexes могут обновляться не мгновенно. Чтобы пользователь или агент сразу видел только что записанное:

- task/session workspace хранит recent event handles;
- query planner добавляет recent-event overlay;
- exact lookup по event_id/object_id работает сразу;
- high-risk workflow может дождаться нужной projection, если без неё нельзя действовать.

## 7.8. Source adapters

Разные источники должны терять как можно меньше native structure.

### Chat/voice

Хранятся роли, speaker, turn boundaries, timestamps, interruption, explicit reply links, ASR confidence.

### Repository

Хранятся commit, branch, path, symbol, diff, test result, command, exit code. Source code не превращается сначала в prose note.

### Tool

Хранятся tool schema/version, arguments, environment, result, errors, side effects.

### Screen

Хранятся app/window/URL, accessibility tree/DOM, selected text, screenshot, OCR fallback, display geometry.

### Web/document

Хранятся URL/source identity, fetch time, content hash, section hierarchy, citation anchors, trust/taint.

### Calendar/messenger/structured service

Хранятся stable external IDs и structured fields; LLM extraction используется только для дополнительного смысла.

# 8. Semantic Projection Plane

## 8.1. Observation

Observation — нормализованный, контекстно самодостаточный фрагмент evidence.

Он нужен, потому что raw chunk часто теряет смысл. Например, строка «давай всё-таки второй вариант» бесполезна без связи с обсуждаемыми вариантами, проектом, speaker и временем.

### Поля Observation

```yaml
observation:
  observation_id: id
  evidence_refs: []
  source_event_ids: []
  normalized_content: text_or_structured_payload
  contextual_frame:
    project_id: optional
    task_id: optional
    participants: []
    active_application: optional
    referenced_entities: []
    preceding_context_handles: []
  observed_interval: interval
  modality: text | audio | visual | code | tool | mixed
  extraction:
    method: deterministic | model | human
    model_version: optional
    confidence: calibrated_or_categorical
  trust_and_access: inherited_plus_overrides
```

Contextualized retrieval improves качество потому, что isolated chunks часто теряют document-level identity, время и предмет; комбинация contextualized dense и lexical retrieval показала существенное снижение retrieval failures в экспериментах Anthropic. [[S20]]

## 8.2. Episode

Episode — связная последовательность observations, объединённая деятельностью, темой, целью, проектом, приложением или событием.

Примеры:

- конкретный разговор о памяти Dennett;
- одна coding session;
- debugging loop;
- встреча;
- поездка в вуз;
- просмотр и выбор интерфейсных референсов;
- один browser workflow;
- одна попытка решить автономное поручение.

### Почему не fixed chunks

Fixed token chunks удобны для индекса, но разрывают причинные и временные границы. Episode segmentation должна использовать несколько сигналов:

- topic shift;
- task/goal switch;
- app/window switch;
- participant change;
- temporal gap;
- explicit session boundary;
- tool phase transition;
- surprise/novelty spike;
- commit/checkpoint;
- user correction;
- outcome boundary.

HiMem использует topic-aware event/surprise segmentation, а HiGMem показывает пользу двухуровневого поиска «event summary → конкретные turns». [[S26]]

### Иерархия Episodes

Episode может иметь уровни:

- micro-event;
- turn/action cluster;
- task episode;
- session episode;
- project phase;
- project history;
- life period.

Иерархия не обязана строиться полностью сразу. Высокие уровни могут появляться во время consolidation.

## 8.3. Scene / Thematic Cluster

Scene объединяет episodes вокруг устойчивой темы или деятельности и служит навигационной точкой.

Примеры:

- «проект Dennett — память»;
- «предпочтения интерфейса»;
- «учебная работа в УрФУ»;
- «ошибки при использовании tool X».

EverMemOS использует MemCells и MemScenes; Infini Memory — maintainable topic documents. В Dennett Scene не является истиной, а хранит summary, representative episodes и stable handles на evidence. [[S12]] [[S24]]

## 8.4. Claim / Belief

Claim — proposition, которую система может использовать в reasoning.

Примеры:

- «пользователь предпочитает B текущему A»;
- «проект использует PostgreSQL»;
- «tool X ломается на Windows при условии Y»;
- «человек N обещал прислать файл»;
- «это решение было отклонено из-за latency».

### Логический контракт Claim

```yaml
claim:
  claim_id: stable_id
  schema_version: string

  natural_language: human_readable_proposition
  structured_form:
    subject_ref: optional
    predicate: optional
    object_value_or_ref: optional

  scope:
    principal_id: id
    project_id: optional
    domain: optional

  epistemic_status: observed | asserted | inferred | hypothesis | contested | superseded | retracted

  time:
    valid_from: optional
    valid_to: optional
    asserted_at: timestamp
    transaction_from: timestamp
    transaction_to: optional

  provenance:
    supporting_observation_ids: []
    supporting_evidence_refs: []
    contradicting_claim_ids: []
    supersedes_claim_ids: []
    derived_by_model_version: optional

  quality:
    source_reliability: categorical_or_calibrated
    extraction_confidence: categorical_or_calibrated
    support_strength: categorical_or_calibrated
    identity_confidence: optional
    freshness_status: fresh | stale_candidate | historical | unknown

  governance:
    trust_domain: typed
    sensitivity: typed
    influence_policy: typed
    access_policy_ref: policy_id

  resolver:
    predicate_contract_ref: optional
```

## 8.5. Почему Claim не всегда должен быть triple

Структурная форма полезна для state lookup, temporal resolution и graph traversal. Но мир пользователя открыт и не укладывается в заранее заданную ontology.

Поэтому Claim может иметь:

- natural-language proposition как обязательное human-readable ядро;
- structured subject/predicate/object только когда это помогает;
- schema-on-read для новых типов;
- project-specific typed extensions;
- no forced entity merge при низкой уверенности.

Так система избегает двух крайностей: плоского текста без машинной логики и огромной жёсткой ontology, которая ломается на каждом новом типе информации.

## 8.6. Confidence — не одно фальшивое число

Вместо «уверенность 83%» система разделяет:

- насколько надёжен источник;
- насколько уверенно evidence было распознано;
- насколько сильно evidence поддерживает claim;
- насколько уверенно разрешена entity identity;
- насколько свежо значение;
- есть ли conflict;
- прошёл ли claim независимую verification.

Один итоговый score может использоваться внутри ranking, но UI и consequential reasoning должны видеть составляющие.

## 8.7. Entity

Entity — стабильная ссылка на человека, проект, устройство, файл, компонент, организацию, модель, инструмент, место или концепт.

### Entity resolution должна быть обратимой

Нельзя молча сливать два «Алексей» или две одинаково названные папки. Операции:

- `candidate_same_as`;
- `merge_entity`;
- `split_entity`;
- `alias_of`;
- `possibly_related_to`;

должны иметь evidence и correction handles.

Project namespace, contact identifiers, repository IDs и external service IDs используются как сильные anchors. LLM similarity — только дополнительный сигнал.

## 8.8. Relation и temporal graph

Graph содержит selective relations:

- entity ↔ entity;
- claim ↔ evidence;
- claim ↔ claim;
- episode ↔ project;
- action ↔ outcome;
- decision ↔ alternative;
- file/symbol ↔ dependency;
- person ↔ project;
- intent ↔ trigger;
- procedure ↔ trace.

Graphiti полезен как reference для temporal knowledge graph, provenance и hybrid graph search, но graph Dennett должен оставаться derived и rebuildable. [[S22]]

## 8.9. Current State Projection

Current State Projection отвечает на вопрос: «какое значение система должна использовать сейчас для данного key/predicate/scope?»

Он содержит:

```yaml
current_state:
  state_key: typed_key
  scope: typed_scope
  candidate_claim_ids: []
  resolved_value_or_claim_ids: []
  resolver_type: latest_verified | authoritative_source | state_machine | multi_value | aggregate | human_override | unresolved
  resolver_version: string
  resolved_at: timestamp
  source_watermark: latest_event_seen
  conflict_status: none | visible | blocking
  freshness_requirement: policy
```

Current State не удаляет candidates и history. Это cache/projection ответа, который можно пересчитать.

## 8.10. Causal Trace

Causal Trace — основная память действий агента.

```yaml
causal_trace:
  trace_id: id
  goal_id: id
  project_id: optional
  task_id: optional
  actor_agent_ids: []
  environment_version: optional

  intent_event_ids: []
  constraints_claim_ids: []
  decision_records: []
  actions: []
  observations: []
  artifacts: []
  verifications: []
  outcome:
    status: success | partial | failure | cancelled | unknown
    evidence_refs: []
  assessment:
    user_feedback: optional
    reviewer_feedback: optional
    metrics: optional
    failure_causes: []
  reusable_lessons: []
```

Decision record хранит краткое rationale и рассмотренные альтернативы, но не приватный скрытый chain-of-thought модели.

## 8.11. Prospective Intent

Prospective Intent хранит то, что должно быть вспомнено или выполнено позже.

```yaml
prospective_intent:
  intent_id: id
  created_from_event_ids: []
  natural_language: text
  trigger_spec:
    deterministic_conditions: optional
    semantic_condition: optional
    temporal_window: optional
    required_context: optional
  action_spec:
    notify | ask | launch_agent | update_project | retrieve_context | custom
  scope: typed_scope
  priority: typed
  false_positive_cost: typed
  false_negative_cost: typed
  cooldown: optional
  expiry: optional
  authorization_requirement: policy
  status: active | paused | triggered | fulfilled | expired | cancelled
  last_evaluated_at: optional
  trigger_evidence_refs: []
```

Prospective Intent не должен автоматически выполнять опасное действие только потому, что semantic matcher сработал. Trigger detection и action authorization — разные этапы.

## 8.12. Procedure / Lesson

Procedure — версия повторно используемого «как делать».

Состояния:

- candidate;
- quarantined;
- validated;
- deprecated;
- revoked.

Procedure содержит:

- условия применимости;
- scope/environment version;
- steps или guidance;
- supporting successful traces;
- failure traces и anti-patterns;
- confidence/utility;
- tests;
- version;
- author/origin;
- security review;
- rollback.

ReMe показывает пользу совместного анализа success, failure и comparative experience, но исследования MemoryGraft, sleeper poisoning и environment-injected poisoning показывают, что даже «успешный опыт» может быть атакующим объектом. Поэтому procedural memory всегда строже обычной factual memory. [[S29]] [[S18]] [[S19]]

## 8.13. Topic Dossier

Dossier — поддерживаемое человекочитаемое представление темы.

Пример разделов:

```markdown
# Предпочтения интерфейса пользователя

## Текущий вывод
...

## Контекст и границы применимости
...

## Подтверждающие эпизоды
- [E:...]

## Противоречия и изменения
...

## Неуверенные гипотезы
...

## История projection
...
```

Dossier не редактирует evidence. Он строится из claims/episodes и содержит stable handles. Ручная правка порождает correction event.

## 8.14. Human Note

Human Note — свободная Markdown-заметка пользователя или агента. Она может быть:

- самостоятельным evidence object, если пользователь написал её вручную;
- projection, если она сгенерирована системой;
- mixed document, где user-authored blocks и generated blocks различаются;
- project-local файл, например Agents.md;
- export/view над graph/claims.

Human Notes остаются важны для контроля и совместимости, но free-form tags не заменяют typed scope, ACL, time и provenance.

## 8.15. Open Observation и Freeform Insight

Open Observation — основной способ сохранить смысл, который не обязан укладываться в заранее известную категорию.

```yaml
open_observation:
  observation_id: id
  evidence_refs: []
  scope: typed_scope
  observed_valid_time: interval
  natural_language: text
  known_facets: {}
  extension_facets: {}
  candidate_entities: []
  candidate_relations: []
  possible_implications: []
  uncertainty: text
  utility_signals: []
  extractor_provenance: model_prompt_versions
```

`extension_facets` допускают новые namespaced keys, но они являются retrieval hints, а не новыми системными типами. Система не обязана превращать каждую новую грань вкуса, поведения или опыта в отдельную schema.

Open Observation доступна поиску немедленно и может быть включена в Topic Dossier без дополнительной формализации.

## 8.16. Selective Semantic Structuring

Semantic memory использует четыре уровня, причём переход вверх не обязателен.

### Уровень A: evidence и episode

Сохраняется то, что произошло, с provenance и временем.

### Уровень B: open observation, claim и topic document

LLM формулирует смысл в человекочитаемом виде, сохраняя evidence handles, scope, исключения и неопределённость. Это default для персональной, социальной, исследовательской и мировой памяти.

### Уровень C: flexible semantic view

Для повторяющейся потребности создаётся materialized или query-time view: например, «принятые пользователем варианты ответа этому человеку» или «инструменты, совместимые с Windows и локальной обработкой». View может иметь поля, но не требует переписывать исходные observations.

### Уровень D: strict operational record

Используется только там, где точное поле влияет на действие или state: разрешение, активное событие, project manifest, current preference override, контактный адрес, deadline, status, tool capability, current model selection.

### Основной принцип

Структура выбирается **по типу операции**, а не по теме знания. Одна и та же тема может существовать как свободное dossier для рассуждения и как маленькая строгая projection для конкретного автоматического действия.

## 8.17. Semantic View Definition

Вместо общей системы саморастущих схем вводится лёгкая сущность `Semantic View Definition`.

```yaml
semantic_view_definition:
  view_id: id
  scope: global | space | project | domain
  purpose: retrieval | aggregation | current_state | action_support | ui
  source_predicate: query_or_rules
  output_fields: []
  evidence_required: boolean
  unknown_policy: omit | explicit_unknown | block_action
  freshness_policy: optional
  materialization: on_demand | cached | maintained
  evaluation_refs: []
  status: experimental | validated | deprecated | disabled
```

Правила:

- view не изменяет Evidence Objects;
- view может строиться on demand и не требовать backfill;
- поля без доказательств остаются unknown, а не угадываются;
- experimental view не влияет на consequential action;
- validation сравнивает view с direct evidence/topic-document baseline;
- если view не приносит измеримой пользы, она удаляется без миграции памяти;
- новые strict operational records создаются разработчиком или проверенным системным изменением, а не автоматически из одного наблюдения.

### Когда допустима эволюция operational schema

Только если меняется настоящий системный контракт: формат Project Manifest, тип разрешения, модель события, protocol version или другой operational record. Для такой эволюции нужны versioning и migration. Это не относится к появлению нового вкуса, идеи или социальной закономерности.


## 8.18. Preference Trajectory

Preference — не одно значение, а temporal, contextual trajectory.

```yaml
preference_trajectory:
  preference_id: id
  subject_principal: user_id
  dimension_or_open_description: text
  scope_predicates: []
  states:
    - valid_interval: interval
      tendency: text_or_typed_value
      strength: qualitative_or_calibrated
      evidence_refs: []
      explicitness: explicit | behavioral | inferred
      exceptions: []
      confidence_factors: {}
  cross_domain_policy: []
  allowed_influence_channels: []
  current_resolution: claim_refs
```

Preference Trajectory может описывать не только заранее известные категории. `dimension_or_open_description` позволяет сохранить новую ось без создания отдельного типа. Если по ней появятся повторяющиеся точные запросы, может быть создана Semantic View.

### Ограничения

- один отказ не превращается в вечное правило;
- поведение в одном проекте не переносится глобально без evidence;
- предпочтение не применяется, если пользователь запросил objective evaluation;
- старая preference остаётся historical state;
- сильное явное изменение пользователя имеет больший вес, но не стирает историю.

## 8.19. Communication Model

Communication Model создаётся по сочетанию `user × counterpart/group × channel × situation` и отвечает не «кто этот человек вообще», а «как пользователь обычно общается здесь».

```yaml
communication_model:
  model_id: id
  user_id: id
  counterpart_scope: person | group | organization | unknown
  channel_scope: telegram | email | messenger | voice | other
  context_predicates: []
  content_boundaries: []
  tone_tendencies: []
  length_tendencies: []
  timing_tendencies: []
  representative_sent_message_refs: []
  accepted_draft_refs: []
  rejected_or_edited_draft_refs: []
  autonomous_action_policy_ref: optional
  uncertainty: structured
  valid_interval: interval
```

Representative messages используются как style evidence, но перед помещением в context проходят sensitivity/disclosure filtering. Content plan строится из актуальных фактов отдельно от style realization.

## 8.20. Decision Precedent

Decision Precedent — компактная projection одного или нескольких Causal Traces:

```yaml
decision_precedent:
  precedent_id: id
  situation_signature: text_and_facets
  goal_refs: []
  context_conditions: []
  alternatives_considered: []
  selected_action: text
  no_action_considered: boolean
  rationale_summary: text
  authorization_mode: text
  outcome_refs: []
  user_feedback_refs: []
  applicability_limits: []
  environment_version: optional
```

Precedent не предписывает повторение. Он является evidence о том, как пользователь или система поступили раньше и чем это закончилось.

## 8.21. User Decision Model и Delegation Policy

User Decision Model — current projection над Preference Trajectories, goals, anti-goals, precedents и feedback.

Она содержит:

- active goals и их относительный приоритет;
- anti-goals и unacceptable outcomes;
- типичные trade-offs;
- отношение к риску и обратимости;
- предпочтительные уровни инициативы;
- границы автономности по action class, person, channel и project;
- intervention timing;
- interruption/annoyance preferences;
- uncertainty и конфликтующие patterns;
- ссылки на evidence и precedents.

Это не статичная «цифровая личность» и не единый reward score. Проекция пересчитывается по scope и времени. Для каждого решения формируется task-specific slice.

## 8.22. Opportunity Candidate

Opportunity Candidate создаётся, когда система видит возможную пользу, но не имеет уже активного Prospective Intent.

```yaml
opportunity_candidate:
  opportunity_id: id
  detected_event_refs: []
  affected_goals: []
  affected_projects: []
  why_potentially_useful: text
  candidate_actions: []
  initial_urgency: typed
  reversibility: typed
  interruption_cost: typed
  uncertainty: structured
  expires_at: optional
  status: new | deliberating | ignored | converted | expired
```

Opportunity не даёт права действовать и не обязана становиться notification.

```yaml
action_deliberation:
  deliberation_id: id
  opportunity_or_intent_ref: id
  current_state_refs: []
  user_alignment_refs: []
  precedent_refs: []

  alternatives:
    - action: do_nothing | remember | watch | defer | draft | simulate | ask | act

```

## 8.24. Safe Executable Projection

Для aggregate queries и детерминированных conditions допускается компиляция памяти в ограниченное исполняемое представление.

Разрешённые формы:

- typed materialized views;
- declarative predicates;
- ограниченный rules DSL;
- deterministic aggregations;
- validated state machines;
- scheduled queries.

Запрещён baseline:

- произвольный Python/JavaScript, сгенерированный моделью и автоматически исполняемый с правами Dennett;
- правило без evidence lineage;
- правило, которое способно расширить permissions;
- silent mutation без version/test/rollback.

Safe Executable Projection имеет source claims, schema version, tests, owner, lifecycle и execution sandbox. [[S50]]

# 9. Epistemic authority и допустимое влияние памяти

Одна из главных ошибок memory-augmented agents — считать, что любая релевантная память может одинаково влиять на ответ и действие. Это неверно.

## 9.1. Типы происхождения

Каждый фрагмент должен иметь origin class:

- **system policy** — неизменяемые системные ограничения;
- **explicit user instruction** — осознанная команда пользователя в допустимом scope;
- **project instruction** — Agents.md, project rules и аналогичные инструкции;
- **authoritative structured source** — repository HEAD, calendar record, database record, external stable ID;
- **direct observation** — screenshot, transcript, tool result;
- **user assertion** — пользователь сообщил факт;
- **agent assertion** — агент сообщил или вывел;
- **derived inference** — система сделала вывод;
- **external untrusted content** — сайт, письмо, README, документ, чужая речь;
- **learned procedure** — опыт, который прошёл или не прошёл validation.

## 9.2. Influence Policy

Memory item должен не только проходить access control, но и иметь ограничения на тип влияния.

Пример каналов влияния:

- `answer_content`;
- `answer_style`;
- `personalization`;
- `factual_judgment`;
- `planning`;
- `tool_selection`;
- `tool_parameters`;
- `authorization`;
- `safety_decision`;
- `external_communication`;
- `procedure_reuse`.

Пример:

- память «пользователь любит краткость» может влиять на style, но не на safety validation;
- память «пользователь не любит ждать» не может отключить rollback или validation в tool call;
- external webpage может быть factual evidence, но не project instruction;
- Agents.md может направлять изменения проекта, но не выдавать доступ к банковскому аккаунту;
- remembered permission не заменяет свежую авторизацию для платежа.

Работа Beyond Similarity показывает, что similarity retrieval способен вызывать cross-domain leakage, sycophancy, tool-call drift и memory-induced jailbreak. Поэтому admission gate должен проверять допустимость влияния, а не только тему. [[S16]]

## 9.3. Память не является разрешением

Ни один Memory Event, Claim, Note или Procedure не может самостоятельно расширить capability агента.

Разрешение выдаётся permission system/оркестратором на основании текущего task/action request. Memory может предоставить контекст для решения, но не считается authority token.

## 9.4. Source-of-truth contracts

Для некоторых domains должна существовать явная authority policy.

Примеры:

- актуальный код → repository/worktree;
- существование файла → filesystem snapshot;
- текущий статус тестов → последний достоверный test run на соответствующем commit;
- calendar event → calendar connector;
- текущий agent process → runtime registry;
- баланс счёта → банковский источник, а не старая screenshot memory;
- текущее project decision → current-state projection, подкреплённая decisions/evidence;
- исторический вопрос → Event Ledger/Evidence, а не current projection.

Память может кэшировать состояние live source, но обязана хранить время наблюдения и revalidation policy.

## 9.5. Freshness requirement

Запрос памяти содержит требование свежести:

- historical — нужна версия на момент времени;
- current-best-known — использовать последнюю достоверную projection;
- current-verified — перепроверить authoritative source;
- timeless — свежесть несущественна;
- bounded-age — не старше заданного policy threshold.

Threshold не обязан быть глобально захардкожен. Он задаётся predicate/source contract или выводится query planner с учётом риска.

## 9.6. Missingness

Отсутствие записи не означает, что событие не происходило.

Система должна различать:

- evidence отсутствует;
- источник в этот период был выключен;
- processing задержан;
- запись была удалена;
- поиск ничего не нашёл;
- доступ запрещён;
- факт действительно отрицательный.

Это особенно важно для сенсорики: отсутствие transcript во время выключенного микрофона нельзя трактовать как тишину.

---

## 9.7. Допустимое влияние User Alignment

User-alignment projections могут влиять на:

- форму и длину ответа;
- выбор примеров;
- порядок вариантов;
- оценку соответствия вкусу;
- timing вмешательства;
- draft versus notify recommendation;
- выбор безопасного reversible experiment;
- priority между несколькими неопасными задачами.

Они не могут самостоятельно:

- изменить objective fact;
- дать доступ к секрету;
- разрешить платеж, отправку или destructive action;
- сделать external content инструкцией;
- отменить validation;
- скрыть существенный риск;
- использовать личный факт в нерелевантном ответе.

## 9.8. Over-personalization Gate

Перед использованием personal memory проверяется:

1. нужна ли персонализация для текущей задачи;
2. относится ли memory к предмету и social context;
3. не повторяет ли агент личный факт без пользы;
4. не подменяет ли preference объективный ответ;
5. не создаёт ли упоминание ощущение слежки;
6. можно ли получить качественный ответ без раскрытия remembered detail.

Если персонализация нужна только модели для выбора стиля, evidence не обязательно явно упоминать пользователю.

## 9.9. Imported Space Authority

Imported Memory Space имеет собственный trust domain.

- evidence может использоваться как проектный материал;
- claims сохраняют автора и source space;
- external procedures имеют статус candidate/quarantined;
- imported instructions действуют только в project scope и после workspace/admission policy;
- никакой импорт не расширяет capabilities;
- promotion в global/world/personal space создаёт отдельный reviewed event;
- fork сохраняет lineage исходного pack.

## 9.10. Instruction authority

Instruction Item имеет два независимых свойства:

- **source authority** — способен ли source задавать project/provider instruction;
- **action authority** — какие action channels разрешено направлять.

Например, `AGENTS.md` может быть authoritative для code conventions своего path scope, но не для доступа к home directory, внешней отправки или денег. `MEMORY.md` по умолчанию является memory source, а не instruction source, пока adapter/profile не объявил иначе.

# 10. Полный жизненный цикл записи

## 10.1. Шаг 1: Capture decision

Перед записью система определяет:

- разрешён ли источник;
- включён ли privacy mode;
- нужен ли raw capture или достаточно structured event;
- какой principal владеет данными;
- есть ли особая policy для приложения/человека/проекта;
- требуется ли локальная обработка до отправки на сервер.

Capture decision максимально детерминирован для privacy и rights. LLM может классифицировать полезность, но не может отменить запрет.

## 10.2. Шаг 2: Canonical envelope

Создаётся Memory Event с:

- orderable ID;
- source sequence;
- observed/ingested time;
- scope;
- causality;
- policy;
- evidence references;
- idempotency key.

Если raw media ещё загружается, event может ссылаться на pending object transaction, но acknowledgement final не выдаётся до durable commit минимально необходимого canonical payload.

## 10.3. Шаг 3: Integrity, dedupe и redaction

Выполняются:

- content hashing;
- duplicate detection;
- malware/content safety scan при необходимости;
- sensitive region detection;
- создание redacted rendition;
- encryption decision;
- policy inheritance.

Dedupe не должен удалять контекст события. Один и тот же screenshot, полученный в разные моменты, может означать разные эпизоды.

## 10.4. Шаг 4: Fast contextualization

Недорогие adapters добавляют native context:

- document title/section;
- repo/branch/path/symbol;
- app/window/URL;
- speaker candidates;
- project focus;
- active task;
- nearby turns;
- tool schema;
- device/location, если разрешено.

Это улучшает lexical/dense retrieval ещё до тяжёлого semantic processing.

## 10.5. Шаг 5: Event segmentation

Система обновляет текущие episode boundaries. Segmentation может быть:

- deterministic;
- lightweight model-assisted;
- revised later during consolidation.

Boundary proposal не изменяет original event order.

## 10.6. Шаг 6: Semantic proposals

Extractor создаёт proposals:

- observations;
- claims;
- entity candidates;
- relations;
- intent/trigger candidates;
- causal links;
- sensitivity/taint suggestions;
- topic/scene assignments;
- possible project links;
- candidate procedure lessons.

Proposal содержит model/prompt/schema version и evidence handles.

## 10.7. Шаг 7: Admission

Admission решает, что делать с proposal:

- принять как semantic projection;
- принять как uncertain hypothesis;
- отправить в quarantine;
- объединить с существующим claim;
- связать как support/contradiction;
- отклонить как дубликат/мусор;
- отложить до дополнительного evidence;
- запретить распространение из-за sensitivity.

Raw event не удаляется из-за отклонения semantic proposal.

### Admission factors

- directness of evidence;
- source trust;
- extraction confidence;
- novelty;
- expected future utility;
- project/domain relevance;
- potential harm;
- privacy/retention;
- contradiction;
- repetition;
- user explicitness;
- outcome evidence.

## 10.8. Шаг 8: Entity resolution

Entity resolver ищет exact IDs, aliases, scope anchors и candidate matches.

Если identity неуверенна:

- не выполнять hard merge;
- создать candidate relation;
- сохранить separate entities;
- позволить будущему evidence или пользователю разрешить.

## 10.9. Шаг 9: Conflict and state transaction

Новые claims добавляются в candidate set. Predicate resolver:

- строит новую current projection;
- оставляет видимый conflict;
- либо откладывает решение;
- либо требует authoritative revalidation;
- либо создаёт Action Inbox только если пользователь действительно нужен.

## 10.10. Шаг 10: Index updates

Derived processors независимо обновляют:

- exact/alias index;
- lexical index;
- dense index;
- late-interaction index;
- temporal index;
- graph;
- code index;
- visual/audio index;
- dossier/summary cache;
- prospective trigger registry.

У каждого update есть source watermark. Query planner знает, до какого event_id индекс актуален.

## 10.11. Шаг 11: Human views

Обновляются:

- timeline;
- dossier;
- note projections;
- project memory view;
- current state card;
- conflict card;
- memory-used UI;
- audit trace.

## 10.12. Шаг 12: Feedback loop

В дальнейшем система фиксирует:

- был ли item retrieved;
- был ли открыт full evidence;
- использовал ли агент item в answer/action;
- оказался ли он полезным;
- исправил ли пользователь;
- привёл ли он к ошибке;
- был ли query успешным;
- стоило ли сработать prospective intent.

Эти данные используются для eval и последующего обучения retrieval/admission policies.

## 10.13. Что записывать обязательно

Каноническое событие обязательно для:

- explicit user instruction;
- user correction;
- tool call и result, если они влияют на задачу;
- изменение project files/state;
- permission decision;
- external side effect;
- task outcome;
- trigger creation/firing;
- memory correction/deletion;
- system policy change;
- consequential agent decision;
- важный sensory capture, который был сохранён.

## 10.14. Что не обязано становиться semantic memory

- повторяющиеся технические heartbeat events;
- каждый token/partial ASR;
- скрытый chain-of-thought;
- одинаковые screenshots без изменения;
- ephemeral tool chatter без outcome;
- случайный background speech без связи с пользователем;
- intermediate model drafts, если они не влияли на решение.

Они могут сохраняться как telemetry с отдельной retention policy, но не должны загрязнять semantic memory.

---

## 10.15. Шаг 6A: Open extraction до schema coercion

Extractor сначала обязан вернуть faithful open description того, что он вынес из evidence. Только после этого он пытается сопоставить части с known schemas.

Это предотвращает потерю содержания из-за требования заполнить заранее определённые поля.

## 10.16. Шаг 7A: решение о semantic structuring

После сохранения observation система выбирает минимально достаточное представление:

1. оставить только evidence/episode;
2. создать Open Observation или Claim;
3. обновить Topic Dossier;
4. добавить flexible facets;
5. использовать существующую Semantic View;
6. предложить новую experimental view, если повторяющийся запрос не решается проще;
7. создать strict operational record только при наличии соответствующего системного контракта.

Этот шаг не задерживает доступность записи. Любая тяжёлая структуризация выполняется асинхронно и может быть отменена без потери знания.


## 10.17. Шаг 12A: User-alignment feedback

После communication/action outcome сохраняются:

- пользователь принял, отредактировал, отклонил или проигнорировал draft;
- отправил ли его позже и в какой форме;
- понравилось ли вмешательство;
- было ли оно слишком ранним/поздним;
- пришлось ли исправлять раскрытие информации;
- какое действие пользователь выбрал вместо предложенного;
- поменялись ли условия или preference.

Feedback создаёт evidence. Обновление Preference Trajectory и Communication Model происходит projection transaction, а не прямой перезаписью профиля.

## 10.18. Routing события по Memory Spaces

Одно событие может быть связано с несколькими spaces, но routing должен быть явным.

Пример:

- Telegram message остаётся в Personal/Social Space;
- содержащаяся в нём публичная техническая ссылка может быть предложена World Intelligence Space;
- решение по проекту может быть promoted в Project Space;
- raw personal context не копируется вместе с решением.

Routing создаёт links/promotion events с redaction и policy, а не дублирует бесконтрольно весь payload.

# 11. Время, изменение, конфликты и параллельность

## 11.1. Bitemporal model

Для значимых сущностей Dennett хранит два времени:

- **valid time** — когда утверждение было верно в мире/проекте;
- **transaction time** — когда Memory Fabric узнала и записала утверждение.

Пример:

- пользователь 1 июля изменил предпочтение;
- Dennett получил запись только 3 июля после offline sync.

Historical query «что Dennett знал 2 июля?» и «что пользователь предпочитал 2 июля?» имеют разные ответы.

## 11.2. Event time interval

События могут быть instant или interval:

- сообщение — instant;
- встреча — interval;
- editing session — interval;
- действие tool — start/end;
- состояние проекта — valid interval.

## 11.3. Epistemic states

Claim может проходить состояния:

```text
PROPOSED → ASSERTED/OBSERVED/INFERRED
           ↘ CONTESTED
           ↘ SUPERSEDED
           ↘ RETRACTED
           ↘ VERIFIED
```

`SUPERSEDED` не означает ложность в прошлом. `RETRACTED` означает, что assertion признано ошибочным. `CONTESTED` означает наличие нерешённого противоречия.

## 11.4. Predicate resolver contracts

Один global resolver невозможен. Для разных predicates используются разные contracts.

### Latest verified observation

Подходит, когда значение действительно заменяется новым наблюдением:

- текущая настройка;
- current status;
- текущая версия.

### Authoritative source

Подходит, когда есть внешняя система истины:

- repo HEAD;
- calendar event;
- live account state;
- process registry.

### State machine

Подходит для workflow:

`created → running → waiting → completed/failed/cancelled`.

Некорректный transition не принимается без correction.

### Multi-value set

Подходит, когда одновременно верно несколько значений:

- участники проекта;
- любимые жанры;
- активные branches.

### Aggregation

Подходит для статистики и summaries:

- частота ошибок;
- средний outcome;
- распределение предпочтений.

### Human override

Пользователь может явно закрепить значение. Override имеет scope и может быть снят новым explicit instruction.

### Unresolved

Если rule недостаточен, система сохраняет конфликт и abstains/asks/revalidates.

## 11.5. Почему нельзя всегда брать latest timestamp

Latest работает только когда semantics гарантируют replacement. Новое сообщение «мне нравится B» может быть контекстным, временным или относиться к другому проекту. Поэтому перед resolver нужны scope, predicate identity и context.

Исследование deterministic freshness доказывает полезность детерминированного resolver для current-value задач, но не универсальность. [[S15]]

## 11.6. Conflict Object

```yaml
conflict:
  conflict_id: id
  scope: typed_scope
  predicate_or_topic: optional
  candidate_claim_ids: []
  conflict_type: temporal | source | identity | concurrent_edit | semantic | policy
  detected_at: timestamp
  blocking: boolean
  resolution_policy_ref: optional
  resolution_events: []
  status: open | resolved | accepted_ambiguity | invalidated
```

Conflict может быть видимым в UI только если важен; мелкие machine conflicts разрешаются автоматически по policy.

## 11.7. Correction handles

Пользователь или агент должен иметь возможность сказать:

- «эта запись неверна»;
- «не тот Алексей»;
- «это относилось только к проекту X»;
- «раньше было так, сейчас иначе»;
- «не использовать это правило».

Correction может ссылаться на exact `claim_id` или semantic `claim_ref`. StateFuse показывает, что semantic correction handles важны, когда точный ID неизвестен. [[S14]]

## 11.8. Параллельные агенты

Каждый agent branch пишет события в собственный branch/correlation context. Общие правила:

- canonical append не блокирует другие branches;
- branch-local working state виден немедленно своему агенту;
- shared current projection обновляется по accepted events;
- conflicting changes сохраняются как conflict;
- project source of truth, например git, определяет фактический file state;
- merge action сама становится событием с результатом и conflict resolution.

## 11.9. Offline sync

Offline device ведёт локальный append-only operation log.

При reconnect:

1. события дедуплицируются;
2. causality/order восстанавливаются настолько, насколько возможно;
3. independent events объединяются;
4. conflicting semantic updates не перезаписывают друг друга;
5. projections пересчитываются;
6. critical conflict выносится в Action Inbox только при необходимости.

Для ordering используются source sequence, observed time, ingested time и causal parents. Wall clock не считается абсолютно надёжным.

## 11.10. Read consistency classes

### Eventual

Подходит для dossiers, graph communities, long-term summaries.

### Read-your-writes

Обязателен для текущего task/session.

### Projection-complete

Нужен, когда ответ зависит от state resolution.

### Strong authoritative read

Нужен перед high-risk external action: live source revalidation + current permissions.

# 12. Retrieval: не один поиск, а план реконструкции

## 12.1. Memory Request

Агент не должен передавать только строку запроса. Полноценный request содержит цель использования памяти.

```yaml
memory_request:
  request_id: id
  actor:
    principal_id: id
    agent_id: optional
    role: optional

  task_context:
    user_query: optional
    task_goal: text
    intended_use: answer | plan | tool_call | review | trigger_check | personalization | research | code_change
    project_id: optional
    task_id: optional
    conversation_id: optional

  temporal:
    mode: current | historical | compare | trend | any
    as_of: optional
    time_range: optional
    freshness_requirement: policy

  scope:
    allowed_projects: []
    cross_project_allowed: boolean
    principals: []
    modalities: []

  risk:
    action_risk: low | normal | high | critical
    external_side_effect: boolean

  retrieval_budget:
    latency_class: instant | interactive | deep
    max_context_tokens: integer
    max_tool_calls: integer
    max_evidence_items: integer

  requested_output:
    current_state: boolean
    history: boolean
    evidence: boolean
    contradictions: boolean
    procedures: boolean
    stable_handles: boolean
```

Project agent/runtime может автоматически заполнить большую часть полей.

## 12.2. Query classification

Query planner классифицирует запрос по нескольким независимым axes:

- exact identifier lookup;
- current state;
- historical state;
- temporal comparison;
- event/episode recall;
- multi-hop relation;
- causal explanation;
- procedural guidance;
- similar experience;
- global/topic overview;
- project code lookup;
- visual/audio recall;
- person/social context;
- prospective trigger check;
- contradiction check;
- source verification;
- preference application;
- safety/authorization context.

Один запрос может иметь несколько labels. Например, «почему мы отказались от graph-only memory и что сейчас используем?» требует historical decisions + causal trace + current state.

## 12.3. Mandatory anchors

До probabilistic retrieval система добавляет deterministic anchors:

- current project manifest;
- task goal and constraints;
- effective project instructions resolved by Instruction Graph;
- current permission/safety policy;
- directly referenced IDs/files/messages;
- recent task events;
- authoritative current state, если запрос его требует;
- user-selected pinned memory;
- required source snapshots.

Mandatory не означает «всегда весь Agents.md». Большие документы тоже могут быть section-addressed и budgeted.

## 12.4. Query expansion

Planner может построить:

- aliases/entity IDs;
- temporal expansions;
- project terminology;
- code symbols/paths;
- exact identifiers/error codes;
- subquestions;
- hypothetical descriptions;
- negative constraints;
- source filters;
- modality-specific queries.

Time-aware query expansion помогала в LongMemEval; contextualized chunk keys улучшают retrieval, потому что raw fragments часто не содержат context identity. [[S06]] [[S20]]

## 12.5. Retrieval lanes

### Lane A: exact and structured lookup

Используется для:

- IDs;
- paths;
- symbols;
- emails;
- event/claim handles;
- state keys;
- external IDs;
- exact phrases;
- timestamps.

Механизмы: key-value, aliases, field filters, exact full-text.

### Lane B: lexical/sparse retrieval

BM25/SPLADE-подобный поиск полезен для редких терминов, имён, названий, error codes и code identifiers. Embeddings не должны заменять lexical lane. [[S20]]

### Lane C: dense semantic retrieval

Используется для paraphrases, concepts, implicit similarity и open-ended recall.

Dense index содержит contextualized representation, scope/time/source fields и version.

### Lane D: late-interaction retrieval

Token-level multi-vector retrieval полезен для сложных documents, visual-text items, code и случаев, где single vector теряет fine-grained matching. ColBERTv2 показывает более качественный late interaction с 6–10x compression по сравнению с ранними вариантами. [[S32]]

Lane включается только там, где domain eval оправдывает дополнительное storage/compute.

### Lane E: temporal/state retrieval

Используется для:

- current state;
- `as_of`;
- intervals;
- update chains;
- trend;
- event ordering;
- stale/verified status.

Она работает по typed fields, а не по semantic similarity.

### Lane F: episode/scene hierarchy

Сначала извлекаются relevant Scenes/Episode summaries, затем agent открывает конкретные episodes и evidence turns. Это снижает context bloat и сохраняет возможность exact dereference. [[S12]] [[S13]] [[S26]]

### Lane G: graph and causal traversal

Используется для:

- multi-hop relations;
- dependency paths;
- person/project links;
- support/contradiction graph;
- cause/outcome;
- goal/action/result;
- lineage of decision.

Graph search не должен запускаться для каждого простого вопроса.

### Lane H: project-native retrieval

Для code project:

- repo tree;
- symbol index;
- AST/call graph;
- commit/diff history;
- tests/errors;
- issues/tasks;
- project decisions.

Для research:

- source/citation index;
- claim/evidence graph;
- hypotheses;
- source quality.

Для design:

- visual embeddings;
- OCR;
- versions;
- feedback links.

### Lane I: multimodal retrieval

Объединяет:

- visual similarity;
- OCR/full-text;
- object/entity recognition;
- audio transcript;
- speaker/time;
- cross-modal temporal neighbors;
- application/window metadata.

### Lane J: procedural/experience retrieval

Ищет validated procedures и similar causal traces с учётом:

- environment version;
- task goal;
- preconditions;
- tools;
- outcomes;
- failures;
- user/project constraints.

### Lane K: full-context fallback

Если relevant corpus мал и помещается в разумный context, full context с prompt caching может быть лучше сложного RAG. Anthropic рекомендует рассматривать такой подход для небольших knowledge bases; точный порог зависит от модели, стоимости и eval, а не является вечной константой. [[S20]]

## 12.6. Candidate fusion

Нельзя складывать cosine, BM25, PageRank и freshness как будто это одна шкала.

Рекомендуемая логика:

1. каждая lane отдаёт ranked candidates и lane-specific features;
2. применяются hard scope/ACL filters;
3. candidates deduplicate по evidence/claim/episode identity;
4. ranks объединяются через Reciprocal Rank Fusion или calibrated rank model;
5. добавляются task-conditioned features;
6. отдельный gate оценивает admissibility;
7. optional reranker сортирует допустимых candidates;
8. evidence clusters формируются для избежания десяти почти одинаковых chunks.

## 12.7. Relevance и admissibility разделены

Candidate может быть очень релевантен семантически, но недопустим:

- память другого проекта;
- private medical detail в рабочем письме;
- outdated preference;
- external prompt injection;
- poisoned success trace;
- факт без разрешения principal;
- user bias в objective research;
- preference, пытающаяся изменить tool safety parameter.

Admission Gate проверяет:

- access;
- scope;
- time/freshness;
- trust;
- taint;
- allowed influence channel;
- conflict;
- sensitivity;
- user/action intent;
- risk;
- evidence quality.

## 12.8. Раннее и позднее filtering

Часть фильтров должна применяться до search:

- principal/ACL;
- encrypted/secret partition;
- hard project policy;
- deletion state.

Часть — после candidate generation:

- semantic domain mismatch;
- inappropriate personalization;
- stale constraint;
- tool-drift risk;
- poisoned procedure suspicion.

Это одновременно снижает leakage и сохраняет recall.

## 12.9. Reranker policy

Reranker не обязателен.

Он включается, когда:

- candidate pool велик;
- query open-ended;
- domain eval показывает выигрыш;
- latency class допускает;
- candidate survival monitoring работает.

Он выключается или обходится, когда:

- exact/state query;
- candidate count мал;
- reranker version не проверена на домене;
- обнаружен collapse/diversity loss;
- high-recall forensic query;
- fallback mode.

## 12.10. Active reconstruction

Для сложного запроса retrieval — iterative process:

1. получить anchors и broad scenes;
2. сформировать evidence gaps;
3. открыть selected episodes/handles;
4. выполнить graph/temporal follow-up;
5. проверить contradictions;
6. при необходимости сделать source revalidation;
7. остановиться при evidence sufficiency или budget limit.

DRIFT Search сочетает global community context и local follow-ups; Infini Memory использует iterative evidence inspection; MemMachine routes direct/parallel/iterative retrieval. [[S11]] [[S12]] [[S21]]

## 12.11. Evidence sufficiency

Система должна оценивать не только top score, но и достаточность:

- есть ли прямое evidence;
- покрыты ли все subquestions;
- есть ли независимые источники;
- остался ли blocking conflict;
- достаточно ли свежи данные;
- требуется ли live check;
- есть ли основание для abstention.

Для high-risk action threshold выше.

## 12.12. Stop conditions

Retrieval останавливается, если:

- обязательные evidence gaps закрыты;
- marginal utility следующего шага мала;
- reached budget;
- current state authoritative;
- найден exact answer;
- дальнейший поиск требует запрещённого доступа;
- evidence остаётся недостаточным и нужно спросить/revalidate/abstain.

## 12.13. Context assembly

Context Composer разделяет содержимое на каналы:

```text
[TRUSTED SYSTEM/PROJECT INSTRUCTIONS]
[CURRENT TASK AND CURRENT STATE]
[AUTHORITATIVE EVIDENCE]
[HISTORICAL EPISODES]
[USER PREFERENCES — ADVISORY SCOPE]
[PROCEDURAL GUIDANCE — VERSION/VALIDATION]
[CONFLICTS AND UNCERTAINTY]
[UNTRUSTED EXTERNAL CONTENT — DATA ONLY]
[STABLE MEMORY HANDLES]
```

Это снижает вероятность, что найденный webpage или старая preference станет командой.

## 12.14. Context budget

Budget распределяется динамически по task type:

- exact lookup получает минимум context;
- research получает больше evidence diversity;
- code task получает project files/symbols;
- historical explanation получает chronology;
- high-risk action получает policy, current state и evidence;
- voice response получает compact answer context, а мыслительный слой может держать больше handles.

Вместо фиксированных процентов policy выбирается eval-driven и может адаптироваться.

## 12.15. Stable handles

Каждый included summary/claim/episode имеет короткий handle:

```text
[E:01J...]
[C:01J...]
[EP:01J...]
[TR:01J...]
```

Агент может вызвать `memory_open(handle)` и получить полный evidence. Working context остаётся компактным, но не теряет путь к деталям. Это следует логике indexed experience memory Memex. [[S13]]

## 12.16. Context ordering

Чтобы уменьшить long-context position effects:

- task/query повторяется рядом с evidence;
- наиболее обязательные/current items располагаются в приоритетных позициях;
- evidence группируется по subquestion;
- contradictions показываются рядом с соответствующим claim;
- длинные unrelated blocks не смешиваются;
- handles позволяют не вставлять всё сразу.

## 12.17. Memory influence log

После ответа или действия сохраняется:

- какие items были retrieved;
- какие rejected и почему;
- какие вошли в context;
- какие были opened;
- какие cited/used моделью;
- какой action был выполнен;
- outcome;
- user feedback.

Это основа observability, root-cause и обучения.

## 12.18. Retrieval cache

Cache ключуется не только query string, но и:

- principal/scope;
- project/task;
- current-state watermark;
- index versions;
- policy version;
- freshness requirement;
- model/profile.

High-risk/current queries не должны использовать stale cache без revalidation.

## 12.21. Lane M: World Intelligence Retrieval

Использует:

- source snapshots;
- release/version timeline;
- source diversity;
- contradictions;
- technology capability graph;
- past project evaluations and Causal Traces;
- live revalidation requirement.

## 12.22. Lane N: Federated Space Retrieval

Planner сначала выбирает candidate spaces по mount table, project/task scope и trust. Затем retrieval выполняется внутри spaces, а fusion сохраняет origin.

Candidate обязан нести:

- `space_id`;
- owner/trust domain;
- mount mode;
- source package version;
- local versus imported origin;
- allowed influence channels.

Cross-space rank boost не способен обойти access/admission gate.

## 12.23. Lane O: Instruction Graph Resolution

Instruction query не является semantic search по всем заметкам.

Resolver учитывает:

- active provider/runtime;
- working directory/path;
- source discovery rules;
- root-to-leaf order;
- override/concat semantics;
- lazy-load conditions;
- provider size/context limits;
- Dennett-native instruction records;
- user prompt overrides;
- conflicts и disabled sources.

Результат — Effective Instruction Set с lineage к каждому source fragment.

## 12.24. Telegram/message context reconstruction

Контекст ответа собирается в независимых каналах:

```text
[CURRENT MESSAGE AND THREAD]
[IDENTITY / RELATIONSHIP / OBLIGATIONS]
[CURRENT VERIFIED SUBJECT FACTS]
[DISCLOSURE AND SAFETY BOUNDARIES]
[RELEVANT PAST DECISIONS]
[COMMUNICATION STYLE — ADVISORY]
[REPRESENTATIVE APPROVED EXAMPLES]
[SEND/DRAFT/IGNORE POLICY]
[UNCERTAINTY]
```

Сначала строится content plan. Затем style realization. Затем отдельный Action Deliberation выбирает delivery action.

## 12.25. Context isolation для deliberation

Action Deliberation получает:

- событие/opportunity;
- current state;
- relevant user alignment slice;
- similar precedents;
- authorization policy;
- possible alternatives;
- interruption context;
- live verification, если факт изменчив.

Он не получает нерелевантную личную историю. Для важных действий recommendation и authorization выполняются разными компонентами или логическими фазами.

# 13. Working Memory и Indexed Context

## 13.1. Working Memory не равна long-term memory

Working Memory — состояние конкретной активной работы. Она нужна, чтобы агент не искал заново собственную цель и последние решения после каждого tool call.

Она содержит:

- goal;
- user intent;
- constraints;
- active plan;
- current step;
- observations;
- decisions;
- open questions;
- pending actions;
- current files/artifacts;
- memory handles;
- unresolved errors;
- permission state;
- context budget state.

## 13.2. Task Workspace

```yaml
task_workspace:
  task_id: id
  branch_id: id
  goal: text
  constraints: []
  current_state_summary: text
  plan_items: []
  decisions: []
  observations: []
  open_questions: []
  pending_actions: []
  memory_handles: []
  recent_event_ids: []
  artifact_refs: []
  permission_snapshot_ref: id
  compaction_version: string
```

## 13.3. Bounded compaction

Когда workspace растёт:

1. completed low-level actions архивируются как Causal Trace;
2. рабочий summary обновляется;
3. exact evidence остаётся под handles;
4. open constraints/decisions не теряются;
5. compaction проходит consistency check;
6. предыдущая workspace version остаётся доступна для debug/replay до retention limit.

## 13.4. Что не сохраняется как память

Скрытый chain-of-thought модели не нужен как долговременная память. Сохраняются:

- решение;
- краткое rationale;
- рассмотренные важные альтернативы;
- evidence;
- action;
- outcome;
- ошибки;
- user/reviewer feedback.

## 13.5. Branch-local context

Subagents получают:

- scoped copy/view общих constraints;
- собственный branch workspace;
- selected memory handles;
- explicit output contract.

Они не должны напрямую менять parent workspace. Они возвращают events/proposals/artifacts, которые parent agent принимает или отклоняет.

## 13.6. Project Chat Working Set

Интерактивный Codex-like project chat имеет persistent working set:

- selected model and profile;
- repo/worktree;
- current conversation thread;
- pinned project instructions;
- recently touched files/symbols;
- active decisions;
- unresolved user requests;
- current branch and tests;
- memory handles.

Это не делает чат автономным black box. Пользователь по-прежнему ведёт агента напрямую; Memory Fabric помогает сохранять continuity.

## 13.7. Commit из working memory в long-term

После meaningful boundary система создаёт:

- task/session episode;
- decisions;
- file/tool events;
- outcome;
- reusable lessons candidates;
- user feedback;
- updated current state.

Не каждый intermediate scratch item переносится.

# 14. Project-Aware Memory Profiles

## 14.1. Общий принцип

Память не должна создаваться одинаково для всех проектов. При создании/подключении проекта Dennett формирует **Memory Profile**: набор source adapters, authority contracts, indexes, retrieval priors, consolidation rules и retention policies.

Это не отдельная hardcoded система на каждый проект. Профиль компонуется из reusable capabilities и может изменяться.

## 14.2. Project Memory Manifest

```yaml
project_memory_manifest:

```

## 14.3. Code/Repository Profile

### Источник истины

Repository/worktree и связанные build/test systems являются authority для текущего кода. Memory notes не дублируют source as truth.

### Что сохранять

- repo identity и branches;
- commits/diffs;
- changed files/symbols;
- commands и results;
- tests;
- errors;
- architecture decisions;
- rejected approaches;
- dependencies;
- user instructions;
- agent actions/outcomes;
- recurring environment gotchas;
- code review comments;
- issue/PR links;
- screenshots UI;
- procedure candidates.

### Индексы

- path/symbol exact;
- lexical code search;
- code embeddings;
- AST/call/dependency graph;
- commit/diff temporal index;
- test/error index;
- decision/evidence index.

### Контекст

Для coding request planner сначала использует:

1. explicit files/symbols;
2. current repo state;
3. Effective Instruction Set из Instruction Graph;
4. recent task workspace;
5. relevant decisions/errors;
6. cross-project lessons только при явной полезности.

### Инкрементальность

Изменение commit/diff должно обновлять только затронутые symbols/chunks, а не переиндексировать весь repo.

### Проблема stale code memory

Любой retrieved code summary содержит commit/branch. Если repo изменился, summary считается historical до revalidation.

## 14.4. Research Profile

### Основные сущности

- source document/snapshot;
- citation anchor;
- claim;
- evidence;
- hypothesis;
- support/contradiction;
- source quality;
- research question;
- conclusion;
- unresolved gap.

### Authority

Research conclusion не является authority; authority — cited evidence и current external sources. Для быстро меняющейся темы query planner требует web/live revalidation.

### Retrieval

- exact citation/source;
- lexical/dense;
- claim graph;
- temporal freshness;
- source diversity;
- multi-hop evidence;
- global topic summary;
- contradiction search.

## 14.5. Design Profile

Хранит:

- screenshots/images;
- visual embeddings;
- OCR/text;
- versions;
- user reactions;
- selected/rejected variants;
- rationale;
- project constraints;
- style references;
- layout/component metadata;
- source/license/provenance.

Visual item должен иметь связь с конкретной реакцией пользователя, а не превращаться в абстрактное «ему нравится синий» без context.

## 14.6. Personal/Social Profile

Хранит episodes, claims и relations о:

- людях;
- стилях общения;
- обещаниях;
- preferences;
- routines;
- recurring context;
- boundaries;
- changes over time.

Не используется жёсткая анкета на все случаи. Структурные fields добавляются там, где они нужны для retrieval/action.

Cross-domain influence особенно ограничивается: private social/medical memory не должна автоматически влиять на objective/professional tasks.

## 14.7. Automation/Monitoring Profile

Фокусируется на:

- events;
- conditions;
- prospective intents;
- tools;
- actions;
- outcomes;
- false positives;
- cooldown;
- health state;
- last successful run;
- recurring failures.

## 14.8. Mixed Project

Проект может одновременно иметь code, research и design profiles. Query planner выбирает lanes по текущей задаче.

## 14.9. Project scope как prior, не тюрьма

Project-local memory получает высокий retrieval prior. Global/cross-project memory ищется, если:

- пользователь явно просит;
- current task требует personal preference;
- есть reusable procedure;
- система видит сильную cross-project relation;
- project agent запросил broader context;
- policy разрешает.

Cross-project candidate проходит отдельный gate и получает явную метку происхождения.

## 14.10A. World Intelligence Profile

World Intelligence Profile обслуживает постоянно обновляемое знание о внешнем мире, а не один завершённый research task.

### Основные сущности

- source feed/post/article/paper/release snapshot;
- author/organization/company;
- product/model/tool/method/dataset/benchmark;
- release/version;
- capability;
- limitation;
- license/pricing/platform/support claim;
- security incident;
- deprecation;
- adoption/use case;
- source quality and diversity;
- volatility/freshness class;
- project fit assessment;
- tool experiment and outcome.

### Lifecycle знания

```text
external signal
→ captured source snapshot
→ candidate claim
→ corroborated or disputed claim
→ current-state projection
→ stale/deprecated/historical state
→ optional project experiment
→ causal experience
```

Tweet, пост или новость являются signal/evidence, но не становятся current truth или instruction автоматически.

### Dynamic schema

Профиль имеет общую минимальную структуру, но поддерживает domain extensions. Если появляется новый класс объекта, например необычный тип agent workflow или hardware interface, он сохраняется Open Observation и может породить schema extension после повторения.

### Retrieval и применение

Перед рекомендацией или интеграцией инструмента planner объединяет:

- current verified capabilities;
- ограничения текущего проекта;
- личные trade-offs пользователя;
- прошлые эксперименты Dennett;
- known failures/security;
- version/platform compatibility;
- source diversity;
- live web/tool verification.

World Intelligence отвечает «что известно о мире», а User Alignment — «подходит ли это пользователю». Эти выводы не должны смешиваться в один неразличимый claim.

### Производные views

- Technology Radar;
- AI Release Timeline;
- Tool Capability Matrix;
- Paper/Method Map;
- Research Queue;
- Watch List;
- Project Opportunity Map;
- Tried/Rejected/Validated Tools.

## 14.10B. User Alignment / Communication Profile

Этот профиль не является анкетой. Он собирает Preference Trajectories, Communication Models, precedents и feedback по релевантным scopes.

Обязательная структура задаётся только там, где она нужна для решения:

- identity and relationship anchors;
- context/time;
- evidence;
- preference tendency and exceptions;
- allowed influence channels;
- autonomy/intervention policy;
- uncertainty;
- current resolution.

Содержание preference dimension, style, taste или social pattern может формироваться динамически.

## 14.10. Закрытие проекта

При закрытии:

- active workspace завершается;
- формируется project-phase episode;
- current decisions и reusable lessons фиксируются;
- hot index может перейти в warm/cold;
- repo остаётся authority snapshot/path;
- события и evidence не теряются;
- prospective intents проверяются на актуальность;
- sensitive temporary data обрабатывается по retention policy.

# 14A. Федеративные пространства памяти

## 14A.1. Почему одной глобальной области недостаточно

Одна физическая или логическая свалка создаёт четыре проблемы:

- гипотеза исследования может всплыть как глобальный факт;
- private social context может повлиять на рабочий проект;
- чужая project memory может получить незаслуженное доверие;
- перенос проекта потребует копировать лишнюю личную историю.

Полностью независимая memory implementation на каждый проект тоже ошибочна: она дублирует инфраструктуру, мешает cross-project learning и усложняет обновление retrieval/security.

Решение — одна Memory Fabric с множеством **Memory Spaces**.

## 14A.2. Memory Space

Memory Space — логически автономная область канонических событий, evidence, schemas, policies и projections, которая может быть локальной, внешней или монтируемой.

```yaml
memory_space:
  space_id: stable_uri_or_id
  kind: personal | world | project | research | task | shared | imported
  owner_principals: []
  trust_domain: id
  parent_or_lineage_refs: []
  canonical_event_namespace: id
  profile_refs: []
  schema_refs: []
  authority_contracts: []
  default_access_policy: policy
  influence_policy: policy
  retention_policy: policy
  encryption_policy: policy
  portability_policy: policy
  status: active | read_only | archived | detached | revoked
```

## 14A.3. Базовые spaces Dennett

### Global Personal Space

Хранит личную историю, preferences, relationships, Communication Models, goals, anti-goals, delegation и system-learning пользователя.

### World Intelligence Space

Хранит внешние sources, world claims, products, papers, tools и их изменения.

### Project Space

Хранит shareable и private project knowledge, decisions, evidence, instructions, research и Causal Traces.

### Research Space

Изолирует source corpus, hypotheses, claims и conclusions конкретного исследования. Может быть дочерним Project Space или самостоятельным.

### Task/Branch Space

Временная область конкретного agent branch. После завершения значимое знание promoted в parent space; остальное архивируется или удаляется по policy.

### Shared/Team Space

Принадлежит нескольким principals, имеет отдельную governance policy и не считается общей персональной памятью каждого участника.

### Imported Space

Монтируется из чужого repository/package и сохраняет исходного owner, trust domain и lineage.

## 14A.4. Mount semantics

Mount описывает не копирование данных, а подключение namespace к retrieval и context system.

```yaml
space_mount:
  mount_id: id
  source_space_id: id
  consumer_principal_or_project: id
  mode: read_only | read_write | fork_on_write | snapshot
  trust_level: external | reviewed | trusted
  retrieval_prior: policy
  allowed_query_classes: []
  allowed_influence_channels: []
  promotion_policy: policy
  refresh_policy: policy
  mounted_version_or_commit: optional
```

### Read-only

Consumer видит evidence/claims, но не меняет source space.

### Read-write

Используется только для явно совместно управляемого space.

### Fork-on-write

Первое изменение создаёт новое space с lineage к исходному. Это безопасный default для полученной от другого пользователя памяти.

### Snapshot

Подключается конкретная version/commit без дальнейшего автоматического обновления.

## 14A.5. Promotion между spaces

Promotion не является copy-paste summary.

```yaml
knowledge_promotion:
  promotion_id: id
  source_space_id: id
  target_space_id: id
  source_handles: []
  selected_content_or_claims: []
  redactions: []
  transformation_summary: text
  reviewer_or_policy: ref
  target_influence_policy: policy
  status: proposed | accepted | rejected | revoked
```

Примеры:

- research hypothesis становится project decision candidate;
- project experiment подтверждает global reusable procedure;
- public technical fact из Telegram предлагается World Intelligence;
- personal reaction остаётся Personal Space и лишь связывается с публичным tool claim.

## 14A.6. Federation query

Federation planner:

1. определяет primary space;
2. выбирает mounted spaces по query intent;
3. применяет access и trust до retrieval;
4. выполняет per-space lanes;
5. сохраняет origin при fusion;
6. запрещает imported instruction/procedure влиять вне разрешённого channel;
7. при promotion возвращает source lineage.

## 14A.7. Передача между пользователями

Когда пользователь B получает project repository пользователя A:

1. Project Memory Pack обнаруживается;
2. integrity/schema versions проверяются;
3. pack монтируется как `Imported Space`, а не сливается с B;
4. внешние procedures и instructions получают исходный trust;
5. B может читать human views и evidence;
6. новые изменения идут в fork или shared branch по выбранному mode;
7. promotion в личную/world memory B требует отдельного решения;
8. origin A сохраняется во всех производных claims.

## 14A.8. Cross-space deduplication

Одинаковый content hash не означает одинаковый смысловой event. Физический Evidence Object может дедуплицироваться, но каждая space сохраняет свою ссылку, context, policy и event lineage.

# 14B. Portable Project Memory Pack

## 14B.1. Решение

Память проекта должна иметь переносимую нативную форму рядом с проектом, но это **не полная копия глобальной Memory Fabric**.

Живой логический объект называется **Project Memory Space**. Его сериализованная передаваемая форма называется **Portable Project Memory Pack**.

Pack может находиться:

- в Git-репозитории;
- в обычной папке презентации/исследования;
- в архиве;
- в object storage с локальным manifest;
- в другом versioned workspace.

## 14B.2. Почему не только Git и не только сервер

### Только server/global memory

Теряет переносимость и контекст при передаче проекта.

### Вся Memory Fabric в Git

Раздувает repository, раскрывает private data, создаёт conflicts и заставляет version control хранить machine-specific indexes.

### Только Markdown notes

Удобно человеку, но недостаточно для immutable history, provenance, conflict-safe concurrency, typed current state и reliable import.

### Git Notes как основной контейнер

`git notes` умеет прикреплять данные к Git objects без изменения самих объектов и хранит историю notes в отдельных refs. Но эти refs требуют отдельной настройки fetch/display, имеют собственные merge strategies и при конфликте используют отдельный notes worktree. [[S59]]

Это полезный дополнительный adapter для commit-specific annotations, например «этот commit проверен таким тестом», но слабый default для всей памяти проекта:

- содержимое невидимо в обычном worktree и на многих GitHub-сценариях;
- обычный clone/fetch не гарантирует получение нужных notes refs без настройки;
- project-wide notes, research и instruction files неудобно привязывать к одному Git object;
- merge/rewrite semantics сложнее ожидаемого поведения обычных файлов.

### Отдельная Git branch/ref

Изолирует машинные данные от основной tree, но ухудшает discoverability: стандартные агенты, архивы, shallow clones и обычный просмотр репозитория могут её не получить. Branch/ref допускается как optional archive transport или large-history backend, но не как единственный portable representation.

### Только внешний sidecar store

Хорош для private overlays и больших media, но сам project перестаёт быть self-describing и теряет память при передаче без sidecar. Поэтому external store дополняет pack, а не заменяет его.

### Принятое решение

Default — скрытая versioned directory в обычном project tree, потому что она обнаруживается обычным filesystem scan, переносится стандартным clone/archive, просматривается через diff/PR и допускает human-readable files. Canonical portable subset состоит из immutable events/evidence и human-authored semantic artifacts. Derived search structures пересобираются у получателя. Git Notes, отдельные refs, Git LFS и external object storage остаются дополнительными adapters для специализированных данных.

## 14B.3. Рекомендуемая логическая раскладка

Название `.dennett/` является текущим baseline, но может быть заменено profile/configuration до стабилизации стандарта.

```text
project-root/
└── .dennett/
    └── memory/
        ├── manifest.yaml
        ├── ro-crate-metadata.json          # generated interoperability view
        ├── events/
        │   └── <writer-or-device>/
        │       └── <immutable-segment>.jsonl
        ├── objects/
        │   └── <content-addressed-object>
        ├── notes/
        │   ├── project-overview.md
        │   └── ...
        ├── instructions/
        │   └── ...
        ├── decisions/
        │   └── ...
        ├── research/
        │   └── ...
        ├── procedures/
        │   └── ...
        ├── schemas/
        │   └── ...
        ├── views/
        │   ├── index.md
        │   └── ...
        └── adapters/
            └── instruction-sources.yaml
```

Это логическая структура, а не окончательное требование к сериализации каждой сущности.

## 14B.4. Native manifest

Manifest описывает:

- format/profile versions;
- `space_id` и lineage;
- repository/project identity;
- event segments;
- object roots and external refs;
- included schemas/profiles;
- export classes;
- encryption/signature/integrity;
- optional content-addressed object root и Merkle provenance root;
- capability/scoped-disclosure declarations;
- instruction adapters;
- generated view versions;
- excluded/private dependencies;
- minimum consumer capabilities;
- migration history.

Manifest является native authority для состава pack. `ro-crate-metadata.json` — generated interoperability projection.

## 14B.5. Почему event log сегментирован

Один бесконечный `events.jsonl` создаёт Git merge conflicts при параллельной работе.

Default:

- каждый writer/device/agent-session пишет свой active segment;
- segment становится immutable после close;
- разные branches обычно добавляют разные files;
- merge объединяет segments без переписывания;
- causal/order восстанавливаются по event IDs, source sequence и parents;
- периодический compaction создаёт новый archive/checkpoint, не меняя прежние committed segments без отдельной migration.

Один event на файл снижает conflicts, но создаёт file explosion. Один общий log экономит files, но конфликтует. Immutable segments являются лучшим baseline-компромиссом и должны проверяться load/Git benchmarks.

## 14B.6. Что коммитится по умолчанию

- manifest и schema/profile definitions;
- immutable shareable project events;
- архитектурные и продуктовые decisions;
- human-authored notes;
- project research claims/evidence, которые разрешено передавать;
- source references/citation anchors;
- portable instructions;
- validated project procedures;
- known failures/gotchas;
- небольшие evidence objects, если license/privacy/size разрешают;
- deterministic stable views, если команда хочет видеть их на GitHub.

## 14B.7. Что не коммитится по умолчанию

- Global Personal Memory;
- Telegram/social history;
- user-private preferences, не предназначенные команде;
- secrets, tokens и credentials;
- raw private screen/audio;
- model hidden reasoning;
- runtime telemetry;
- active task scratch;
- embeddings;
- ANN/lexical indexes;
- graph database files;
- reranker caches;
- machine-local path caches;
- provider session tokens;
- transient generated views.

Private project overlay хранится в глобальном Dennett store вне repository и связывается через local mount. Gitignored `.dennett/local/` допускается только как удобный cache, но не как единственная защита private data.

## 14B.8. Export classes

Каждый item получает export class:

- `public-project`;
- `team-project`;
- `portable-private-encrypted`;
- `local-user-only`;
- `external-reference-only`;
- `non-exportable`;
- `ephemeral`.

Export policy может быть выведена, но consequential classification проходит deterministic rules и privacy scan.

## 14B.9. Большие и внешние объекты

Pack не обязан включать все bytes.

Допускаются:

- URI/PID;
- checksum;
- content type/size;
- license/access condition;
- Git LFS/object-store reference;
- optional encrypted attachment;
- fetch adapter.

Source snapshot хранится локально, если это необходимо для reproducibility и разрешено; иначе сохраняется citation/provenance и признак, что bytes отсутствуют.

## 14B.10. Human-authored и generated files

### `notes/`, `decisions/`, `instructions/`

Могут быть canonical human/agent-authored evidence documents. Изменение создаёт file-change event и новую version.

### `views/`

Являются generated projections. По умолчанию не редактируются как source of truth. Если человек правит view, Dennett предлагает:

- преобразовать diff в correction/proposal;
- создать новую canonical note;
- отменить изменение и regenerate.

Это предотвращает рассинхронизацию projection и ledger.

## 14B.11. RO-Crate и W3C PROV

Dennett-native schema остаётся основной, потому что она содержит event/state/influence semantics, специфичные для агента.

При export генерируется RO-Crate-compatible metadata graph:

- root data entity — project memory pack;
- files/objects — Data Entities;
- people, agents, software, schemas — Contextual Entities;
- `conformsTo` указывает на Dennett Project Memory Profile;
- W3C PROV mappings описывают entity/activity/agent, derivation и generation;
- PROV-AGENT-compatible extensions могут отражать prompts, decisions и agent workflows.

Это даёт стандартную переносимость metadata без подчинения hot-path памяти внешней ontology. [[S52]] [[S53]] [[S57]]

## 14B.12. Import/mount pipeline

1. обнаружить manifest;
2. проверить syntax/schema version;
3. проверить hashes/signatures, content-addressed root и Merkle provenance root, если они объявлены;
4. построить inventory без исполнения и определить запрошенный disclosure subset;
5. выполнить privacy/malware/prompt-injection scan;
6. определить trust domain, mount mode и разрешённые capability-scoped views;
7. зарегистрировать source adapters;
8. построить local derived indexes;
9. parse instruction sources отдельно;
10. показать unresolved migrations/conflicts;
11. разрешить query через project space;
12. не промотировать ничего глобально автоматически.

## 14B.13. Fork, merge и shared collaboration

### Fork

Создаёт новый `space_id`, сохраняя `derived_from` и source commit.

### Merge

Объединяет immutable event segments. Semantic conflicts сохраняются и пересчитываются; Git merge не считается автоматическим semantic resolution.

### Shared write

Разрешён только если участники используют совместимый profile и ownership policy. Иначе safer default — fork-on-write + pull request/merge.

## 14B.14. Закрытие, архив и удаление

Удаление repository не обязано удалять global links и private overlays автоматически. Dennett показывает dependency graph.

При удалении pack:

- mounted space становится unavailable/detached;
- cached indexes удаляются;
- promoted knowledge сохраняет provenance и статус source unavailable;
- external object references не удаляются без отдельной authority;
- encrypted private attachments обрабатываются по retention.

# 14C. Instruction Interoperability Layer

## 14C.1. Основное решение

Dennett-native project instructions должны жить в Memory Fabric как типизированные Instruction Items. `AGENTS.md`, `CLAUDE.md`, rules-файлы и другие conventions являются external instruction sources или provider-specific rendered views.

Dennett **не обязан** создавать `AGENTS.md` в каждом новом проекте. Если active provider может получить Effective Instruction Set через SDK/runtime, используется прямой путь. Если provider требует файл, adapter может создать managed bridge в worktree или repository по export policy.

## 14C.2. Instruction Source

```yaml
instruction_source:
  source_id: id
  path_or_uri: text
  provider_family: dennett | agents_md | codex | claude | gemini | cursor | windsurf | generic
  source_role: instruction | memory | rules | config | reference
  scope_root: path
  path_matchers: []
  discovery_order: typed
  load_mode: startup | root_to_cwd | nearest | concat | override | lazy_on_path | on_demand
  size_or_context_policy: optional
  content_hash: hash
  version_ref: commit_or_file_version
  owner_manager: id
  trust_domain: id
  status: active | disabled | conflicted | missing
```

## 14C.3. Instruction Item

Parser не делает независимую копию смысла без связи с source.

```yaml
instruction_item:
  instruction_id: stable_id
  source_fragment_ref: file_section_or_native_record
  logical_subject: text
  instruction_text_or_structured_rule: content
  scope_predicates: []
  precedence_metadata: {}
  provider_applicability: []
  influence_channels: [planning, code_change, testing, style]
  hard_enforcement_ref: optional
  owner_managers: []
  last_sync_base_hash: optional
  status: active | shadowed | conflicted | deprecated
```

## 14C.4. Instruction Graph

Graph хранит:

- sources и imports;
- instruction items;
- path scopes;
- overrides/shadowing;
- equivalent common rules;
- provider-specific additions;
- conflicts;
- rendered outputs;
- ownership;
- provenance.

Effective Instruction Set строится под `provider + workspace root + cwd + task + files`.

## 14C.5. Provider-native semantics

### AGENTS.md / Codex

- discovery начинается с global guidance, затем project root → current working directory;
- на directory выбирается один файл по provider order;
- более близкие instructions идут позже и могут override;
- `AGENTS.override.md` учитывается как отдельный provider rule;
- combined context ограничен provider setting;
- explicit user prompt имеет более высокий приоритет.

### Claude Code

- `CLAUDE.md` и `CLAUDE.local.md` из ancestor path конкатенируются;
- nested files под working directory могут загружаться при чтении соответствующих файлов;
- `.claude/rules/**` имеет собственные path scopes;
- `AGENTS.md` не читается напрямую, но может импортироваться из `CLAUDE.md`;
- auto-memory `MEMORY.md` является learned memory, а не project instruction;
- instruction text является context, а не enforcement.

### Другие providers

Adapters обязаны описывать discovery, precedence, path scope, lazy loading, imports, limits и enforcement boundary. Если semantics неизвестны, source импортируется как reference/memory, а не как trusted instruction.

## 14C.6. Почему нельзя flatten

Если все files склеить в одну Dennett note:

- потеряется nearest/override semantics;
- Claude concat будет ошибочно интерпретирован как Codex override;
- nested rule может стать глобальным;
- `MEMORY.md` превратится в instruction;
- source change нельзя будет корректно отразить;
- конфликтующие правила станут неразличимыми.

Поэтому Dennett хранит Effective Instruction Set как projection, а не как единственный файл.

## 14C.7. Discovery чужого проекта

При подключении repository Dennett ищет через configurable adapters:

- `AGENTS.md`, `AGENTS.override.md`;
- `CLAUDE.md`, `CLAUDE.local.md`, `.claude/CLAUDE.md`, `.claude/rules/**/*.md`;
- `GEMINI.md` и configured context filenames;
- `.cursorrules`, `.cursor/rules/**`;
- `.windsurfrules`, `.devin/rules/**`;
- `MEMORY.md`, `MEMORIES/**`, `.memory/**` и configured project-memory paths;
- Dennett Project Memory Pack.

Files не удаляются и не перемещаются. Каждый становится source object с исходным hash/commit.

## 14C.8. Роль MEMORY.md и произвольных memory folders

Имя файла не является достаточным основанием для instruction authority.

Default:

- provider-documented auto memory импортируется как learned memory source;
- repository `MEMORY.md`/`MEMORIES/` монтируется как Human Notes/evidence;
- если manifest или user явно объявил instruction role, соответствующие sections становятся Instruction Items;
- ambiguous imperative text остаётся data-only до adjudication.

## 14C.9. Режимы синхронизации

### Import-only — baseline для существующих файлов

Dennett отслеживает изменения и обновляет Instruction Graph, но не пишет файл.

### Managed block

Dennett владеет только отмеченной section/block. Остальной файл принадлежит человеку или другому tool.

### Managed file

Dennett владеет целым generated compatibility file. Разрешается только явным profile/opt-in.

### Suggested patch

Dennett формирует diff, а пользователь/PR принимает его. Это default reverse-sync для чужих files.

### Native-only

Instruction существует только в Dennett и подаётся provider через runtime/SDK.

## 14C.10. Ownership и three-way sync

Для каждого managed logical record хранятся:

- manager identity;
- fields/sections, которыми он владеет;
- last synchronized base;
- source hash;
- rendered hash;
- target provider;
- sync generation ID.

Алгоритм:

1. прочитать base, current source и desired render;
2. если source не менялся — применить patch;
3. если менялись разные owned sections — merge;
4. если одна logical field изменена двумя managers — создать Instruction Conflict;
5. не overwrite автоматически;
6. transfer ownership только явно;
7. записать sync event и новый base.

Это заимствует принцип managed fields, но не реализацию Kubernetes. [[S56]]

## 14C.11. Предотвращение sync loops

- export получает operation ID;
- watcher распознаёт собственный exact output;
- source hash сравнивается с rendered hash;
- generated projection не re-imported как новая independent instruction;
- mapping хранит logical instruction ID;
- повторный одинаковый patch idempotent;
- цикл или oscillation создаёт alert и переводит mapping в import-only.

## 14C.12. Upstream commit меняет AGENTS.md

1. Git/source adapter фиксирует file diff и commit;
2. parser обновляет только затронутые fragments;
3. provider-specific precedence пересчитывается;
4. Effective Instruction Set invalidates cache;
5. если fragment был human-owned — Dennett принимает source update;
6. если fragment был Dennett-managed и изменён извне — создаётся conflict;
7. project memory records why/how instruction changed;
8. agent получает новое effective set в следующей session/task boundary либо немедленно, если risk требует.

## 14C.13. Dennett memory меняет project instruction

1. изменение создаётся как native Instruction Item или correction;
2. определяется, каким providers оно нужно;
3. direct runtime injection предпочтительнее изменения repository;
4. если нужен file export, renderer выбирает target и managed scope;
5. existing human instructions не перезаписываются;
6. generated diff проходит conflict/size/smell checks;
7. commit/PR policy решает, попадёт ли изменение в repository.

## 14C.14. Root AGENTS.md — решение

`AGENTS.md` не нужен как каноническая память Dennett и не обязан существовать в native project.

Он полезен как optional compatibility bridge, если:

- проект должен работать с внешними coding agents;
- Codex запускается напрямую вне Dennett;
- есть несколько разработчиков без Dennett;
- нужны минимальные общие команды и правила.

Если создаётся, он должен быть коротким:

- essential build/test commands;
- non-negotiable common rules;
- pointer на человекочитаемый project memory index;
- provider-neutral instructions.

Полная project memory, история, исследования и dynamic state туда не копируются. Nested AGENTS создаются только для реальных path-local overrides.

## 14C.15. Compatibility bridge для Claude

Если существует human-owned `AGENTS.md` и нужен Claude, Dennett может предложить:

```markdown
@AGENTS.md

## Claude-specific instructions
...
```

или symlink, если платформа и repository policy позволяют. Existing `CLAUDE.md` не заменяется. [[S54]]

## 14C.16. Configuration smell checks

Перед export или context assembly проверяются:

- context bloat;
- duplicated rules;
- conflicts;
- skill/procedure, ошибочно помещённые в always-loaded instruction;
- vague non-verifiable guidance;
- outdated paths/commands;
- secrets;
- path scope leakage;
- provider limit truncation.

Результат не обязательно автоматически переписывает file; чаще создаётся suggested patch. [[S55]]

## 14C.17. Enforcement boundary

Instruction file формирует поведение модели, но не является hard security control.

Запреты, которые должны соблюдаться независимо от модели, реализуются permission system, hooks, sandbox, tool policy или runtime enforcement. Это соответствует и provider docs: instruction context не гарантирует enforcement. [[S54]]

# 15. Мультимодальная и сенсорная память

## 15.1. Единица памяти — synchronized event, а не отдельный media file

Фотография, transcript и screenshot полезнее вместе с контекстом:

- время;
- устройство;
- active app;
- project/task;
- участники;
- surrounding speech;
- user action;
- location, если разрешена;
- последующий outcome.

Поэтому все modalities привязываются к общей temporal event graph.

## 15.2. Capture tiers

### Ring Buffer

Краткий локальный raw buffer, который позволяет сохранить момент после detection. Он может храниться только на устройстве и автоматически перезаписываться.

### Committed Event

Значимый фрагмент, выбранный explicit capture, event detector или policy.

### Derived Semantic Memory

Transcript, OCR, observations, episode, entities, claims, project links.

### Archived Raw

Полный media segment в warm/cold storage.

## 15.3. Audio pipeline

1. Voice Activity Detection;
2. audio segmentation;
3. speaker diarization;
4. speaker identity estimation с confidence;
5. ASR;
6. punctuation/turn recovery;
7. language detection;
8. address-to-Dennett detection;
9. context/project association;
10. sensitivity detection;
11. event/episode creation;
12. semantic extraction.

### Speaker identity

Нельзя превращать similarity голоса в абсолютную аутентификацию. Хранятся:

- speaker cluster;
- candidate identity;
- confidence;
- device proximity/context;
- authentication evidence.

Опасные действия требуют дополнительных факторов независимо от memory result.

## 15.4. Screen pipeline

Приоритет источников:

1. accessibility tree/DOM/structured app data;
2. active window/app/URL metadata;
3. selected text;
4. screenshot/keyframe;
5. OCR fallback;
6. visual model description.

Screenpipe демонстрирует практичный event-driven подход: captures на app switches, clicks, typing pauses и scrolling, accessibility tree с OCR fallback, локальный search и speaker/audio pipeline. Это подходящая основа для source adapter, если конкретная версия пройдёт eval и privacy review. [[S34]]

## 15.5. Event-driven screenshot policy

Capture signals:

- смена окна/приложения;
- существенное visual diff;
- открытие нового document/page;
- error/dialog;
- user pause после действия;
- click/selection;
- copy/paste;
- explicit hotkey;
- начало/конец task;
- test UI checkpoint;
- модель видит потенциально важный state.

Похожие frames deduplicate; raw frequency динамически снижается при static screen.

## 15.6. Visual representation

Каждый committed visual object может иметь:

- original pixels;
- redacted variant;
- thumbnail;
- OCR;
- accessibility tree;
- app/window/URL;
- layout regions;
- detected objects/entities;
- visual embedding;
- text embedding;
- late-interaction representation;
- cross-modal neighbors;
- project and episode links.

## 15.7. Video

Video хранится multi-scale:

- event boundaries;
- shots/scenes;
- keyframes;
- low-resolution preview;
- transcript/audio;
- motion/activity summary;
- original cold object, если retention разрешает.

Query сначала ищет scenes/keyframes, затем dereferences нужный interval.

## 15.8. Cross-modal retrieval

Пример запроса: «Что я говорил о том интерфейсе, который сфотографировал на лекции?»

Planner:

1. ищет visual items по design/interface/lecture/time;
2. использует event temporal neighbors;
3. поднимает nearby audio transcript;
4. связывает user utterance и photo event;
5. открывает project relation;
6. возвращает photo + exact phrase + current interpretation.

## 15.9. Silence и отсутствие события

Тишина сохраняется только когда она семантически значима:

- пауза на встрече после вопроса;
- отсутствие ожидаемого ответа;
- длительное отсутствие activity в monitored workflow.

Обычные пустые audio intervals не становятся memories.

## 15.10. Privacy и per-source policies

Пользователь может задать:

- не захватывать приложение;
- только local processing;
- не сохранять raw, только event summary;
- удалить raw через N дней;
- зашифровать;
- запретить cloud;
- запретить model providers;
- выключить на время;
- исключить конкретного человека/место;
- сохранять только explicit captures.

User preference «включено по умолчанию» совместима с быстрыми privacy modes, но capture policy должна быть видимой и обратимой.

## 15.11. Sensitivity detection

Система детектирует:

- пароли/секреты;
- банковские данные;
- private keys;
- authentication screens;
- personal identifiers;
- confidential documents;
- intimate/medical content;
- third-party sensitive data.

Detection не обязательно означает удаление. Возможные действия:

- encrypt;
- store redacted search representation;
- exclude from normal embeddings;
- move to vault;
- require biometric/PIN;
- shorten raw retention;
- block propagation to external model.

## 15.12. Multimodal ground truth

Текстовое описание image полезно для search, но не заменяет original. Mem-Gallery показывает, что explicit multimodal retention и organization необходимы для долгосрочного visual memory. [[S10]]

# 16. Prospective Memory и proactive behavior

## 16.1. Почему обычной заметки недостаточно

Заметка «когда пойду в вуз, напомнить X» не должна надеяться на случайный vector retrieval. Система должна хранить future obligation как активную сущность и проверять её по сигналам.

## 16.2. Два класса triggers

### Deterministic

- timestamp/cron;
- calendar;
- file change;
- message arrival;
- agent completed;
- process status;
- geofence, если разрешён;
- external webhook.

### Semantic

- «когда снова заговорим о проекте»;
- «когда я буду идти в вуз»;
- «когда появится подходящая новость»;
- «когда дизайн будет достаточно хорош»;
- «если я снова начну повторять эту ошибку».

## 16.3. Trigger evaluation pipeline

1. incoming event формирует compact trigger query;
2. deterministic registry проверяется first;
3. semantic prospective candidates извлекаются по scope/time/entities;
4. lightweight matcher оценивает match;
5. при высокой стоимости ошибки подключается stronger verifier;
6. проверяются cooldown/expiry;
7. создаётся trigger evidence package;
8. action authorization оценивается отдельно;
9. действие выполняется, спрашивается или подавляется;
10. outcome/false alarm фиксируется.

## 16.4. Precision-recall policy

Для каждого intent задаются:

- false positive cost;
- false negative cost;
- expected urgency;
- acceptable delay;
- intervention channel;
- confidence threshold;
- repeat/cooldown policy.

Пример:

- напоминание купить молоко допускает небольшой false positive;
- команда отправить сообщение — высокий threshold;
- удаление проекта никогда не выполняется semantic trigger без fresh confirmation.

TriggerBench показывает, что prospective recall имеет собственный precision-recall trade-off и ухудшается при implicit/overloaded triggers. [[S08]]

## 16.5. Intent lifecycle

```text
CREATED → ACTIVE → CANDIDATE_MATCH → VERIFIED
                    ↘ REJECTED_MATCH
VERIFIED → NOTIFIED/ASKED/ACTION_STARTED
         → FULFILLED | FAILED | SNOOZED | EXPIRED | CANCELLED
```

## 16.6. Trigger memory и раздражение

Каждое срабатывание учитывает annoyance budget:

- сколько раз уже напоминали;
- пользователь занят;
- можно ли объединить;
- нужен ли immediate vibration;
- можно ли записать в digest;
- был ли предыдущий false alarm.

## 16.7. Learning triggers

Система может улучшать matcher на feedback:

- пользователь подтвердил, что условие наступило;
- отклонил false alarm;
- пожаловался, что Dennett не вспомнил;
- изменил формулировку;
- действие оказалось полезным/бесполезным.

Но автоматическое изменение trigger semantics versioned и проверяется на replay.

# 16A. User-Aligned Action Deliberation

## 16A.1. Почему trigger detection недостаточно

Trigger отвечает: «похоже ли, что наступило условие?». Deliberation отвечает на другой вопрос: «какая реакция в этой ситуации соответствует целям и воле пользователя?»

Особенно это важно для незапрошенных событий:

- пришло Telegram-сообщение;
- появился новый AI-tool;
- project agent заметил потенциальное улучшение;
- система увидела ошибку пользователя;
- новость касается спящего проекта;
- разговор содержит возможность, которой раньше не было в intents.

Нельзя автоматически превращать detection в notification или action.

## 16A.2. Opportunity pipeline

```text
EVENT
→ OPPORTUNITY DETECTION
→ RELEVANCE / SCOPE CHECK
→ USER-ALIGNMENT RETRIEVAL
→ LIVE CURRENT-STATE VERIFICATION
→ ALTERNATIVE GENERATION
→ DELIBERATION
→ RECOMMENDATION
→ SEPARATE AUTHORIZATION
→ EXECUTION / ASK / DEFER / NO-OP
→ OUTCOME + USER FEEDBACK
```

## 16A.3. Обязательные альтернативы

Каждый deliberation обязан рассмотреть применимые варианты:

- `do_nothing` — событие не заслуживает реакции;
- `remember_only` — сохранить, но не вмешиваться;
- `watch` — продолжить наблюдение до дополнительных evidence;
- `defer` — поднять на следующей естественной границе workflow;
- `digest` — включить в обзор;
- `draft` — подготовить результат без внешнего эффекта;
- `simulate/experiment` — проверить в reversible среде;
- `ask` — запросить выбор;
- `act` — выполнить разрешённое действие.

Отсутствие `do_nothing` считается design defect: система неизбежно станет навязчивой.

## 16A.4. Deliberation factors

Решение строится не по одной utility score, а по объяснимым факторам:

- связь с active goal;
- связь с anti-goal;
- важность результата;
- urgency и deadline;
- cost of delay;
- possible harm;
- reversibility;
- стоимость эксперимента;
- confidence в interpretation;
- confidence, что пользователь одобрит;
- similarity и различия с precedents;
- interruption cost;
- текущая занятость/состояние пользователя;
- annoyance budget;
- external side effect;
- authorization requirement;
- freshness и quality evidence;
- возможность получить больше информации дешёво.

На первой версии факторы могут быть qualitative и rule/LLM-assisted. Калиброванный learned policy допускается только после накопления feedback и offline evaluation.

## 16A.5. Timing policy

Preferred intervention points:

- завершение сообщения/диалога;
- окончание tool action;
- commit/test/build boundary;
- переход между задачами;
- открытие связанного проекта;
- запланированный digest;
- момент, когда промедление реально повышает риск.

Mid-task interruption допускается только при высокой urgency или предотвращении существенного вреда. Это соответствует наблюдению, что proactive suggestions на workflow boundaries принимаются лучше вмешательств посреди работы. [[S49]]

## 16A.6. Pre-action clarification

Clarification вызывается, когда:

- несколько plausible user intents имеют разные последствия;
- preference evidence конфликтует;
- необратимость высока;
- disclosure boundary неизвестна;
- user model new/low-confidence;
- action меняет внешний мир;
- неправильный выбор трудно откатить.

Clarification не нужен, если:

- безопасный reversible draft может закрыть неопределённость;
- ответ можно подготовить без отправки;
- действие микроскопично и соответствует устойчивому precedent;
- вопрос повторяет уже явно делегированный pattern.

PAHF поддерживает цикл clarification → preference-grounded action → feedback update, но Dennett должен выбирать clarification по value of information, а не спрашивать всегда. [[S46]]

## 16A.7. Post-action learning

Outcome update включает:

- recommendation и actual action;
- authorization path;
- user correction;
- accepted/rejected/deferred;
- edit distance для drafts;
- effect on project/world state;
- annoyance response;
- whether timing was appropriate;
- unexpected consequence;
- new exception to preference;
- whether precedent is reusable.

Один feedback не переписывает модель целиком. Он становится evidence, после чего resolver обновляет relevant trajectory.

## 16A.8. Telegram и другие сообщения

### Фаза A: понять содержание

- определить conversation/thread и участников;
- поднять nearby turns и unresolved questions;
- найти связанные обещания/projects;
- проверить current facts по authoritative sources;
- определить, чего реально спрашивает собеседник;
- построить content plan.

### Фаза B: адаптировать выражение

- выбрать relevant Communication Model;
- поднять approved exemplars того же person/channel/context;
- применить tone, length, vocabulary и idiolect;
- не копировать чувствительное содержание из exemplar;
- не упоминать remembered personal facts без необходимости.

### Фаза C: выбрать delivery action

- ignore;
- archive/read later;
- notify;
- draft;
- ask user;
- send autonomously.

### Фаза D: disclosure gate

Проверить:

- можно ли раскрывать project status/files;
- не содержит ли ответ private/secret information;
- является ли получатель тем, кем считается;
- не устарела ли social permission;
- не требует ли действие fresh confirmation.

### Фаза E: feedback

Если пользователь редактирует draft, изменение сохраняется как feedback на content, style и action policy раздельно.

## 16A.9. Условия автономной отправки

Autonomous send допускается только если одновременно:

- recipient identity достаточно уверенно установлена;
- message class и scope заранее делегированы либо precedents крайне устойчивы;
- content не содержит sensitive disclosure;
- факты current/verified;
- urgency делает задержку хуже отправки;
- текст короткий и reversible в социальном смысле;
- uncertainty ниже policy threshold;
- runtime authorization разрешает external communication.

Во всех остальных случаях предпочтительнее draft или ask.

## 16A.10. Защита от over-personalization и sycophancy

Перед финальным ответом Self-ReCheck проверяет:

- использована ли personal memory только потому, что она была retrieved;
- не выглядит ли текст как демонстрация слежки;
- не повторяется ли одна preference;
- не подменён ли объективный вывод желанием пользователя;
- есть ли более нейтральная формулировка;
- не смягчена ли важная критика ради соответствия taste.

OP-Bench показывает, что over-personalization распространена именно после подключения памяти, поэтому этот check является обязательным для direct user/social response. [[S47]]

## 16A.11. Объяснимость

Для consequential recommendation Dennett может показать:

- какое событие обнаружено;
- почему оно связано с целью;
- какие precedents найдены;
- какие alternatives рассматривались;
- почему выбрано действие или бездействие;
- что требовало live verification;
- где user model неуверен;
- почему потребовалась авторизация.

Полный private reasoning не сохраняется; сохраняется decision record и evidence handles.

## 16A.12. Safe Executable Projection в proactive behavior

Для устойчивых детерминированных patterns допускается rule:

```text
WHEN verified_condition
AND scope/validity predicates
THEN create Opportunity/Intent
NOT execute privileged action
```

Пример: новый confirmed release инструмента из watch list создаёт Opportunity Candidate. Решение об установке всё равно проходит deliberation и authorization.

# 17. Causal и Procedural Memory

## 17.1. Зачем нужна causal memory

Similarity experience retrieval часто находит похожую поверхность, но не объясняет:

- что именно вызвало успех;
- какая precondition была критична;
- какой tool version использовался;
- почему предыдущая попытка провалилась;
- применим ли опыт в новом environment.

Causal Trace сохраняет действие в контексте outcome.

## 17.2. Decision Record

Для значимого решения сохраняется:

```yaml
decision_record:
  decision_id: id
  question_or_choice: text
  selected_option: text_or_ref
  alternatives_considered: []
  rationale_summary: text
  evidence_refs: []
  constraints: []
  actor: user | agent | orchestrator | reviewer
  decided_at: timestamp
  valid_scope: typed
  reversible: boolean
  superseded_by: optional
```

## 17.3. Outcome verification

Success не определяется только словами агента «готово».

Возможные evidence:

- tests passed;
- artifact exists;
- user accepted;
- reviewer verified;
- external API returned success;
- observed state changed;
- metric improved;
- no regression on replay.

Outcome может оставаться `unknown`.

## 17.4. Experience distillation

Procedure candidate создаётся не из одного summary, а из сравнения:

- successful traces;
- failed traces;
- near-miss;
- changed environment;
- user correction;
- tool constraints;
- reviewer feedback.

ReMe использует success, failure и comparative insights; DeltaMem residual experience показывает полезность хранения общего основания и deltas между похожими ситуациями. [[S29]]

## 17.5. Residual procedures

Чтобы не дублировать почти одинаковые skills:

- base procedure хранит общий pattern;
- delta procedure описывает отличия для environment/project/tool version;
- retrieval собирает base + applicable deltas;
- frequent delta path может быть consolidated в новую version.

Это полезно для repeated project workflows и разных OS/provider configurations.

## 17.6. Procedure validation

Перед статусом validated:

- evidence links проверены;
- outcome не только self-reported;
- нет подозрительного external instruction;
- environment/version обозначены;
- procedure протестирована на replay/shadow;
- high-risk steps имеют policy gates;
- rollback описан;
- procedure не запрашивает больше прав, чем нужно.

## 17.7. Quarantine

Procedure попадает в quarantine, если:

- создана из внешнего content;
- основана на одном trace;
- содержит команды раскрытия данных;
- необычно меняет safety/tool parameters;
- success evidence слабое;
- источник poisoned/tainted;
- поведение резко отличается от прошлых versions.

## 17.8. Generated guidance

Mem-π показывает, что dedicated model может генерировать context-specific guidance лучше static retrieval. В Dennett такой слой допустим как **advisory memory synthesis**:

- он получает current context и validated traces;
- решает, нужна ли guidance;
- генерирует concise advice;
- обязан вернуть supporting handles;
- guidance не становится canonical truth;
- action agent может открыть evidence;
- high-risk действие не основывается только на generated guidance. [[S31]]

## 17.9. Anti-pattern memory

Failures сохраняются не как отдельная магическая Negative Memory, а как causal traces и procedure anti-patterns:

- failure condition;
- symptom;
- root cause;
- failed action;
- environment;
- recovery;
- prevention.

## 17.10. Model/provider experience

Memory Fabric может помнить:

- какая модель лучше справилась;
- на каком типе задач;
- с каким prompt/profile;
- latency/limits;
- tool support;
- failures;
- cost;
- date/version.

Это factual/experience evidence для оркестратора, а не вечный рейтинг модели.

# 18. Consolidation и «сон»

## 18.1. Сон не является одной ночной job

Maintenance идёт на нескольких временных масштабах.

### Continuous micro-consolidation

После каждого события:

- dedupe;
- contextualization;
- recent episode update;
- lightweight indexing;
- obvious exact state update.

### Boundary consolidation

На границе:

- завершения разговора;
- task completion;
- meeting end;
- commit/checkpoint;
- project phase;
- app/activity switch.

Создаются episode summary, decisions, outcome и pending intents.

### Idle consolidation

Когда устройство/сервер свободен:

- grouping;
- claim updates;
- dossier regeneration;
- graph linking;
- index maintenance.

### Periodic sleep

Ежедневно/еженедельно по policy:

- conflict scan;
- procedural candidate analysis;
- memory quality review;
- trigger utility review;
- source/index health;
- retention;
- eval case generation.

### Deep maintenance

Редко:

- re-embedding;
- graph rebuild;
- operational schema migration или rebuild проверенных Semantic Views;
- cold archive compaction;
- benchmark replay;
- restore test;
- deletion verification.

## 18.2. Priority selection

Сон не анализирует весь архив одинаково. Priority повышают:

- новые события;
- пользовательская коррекция;
- blocking conflict;
- failed task;
- repeated retrieval;
- high-value project;
- novelty/surprise;
- active trigger;
- security anomaly;
- stale current state;
- procedure candidate;
- index lag;
- impending retention deadline.

## 18.3. Consolidation никогда не переписывает evidence

Модель сна может:

- предложить new claims;
- supersede projection;
- regenerate dossier;
- link episodes;
- create procedure candidate;
- lower ranking weight;
- mark stale;

Но original events/evidence остаются неизменными, если нет explicit retention/delete operation.

## 18.4. Fidelity checks

Generated summary/dossier проверяется:

- все значимые statements имеют evidence handles;
- unsupported additions marked inference;
- dates/entities preserved;
- contradictions not silently removed;
- scope remains correct;
- sensitivity/influence policy inherited;
- no instruction laundering from external content.

Для важного content можно использовать NLI/attribution verifier, второй model pass или deterministic checks.

## 18.5. Incremental dossiers

Topic Dossier не переписывается с нуля при каждом событии. Возможна логика:

1. определить affected sections;
2. добавить evidence;
3. обновить current conclusion;
4. сохранить previous projection version;
5. проверить conflict;
6. update handles/index.

При накоплении drift выполняется full regeneration из claims/episodes.

## 18.6. Weights и utility

Ranking weight не является truth score. Он отражает вероятную полезность retrieval.

Факторы:

- recent use;
- successful task contribution;
- user correction/confirmation;
- project activity;
- evidence quality;
- novelty;
- redundancy;
- false-positive history;
- stale context;
- security risk;
- retrieval frequency without use.

Weight versioned и может быть пересчитан.

## 18.7. Sleep budget

Maintenance scheduler учитывает:

- provider limits;
- local compute;
- battery/device status;
- queue lag;
- importance;
- expected gain;
- last successful maintenance;
- project deadlines.

LightMem и Lightweight Agent Memory показывают пользу разделения дешёвого online processing и offline consolidation. [[S28]]

## 18.8. Memory doctor

Периодический diagnostic проверяет:

- orphan claims;
- broken evidence handles;
- index watermark lag;
- ungrounded summaries;
- duplicate entities;
- unresolved conflicts;
- stale current state;
- poisoned procedure candidates;
- scope anomalies;
- deletion incompleteness;
- projection divergence;
- restore readiness.

## 18.9. Consolidation rollback

Каждая projection version хранит:

- input watermark;
- algorithm/model/prompt version;
- output hash;
- previous version;
- quality checks.

Если новая version ухудшила eval или содержит ошибку, UI/query planner может откатиться на previous projection, не меняя ledger.

---

## 18.10. Semantic view maintenance во время consolidation

Consolidation может:

- группировать повторяющиеся Open Observations;
- обновлять Topic Dossiers;
- добавлять aliases и flexible facets;
- находить recurring query patterns;
- предложить experimental Semantic View;
- измерить её пользу против freeform baseline;
- отключить view при отсутствии выигрыша.

Consolidation не должна ночью создавать новые action-authoritative types или выполнять массовый backfill semantic memory. Operational schema migration запускается только отдельным системным изменением.


## 18.11. User Alignment maintenance

Sleep обновляет только затронутые trajectories/models:

- связывает accepted и edited drafts;
- обнаруживает preference drift;
- ищет исключения;
- снижает вес старых behavior-only inference;
- разделяет person/channel contexts;
- отмечает cross-domain interference;
- проверяет over-personalization incidents;
- пересчитывает delegation/intervention projection.

## 18.12. World Intelligence maintenance

- dedupe одинаковых новостей/release reports;
- resolve entities и versions;
- закрывает устаревшие current claims;
- формирует contradiction tasks;
- пересматривает watch list;
- связывает papers с methods/tools/projects;
- не обновляет быстро меняющийся факт без source timestamp;
- выбирает, что стоит live-revalidate, а что можно оставить historical.

## 18.13. Portable pack maintenance

- закрывает immutable event segments;
- проверяет manifest completeness;
- регенерирует optional stable views;
- обнаруживает accidental private export;
- проверяет external refs;
- compact/archive только через versioned operation;
- не коммитит изменения автоматически, если repository policy этого не разрешает.

# 19. Forgetting, retention, archival и deletion

## 19.1. Четыре разные операции

### Attention decay

Item остаётся, но реже извлекается.

### Supersession

Исторический claim остаётся, current state использует новый.

### Archival

Object перемещается в более дешёвое/медленное storage.

### Physical deletion

Содержание удаляется по запросу пользователя, policy или безопасности.

Их нельзя называть одним словом «забывание».

## 19.2. Почему старое не нужно удалять автоматически

Старое может быть полезно для:

- history;
- trend;
- project retrospective;
- price evolution;
- preference change;
- debugging;
- model/skill comparison;
- legal/audit context;
- recovery.

Поэтому возраст обычно влияет на current-use weight/freshness, а не на физическое существование.

## 19.3. Retention classes

Примеры логических classes:

- permanent-until-user-delete;
- long-term;
- project-lifetime;
- rolling-window;
- raw-expire-derived-keep;
- local-only;
- encrypted-vault;
- legal/mandatory;
- ephemeral/task-only;
- debug/telemetry.

Policy может зависеть от source, sensitivity, project и user preference.

## 19.4. Raw sensory retention

Возможная схема:

- semantic events/important captures — long-term;
- raw screenshots — configurable rolling/cold;
- duplicate/static frames — short;
- audio raw — shorter, transcript/important clips longer;
- explicitly saved photo/audio — long-term;
- sensitive raw — encrypted and possibly shorter;
- user-pinned raw — protected.

Это не фиксированные сроки, а policy framework.

## 19.5. Delete graph

Forget request строит dependency graph:

- Evidence Objects;
- Memory Events content fields;
- Observations;
- Claims;
- Episodes/Scenes;
- Current State;
- Graph edges;
- Dossiers/Notes;
- Sparse/dense/multimodal indexes;
- caches;
- procedure lessons;
- backups/replicas;
- exported artifacts, если управляются Dennett.

Система должна предотвратить resurrection из derived layers.

## 19.6. Cryptographic erase

Для encrypted objects возможно уничтожение data encryption key. Это ускоряет гарантированное erase, но metadata/tombstone и backup lifecycle всё равно должны быть учтены.

## 19.7. Deletion verification

После delete job выполняются tests:

- exact handle не открывается;
- search не возвращает content;
- related summary не раскрывает удалённое;
- procedure не содержит distilled secret;
- unauthorized principal не получает;
- backup deletion scheduled/confirmed;
- audit record создан.

GateMem показывает, что active forgetting и access control остаются слабым местом current systems. [[S09]]

## 19.8. Correction вместо deletion

Если информация просто ошибочна, предпочтительнее:

- retract claim;
- add correction;
- update current state;
- сохранить historical evidence.

Physical deletion нужна для privacy, secrets, harmful data или explicit user request.

## 19.9. User control

Пользователь может:

- удалить object/event/topic/time range/person/project;
- запретить использовать, но сохранить;
- ограничить model providers;
- сделать local-only;
- снять pin;
- изменить retention;
- запросить export;
- увидеть deletion progress.

# 20. Security and Governance

## 20.1. Память — долговременный control channel

Writable persistent memory опаснее обычного prompt injection по трём причинам:

- persistence — атака переживает сессию;
- statefulness — меняет дальнейшее поведение;
- propagation — может перейти в summaries, procedures, agents и другие проекты.

Security survey 2026 показывает, что защиту нельзя добавить только на retrieval/execution; provenance, versioning и retention должны закладываться в storage с начала. [[S17]]

## 20.2. Security lifecycle

### Write

Риски:

- malicious injection;
- ложная speaker identity;
- poisoned webpage;
- scope confusion;
- malicious agent proposal;
- secret capture.

Защиты:

- source/actor identity;
- trust/taint;
- raw evidence;
- semantic proposal quarantine;
- no direct procedure activation;
- sensitivity detection;
- signed/structured tool sources.

### Store

Риски:

- unauthorized access;
- index leakage;
- backup leakage;
- scope bugs;
- corruption.

Защиты:

- encryption;
- policy-aware partitions;
- integrity hashes;
- audit;
- independently rebuildable derived stores;
- least-privilege service access;
- backup policy.

### Retrieve

Риски:

- cross-domain leakage;
- stale memory;
- deleted memory;
- sycophancy;
- poisoned candidates;
- secret retrieval.

Защиты:

- pre-filter ACL;
- trust/admission gate;
- influence policy;
- freshness/conflict checks;
- secret broker;
- taint-aware ranking;
- provenance labels.

### Execute

Риски:

- tool-call drift;
- remembered permission;
- dangerous parameter;
- poisoned procedure.

Защиты:

- fresh action authorization;
- deterministic high-risk rules;
- memory cannot grant rights;
- tool parameter validation;
- execution preview/rollback;
- procedure validation.

### Share/Propagate

Риски:

- secret copied to project;
- poisoned summary becomes global;
- cross-agent contamination;
- external model exposure.

Защиты:

- propagation policy;
- taint inheritance;
- scope-aware summaries;
- provider restrictions;
- redacted renditions;
- explicit declassification.

### Forget/Rollback

Риски:

- content survives in cache/index;
- summary resurrection;
- backup persistence;
- procedure retains secret.

Защиты:

- delete graph;
- tombstones;
- cryptographic erase;
- projection/index purge;
- verification tests;
- rollback versioning.

## 20.3. Trust domains

Примерные domains:

- system-trusted;
- user-explicit;
- project-trusted;
- verified-tool;
- local-sensor;
- agent-generated;
- external-web;
- external-message;
- imported-project;
- learned-procedure;
- quarantined;
- secret-vault.

Domain — не единственный score. Он определяет default policies и allowed influence.

## 20.4. Taint propagation

Если summary построен из untrusted webpage, он не становится trusted только потому, что его написал Dennett.

Derived entity наследует:

- source taint;
- sensitivity;
- access restrictions;
- instruction prohibition;
- retention dependencies.

Verification может повысить epistemic quality, но declassification — отдельная явная операция.

## 20.5. Instruction laundering

Особый риск: external content содержит инструкцию, extractor превращает её в note/procedure, а позже агент воспринимает как собственный опыт.

Защита:

- external content маркируется data-only;
- imperative sentences не становятся procedure без causal outcome и validation;
- procedure candidate показывает source lineage;
- suspicious permission/safety changes блокируются;
- project/imported instructions отделены от user/system instructions;
- memory context channel явно подписан.

## 20.6. Environment-injected poisoning

Исследование eTAMP показывает, что одна manipulated webpage observation может отравить memory и сработать позже на другом сайте; более сильная модель не гарантирует защищённость. [[S18]]

Для Dennett:

- webpage observation не создаёт global procedure;
- cross-site transfer требует evidence of generality;
- environment-specific lesson сохраняет domain/version;
- future retrieval учитывает source site/taint;
- consequential reuse требует verifier;
- user frustration не должна снижать gate.

## 20.7. Poisoned experience

MemoryGraft и sleeper memory poisoning показывают, что вредный опыт может выглядеть успешным и активироваться позже. [[S19]]

Поэтому:

- success не равен trust;
- outcome проверяется объективно;
- procedure использует несколько traces;
- anomaly detection ищет unusual commands/permissions;
- delayed activation tests входят в replay;
- quarantine не снимается только по similarity.

## 20.8. Secret Vault

Secret Vault отделён от обычной памяти.

Memory Fabric хранит:

- факт существования секрета;
- purpose/scope;
- capability handle;
- audit;
- не сам secret в ordinary context.

Агент вызывает capability:

```text
use_secret(secret_handle, allowed_action, target, task_id)
```

Broker проверяет policy и выполняет действие без раскрытия bytes, если возможно.

Если secret должен быть записан в SSH/config file, это отдельная controlled action с target, permissions и audit.

## 20.9. Cross-project leakage

Project memory query по умолчанию:

- ограничен project scope;
- может искать global user preferences/advisory procedures;
- не получает private other-project data без explicit relevance и policy;
- маркирует cross-project origin;
- не переносит project instructions.

## 20.10. Cross-principal/shared memory

Если Dennett когда-либо обслуживает нескольких людей/семью/команду:

- principal ownership обязателен;
- relationship/context authorization динамичны;
- shared item не означает universal access;
- delete одного principal учитывает joint ownership;
- answer construction не должен выдавать existence secret item;
- GateMem-like suite обязательна. [[S09]]

## 20.11. High-risk memory mode

Перед платежом, удалением, отправкой сообщения, раскрытием секрета или системным действием retrieval policy ужесточается:

- only authoritative/current sources;
- no advisory preference override;
- conflict blocking;
- fresh permission check;
- exact target verification;
- preview;
- provenance shown;
- strong user confirmation where required.

## 20.12. Memory firewall

Memory Firewall — boundary между retrieved candidates и agent context.

Он выполняет:

- ACL;
- trust/admissibility;
- instruction stripping/labeling;
- influence restriction;
- secret filtering;
- taint propagation;
- risk-conditioned thresholds;
- audit of included/rejected items.

MemGate может быть кандидатом на один из lightweight semantic gates, но Dennett всё равно нуждается в deterministic policy layers вокруг него. [[S16]]

---

## 20.13. Imported Project Pack Threat Model

Imported pack может содержать:

- poisoned procedure;
- malicious instruction;
- fake provenance;
- secret другого пользователя;
- crafted Markdown/import;
- huge object/zip bomb;
- schema extension, расширяющую influence;
- symlink/path traversal;
- executable attachment.

Mount pipeline должен:

- inventory без исполнения;
- canonicalize paths;
- запрещать выход за root;
- limit size/count/decompression;
- verify hashes/signatures, если доступны;
- маркировать всё как external-shared;
- отделять instructions от evidence;
- quarantine procedures;
- не загружать external secrets в обычные indexes;
- показывать workspace trust summary.

## 20.14. Instruction source threat model

Provider files могут содержать prompt injection по дизайну, поскольку они являются инструкциями. Их authority ограничивается:

- repository/project scope;
- provider/path scope;
- allowed influence channels;
- current workspace trust;
- system permission boundary.

Instruction source не может выдать себе доступ за пределы task/project, даже если текст утверждает обратное.

## 20.15. Semantic-structure poisoning

Атакующий может попытаться заставить систему создать facet, relation, Semantic View или operational record, который затем повышает вероятность вредного retrieval либо незаметно расширяет влияние внешнего содержимого.

Защита:

- Open Observation сохраняется как evidence, но не получает instruction/authorization authority;
- reusable facet или Semantic View создаётся только из grounded observations и повторяющейся доказанной потребности;
- operational type меняется только через отдельный управляемый процесс;
- source trust/diversity и evidence lineage обязательны;
- no automatic influence expansion;
- новые views/projections сначала работают в shadow/quarantine;
- semantic-structure growth и retrieval influence наблюдаются;
- replay attack cases обязательны;
- любую view/facet можно отозвать и пересобрать projections без потери evidence.

## 20.16. User-model poisoning

External content и чужая речь не должны напрямую формировать preference пользователя. Preference evidence classes:

1. explicit user statement;
2. observed user choice;
3. accepted/edited action;
4. repeated behavior;
5. third-party claim — не принимается без user evidence.

Собеседник не может написать «пользователь всегда разрешает мне получать исходники» и превратить это в Delegation Policy.

## 20.17. Privacy export audit

Перед созданием/share Project Memory Pack проверяются:

- principals;
- private/social sources;
- secret/sensitive data;
- license/access restrictions;
- hidden metadata;
- generated summaries, содержащие private derivation;
- external object permissions;
- Git history risk;
- recipient policy.

Удаление sensitive item из текущего tree не гарантирует его исчезновение из Git history. Export system должна предупреждать и при необходимости создавать очищенный pack/archive вместо обычного commit.

# 21. Масштабирование, распределённость и производительность

## 21.1. Что означает масштабирование для Dennett

Система должна выдерживать:

- годы текстовых событий;
- многие проекты и repositories;
- continuous screen/audio metadata;
- media archive;
- миллионы observations/claims;
- множество embeddings и их versions;
- concurrent agents;
- server + PC + laptop + phone;
- offline periods;
- reindex/rebuild;
- deletion and backup.

## 21.2. Tiered storage

### Hot

- active task/session workspace;
- recent events;
- current project state;
- current projections;
- high-use indexes;
- active prospective intents.

### Warm

- semantic projections;
- episodes/scenes;
- graph;
- standard evidence text;
- repository history indexes;
- frequently used media renditions.

### Cold

- raw audio/video;
- old screenshots;
- archived project artifacts;
- historical index snapshots;
- low-use evidence.

### Offline/Deep archive

- encrypted backups;
- rarely needed raw media;
- recovery snapshots.

Cold data остаётся логически searchable через deep retrieval, но latency выше.

## 21.3. Partitioning

Canonical events можно логически partition по:

- principal;
- project;
- time bucket;
- source/device;
- sensitivity;
- tenant, если появится multi-user.

Partitioning не должно уничтожать global timeline: cross-partition query выполняет federated planner.

## 21.4. Derived stores независимы

Возможные logical stores:

- event log;
- object store;
- state/projection store;
- lexical index;
- vector index;
- graph store;
- code index;
- multimodal index;
- trigger registry;
- audit store.

Конкретная архитектура будет выбрана позже. Бизнес-инвариант: каждое derived store можно пересобрать и временно обойти.

## 21.5. Incremental indexing

Indexer обрабатывает только новые/изменённые entities:

- new event;
- affected episode;
- changed claim;
- touched code symbol;
- changed dossier section;
- new media object.

Full rebuild нужен только при schema/model migration или corruption.

## 21.6. Index watermarks

Каждый индекс сообщает:

- latest canonical event processed;
- processing version;
- model version;
- error/lag;
- partition coverage.

Query result содержит freshness of index. Recent overlay закрывает lag.

## 21.7. Embedding migration

При смене embedding model:

1. старый index остаётся read-active;
2. новый index строится параллельно;
3. queries могут dual-read;
4. eval сравнивает;
5. traffic переключается постепенно;
6. старый index архивируется после acceptance;
7. canonical evidence не мигрирует.

## 21.8. Large-scale ANN

Исследования FreshDiskANN и SPANN показывают, что billion-scale ANN с SSD и streaming updates технически достижим. Это означает, что vector capacity не является фундаментальным пределом, но качество Dennett всё равно зависит от query planning, scope, temporal logic и evidence reconstruction. [[S33]]

Не следует заранее привязываться к одному ANN algorithm; выбор делается на стадии архитектуры и benchmark-а hardware/workload.

## 21.9. Storage amplification

Нужно измерять:

- raw bytes;
- normalized text;
- thumbnails/keyframes;
- sparse index;
- dense vectors;
- late-interaction vectors;
- graph edges;
- projection versions;
- backups.

Каждый optional representation имеет retention/ablation. Например, late-interaction index для всех screenshots может оказаться слишком дорогим и использоваться только для active projects.

## 21.10. Media efficiency

- perceptual hash dedupe;
- delta encoding/static screen suppression;
- variable keyframe rate;
- low-resolution previews;
- raw cold storage;
- separate OCR/text indexes;
- user-pinned originals;
- project lifecycle tiering.

## 21.11. Query latency classes

### Instant

Примеры: current task state, exact handle, active trigger.  
Использует deterministic stores/cache/recent overlay.

### Interactive

Примеры: project chat, personal recall.  
Использует standard hybrid lanes, optional lightweight rerank.

### Deep

Примеры: расследование истории проекта, causal analysis, old sensory search.  
Использует iterative retrieval, cold dereference, graph exploration, source revalidation.

Точные SLA выбираются после benchmark-а; документ задаёт классы, а не обещает необоснованные миллисекунды.

## 21.12. Cost-aware planning

RAG-Stack подчёркивает, что retrieval quality и performance имеют много совместных knobs и нуждаются в планировании. [[S36]]

Planner учитывает:

- latency budget;
- token budget;
- local vs remote models;
- provider limits;
- storage tier;
- expected utility lane;
- action risk;
- current load;
- cached results;
- index health.

## 21.13. Backpressure

При потоке sensor events система должна:

- никогда не блокировать critical canonical events;
- снижать capture detail;
- postpone heavy extraction;
- prioritize explicit user captures;
- drop redundant telemetry по policy;
- показывать queue lag;
- не утверждать, что memory fully processed.

## 21.14. Degraded modes

### Dense index unavailable

Exact, lexical, state, graph и recent events продолжают работать.

### Graph unavailable

Lexical/dense/episode retrieval работает; multi-hop quality ниже и обозначается.

### Reranker unavailable

Используется fused rank fallback.

### LLM extractor unavailable

Canonical writes продолжаются; deterministic/contextual indexes обновляются; semantic proposals ждут.

### Object store cold unavailable

Summary/metadata видны, но exact evidence помечено unavailable.

### Sync conflict

Conflict сохраняется; current projection может abstain.

### Corrupted index

Index quarantined и rebuild from ledger.

## 21.15. Backup and restore

Backup target:

- canonical event log;
- evidence objects;
- policies/keys metadata;
- schema definitions;
- optional derived snapshots для быстрого восстановления.

Restore test должен доказывать, что derived indexes можно rebuild и stable handles сохраняют identity.

## 21.16. Local/offline operation

На устройстве:

- local append log;
- local object buffer;
- minimal exact/working index;
- local privacy processing;
- optional small-model extraction;
- later sync.

Server merge не делает last-write-wins для semantic state; он принимает events и пересчитывает projections.

## 21.17. Phase-aware scheduling и amortization

Memory workload состоит как минимум из трёх разных фаз:

- construction/write enrichment;
- retrieval/reconstruction;
- generation/action.

Они должны измеряться отдельно. Системное профилирование agent-memory подходов показывает, что выигрыш одной фазы часто оплачивается другой, а оптимальная схема зависит от query volume, freshness target и capability используемых моделей. [[S39]]

Scheduler должен учитывать:

- сколько раз тема или проект, вероятно, будет запрошен снова;
- насколько важна немедленная свежесть;
- можно ли перенести enrichment в idle time;
- сколько стоит каждый projection и index;
- какой минимальный model capability нужен для extraction/consolidation;
- не дешевле ли на малом corpus использовать full-context или direct evidence scan;
- окупится ли dossier/graph construction будущими reads.

Пример: одноразовый web research может сохранить evidence и lightweight index без глубокого dossier. Активный многолетний project получает incremental scene hierarchy, code index, current-state resolvers и регулярно используемые dossiers.

## 21.18. State-trajectory health

Помимо latency и throughput система должна измерять здоровье эволюции памяти:

- долю revisions, которые корректно supersede старое, не уничтожая history;
- unbounded growth по типам projection;
- claims без evidence;
- stale current-state keys;
- забытые deletion obligations;
- orphaned objects и projections;
- scope drift;
- число unresolved conflicts;
- semantic churn: сколько раз projection переписывается без новой evidence;
- read-only memories, которые никогда не превращаются в полезное действие или ответ.

Такой мониторинг следует из идеи, что correctness памяти является свойством state trajectory, а не отдельных records. [[S40]]

---

## 21.19. Масштабирование Memory Spaces

Space является естественной единицей partitioning, но не обязательно физической database.

Возможные partitions:

- principal;
- project/space;
- event time;
- source modality;
- trust domain;
- storage tier.

Query не должен fan-out по всем spaces. Space selector использует current project, actor, task, mounts и query class.

## 21.20. Масштабирование portable event segments

Параметры segment policy:

- maximum events/bytes;
- maximum duration;
- close on task/commit/session boundary;
- writer/device identity;
- checksum and parent segment;
- compression after close;
- Git-friendly filename/order.

Benchmark должен сравнить:

- one-event-per-file;
- per-session JSONL;
- per-device rolling segments;
- embedded database snapshot plus export log;
- pack compaction.

Default выбирается по merge rate, Git performance, import latency и corruption recovery, а не эстетике directory tree.

## 21.21. Derived indexes portable space

После mount потребитель строит собственные:

- exact/FTS indexes;
- dense/late-interaction indexes;
- graph projection;
- current-state caches;
- dossiers;
- instruction effective-set cache.

Manifest хранит schema/model versions, но не требует переносить index bytes. Optional prebuilt index может существовать как accelerator, однако consumer обязан уметь его отвергнуть и rebuild.

## 21.22. Schema Registry scale

Schema Registry должен:

- namespaces per domain/space;
- aliases и canonical IDs;
- semver/compatibility;
- usage counts;
- proposal queue;
- deprecated/revoked state;
- migration progress;
- ability to keep long-tail extensions local to one space.

Не каждый novel facet становится global type. Большинство остаётся project/space extension или open facet.

## 21.23. Instruction sync performance

File watcher и parser работают инкрементально:

- content hash не изменился → no-op;
- изменён один section → reparse fragment;
- commit changed path scope → invalidate affected effective sets;
- provider config changed → rebuild only corresponding provider view;
- sync conflict не блокирует ordinary project memory queries.

## 21.24. World Intelligence growth

World memory растёт быстрее personal semantic memory. Поэтому:

- raw feeds имеют shorter hot retention;
- deduplicated source snapshots и important claims сохраняются дольше;
- repeated syndication не создаёт independent truth;
- product/release timeline partitioned by entity/time;
- stale claims остаются historical, но получают low current-retrieval prior;
- long-tail sources уходят в cold storage;
- live fetch используется вместо бессмысленного постоянного копирования всего Web.

# 22. Инструменты Memory Fabric для агентов

Агенты не получают прямой write-доступ к graph, state table или index. Они работают через контрактные операции.

## 22.1. `memory_record_event`

Назначение: записать observation/action/result/correction.

Обязательные поля:

- event kind;
- scope;
- evidence/content;
- source/actor;
- causality/task;
- sensitivity hint.

Сервис назначает canonical IDs и policies.

## 22.2. `memory_attach_evidence`

Добавляет media/document/tool artifact к event. Поддерживает content hash, encryption и rendition policy.

## 22.3. `memory_propose_claim`

Агент предлагает semantic claim с supporting handles. Не гарантирует принятия или current-state update.

## 22.4. `memory_correct`

Создаёт correction/retraction/scope/time/entity patch event.

Пример:

```yaml
operation: correct
claim_ref: "пользователь предпочитает пустой минимализм"
correction: "Относилось только к экрану настроек; глобальный вывод неверен"
evidence_refs: [user_message_handle]
```

## 22.5. `memory_query`

Принимает полноценный Memory Request и возвращает:

- context summary;
- current state;
- evidence items;
- conflicts;
- procedures;
- stable handles;
- provenance;
- admissibility labels;
- sufficiency status;
- query trace ID.

## 22.6. `memory_get_current_state`

Детерминированный запрос по typed key/scope с resolver explanation.

## 22.7. `memory_get_history`

Возвращает timeline/as-of changes и evidence.

## 22.8. `memory_open`

Dereference handle в полный evidence/episode/trace, если access policy разрешает.

## 22.9. `memory_follow`

Идёт по typed relations/causal path с budget.

## 22.10. `memory_search_multimodal`

Поддерживает text/image/audio/time/app/project query и возвращает cross-modal events.

## 22.11. `memory_create_intent`

Создаёт prospective intent. Опасные action specs требуют policy validation.

## 22.12. `memory_record_outcome`

Связывает task/action с verification и user/reviewer feedback.

## 22.13. `memory_explain_context`

Показывает:

- что было retrieved;
- что включено;
- что rejected;
- почему;
- какие policies применены;
- какие indexes использовались;
- какие conflicts/freshness limits есть.

## 22.14. `memory_request_broader_scope`

Агент объясняет, зачем нужен cross-project/global/secret context. Оркестратор/policy решает.

## 22.15. `memory_forget`

Создаёт scoped deletion request с preview и status. Физическое удаление выполняет governance service.

## 22.16. `memory_pin` / `memory_unpin`

Меняет retrieval priority/retention, но не truth.

## 22.17. Agent write rules

Агент обязан:

- указывать evidence;
- не маскировать inference как observation;
- не записывать external instruction как procedure;
- не менять scope произвольно;
- не сохранять secret в обычном тексте;
- не создавать current state напрямую;
- не удалять memory через обычную correction;
- фиксировать outcome значимой работы;
- использовать exact handle при correction, если доступен.

---
## 22.18. `memory_record_open_observation`

Сохраняет evidence-grounded freeform insight без принудительного выбора known schema.

Параметры:

- evidence handles;
- scope/space;
- natural-language insight;
- optional known/extension facets;
- unmodeled aspects;
- expected utility.

## 22.19. `memory_propose_schema`

Создаёт Semantic Structure Proposal. Не публикует schema автоматически.

Требует:

- supporting observations;
- expected use cases;
- proposed scope;
- alternatives/overlap check;
- migration/eval plan.

## 22.20. `memory_get_user_alignment`

Возвращает task-specific slice:

- relevant Preference Trajectories;
- goals/anti-goals;
- Communication Model;
- precedents;
- delegation/intervention policy;
- uncertainty;
- allowed influence channels.

## 22.21. `memory_deliberate_action`

Получает Opportunity/Intent и формирует alternatives + recommendation. Он не исполняет action и не выдаёт permission.

## 22.22. `memory_record_user_feedback`

Раздельно записывает feedback на:

- content correctness;
- style;
- timing;
- autonomy choice;
- disclosure;
- final action;
- outcome.

## 22.23. `memory_mount_space` / `memory_unmount_space`

Регистрирует portable/imported/shared space с mode, trust и policy. Mount не копирует все items в global ledger.

## 22.24. `memory_fork_space`

Создаёт новый writable space с lineage к source snapshot/version.

## 22.25. `memory_promote`

Предлагает или выполняет governed promotion между spaces с redaction, evidence refs и target influence policy.

## 22.26. `memory_export_project_pack`

- выбирает export classes;
- строит dependency graph;
- выполняет privacy/license audit;
- закрывает event segments;
- создаёт manifest;
- при необходимости генерирует RO-Crate/PROV view;
- возвращает completeness/omission report.

## 22.27. `memory_import_project_pack`

Выполняет inventory, validation, trust assignment, migrations и mount. Не активирует procedures/instructions автоматически.

## 22.28. `memory_resolve_instructions`

Вход:

- provider/runtime;
- workspace root;
- cwd/files;
- task;
- source policy.

Выход:

- Effective Instruction Set;
- ordered source fragments;
- conflicts;
- truncation/limit warnings;
- enforcement refs;
- lineage.

## 22.29. `memory_sync_instruction_source`

Режимы:

- import-only;
- managed-block;
- managed-file;
- suggested-patch;
- native-only.

# 25. Сквозные сценарии

## 25.1. Пользователь меняет предпочтение

1. Ранее есть Claim A: «пользователь любит пустой минимализм».
2. Новый explicit user statement создаёт Event + Evidence.
3. Extractor предлагает Claim B с scope «desktop app».
4. Resolver видит, что A был слишком общим, а B более конкретен.
5. A не удаляется; может быть contested/scope-corrected.
6. Current State для desktop design использует B.
7. Dossier обновляется.
8. Старый project historical query всё ещё видит A как прошлый контекст.
9. Новый design-agent получает B и conflict note, если релевантно.

## 25.2. Project agent меняет код

1. User prompt записан как instruction event.
2. Workspace фиксирует goal.
3. Agent читает repo authority state.
4. Tool calls и diffs записываются.
5. Tests создают verification events.
6. Commit становится checkpoint.
7. Causal Trace связывает prompt, decisions, changes, tests и outcome.
8. Project current state обновляется из repo, не из prose summary.
9. Summary/dossier хранит ссылки на commit/diff.
10. При следующем вопросе retrieval сначала смотрит current branch/commit.

## 25.3. Два агента принимают разные решения

1. Branch A и B создают разные decision records.
2. Ledger сохраняет обе ветки.
3. Если они меняют разные файлы, merge может пройти.
4. Если semantic/current decision конфликтует, создаётся Conflict Object.
5. Оркестратор/reviewer сравнивает evidence/outcomes.
6. Resolution event выбирает или объединяет.
7. Не выбранная альтернатива остаётся history/rejected approach.

## 25.4. Пользователь спрашивает «почему?»

1. Planner классифицирует causal explanation.
2. Из current state находится decision.
3. Graph идёт к decision record, constraints и alternatives.
4. Открываются exact evidence и outcome.
5. Ответ показывает reason + источники + что изменилось после.

## 25.5. Старый screenshot

1. Query содержит визуальное описание и approximate time.
2. Planner использует OCR/lexical + visual + app/time.
3. Находит Scene/episode.
4. Открывает keyframe и nearby audio.
5. Возвращает original/redacted screenshot, phrase и project link.

## 25.6. Semantic trigger «когда я пойду в вуз»

1. Intent active.
2. Sensor events показывают time/location/movement/voice context.
3. Matcher создаёт candidate match.
4. Verifier проверяет context.
5. Cooldown/expiry/annoyance policy.
6. Напоминание срабатывает.
7. User feedback учит matcher.

## 25.7. Poisoned webpage

1. Webpage stored as external-untrusted evidence.
2. Imperative text remains data-only.
3. Extracted «procedure» quarantined.
4. Поздний task semantic search может найти факт, но Memory Firewall не разрешает странице менять tool permission.
5. Если attempted cross-site guidance обнаружена, alert и poisoning case для replay.

## 25.8. Удаление чувствительного screenshot

1. User issues forget request.
2. Delete graph находит raw, OCR, visual vector, episode snippet, dossier mention, cache и backups.
3. Content erased/key destroyed.
4. Derived summaries regenerate без содержания.
5. Tombstone сохраняет факт удаления.
6. Verification queries не находят data.

## 25.9. Модель consolidation ошиблась

1. New dossier version содержит unsupported claim.
2. Attribution check или user correction обнаруживает.
3. Projection rollback.
4. Claim retracted/correction event.
5. Extractor/prompt version flagged.
6. Raw evidence остаётся, поэтому rebuild возможен.

## 25.10. Index outage

1. Dense index недоступен.
2. Planner использует exact/BM25/state/episodes.
3. UI показывает degraded retrieval.
4. Index rebuild asynchronously.
5. Canonical writes продолжаются.

## 25.11. Текущая цена

1. Memory находит historical prices.
2. Query freshness = current-verified.
3. Historical snapshot не используется как current answer.
4. Live tool/web source revalidates.
5. New observation сохраняется с timestamp/source.
6. Historical trend остаётся доступен.

## 25.12. Пользователь просит объективную оценку

1. Planner marks objective mode.
2. Personal taste memories допускаются только как separate «соответствие вашему вкусу», но не как factual evidence.
3. Sycophancy-prone memories gate rejects from factual judgment.
4. Ответ разделяет objective assessment и personalized fit.

## 25.14. Telegram: вопрос о статусе проекта

Преподаватель спрашивает: «Работа готова?»

1. social memory определяет человека и обещания;
2. project relation находит нужный repository;
3. текущий status берётся из repo HEAD, test run и task runtime, а не вчерашней summary;
4. memory reconstructs обещанный срок;
5. ответ формируется: factual content отдельно, formal tone отдельно;
6. если статус неоднозначен, Dennett задаёт clarification проектному агенту или пользователю;
7. отправка требует policy, потому что ответ создаёт внешнее обязательство.

## 25.15. Telegram: просьба отправить исходники

1. relationship memory может показать, что пользователь доверяет человеку;
2. это влияет на recommendation, но не является permission;
3. disclosure gate проверяет project visibility, secrets, license и previous explicit decisions;
4. possible actions: отказ, уточнение, sanitized archive, link на public repo, спросить пользователя;
5. автономная отправка файлов не выполняется на основании одного similarity precedent.

## 25.16. Оркестратор обнаружил новый AI-tool

1. World Intelligence source adapter фиксирует release/tweet/article как external signal;
2. candidate claims связываются с product entity и version;
3. система замечает relation с проблемой активного проекта и создаёт Opportunity Candidate;
4. Action Deliberation рассматривает no-op, watch, digest, research, sandbox experiment, ask, integrate;
5. поднимаются user preference «готовое лучше самописного, если не хуже», project constraints и прошлые tool experiments;
6. mutable facts перепроверяются live;
7. если инструмент сырой — watch/digest;
8. если эксперимент дешёв и reversible — отдельная branch/task;
9. integration всё равно требует project agent/review и permissions;
10. outcome пополняет Tool Experience Matrix.

## 25.17. Изолированное исследование

1. создаётся Research Space, дочерний Project Space;
2. sources, claims, hypotheses и contradictions остаются local;
3. несколько agents пишут branch-local events;
4. researcher conclusions не становятся global truth;
5. accepted project conclusion promoted с evidence handles;
6. general reusable finding может быть proposed в World Intelligence;
7. private project constraints не переносятся вместе с public claim;
8. закрытие исследования создаёт portable research subset при необходимости.

## 25.18. Передача проекта другому пользователю

Пользователь A отправляет repository пользователю B.

1. B клонирует repository;
2. Dennett обнаруживает `.dennett/memory/manifest.yaml`;
3. pack проходит integrity/privacy/import scan;
4. монтируется read-only или fork-on-write как Imported Space;
5. local indexes rebuild;
6. B видит project history, decisions, research и instructions;
7. personal notes A отсутствуют либо encrypted по отдельной договорённости;
8. B создаёт fork space при первых изменениях;
9. pull request/merge объединяет event segments, а semantic conflicts остаются видимыми;
10. B не наследует автоматически delegation/autonomy пользователя A.

## 25.19. Чужой repository с вложенными AGENTS.md

1. discovery adapter находит root и nested files;
2. каждый source сохраняет path, hash и provider semantics;
3. Instruction Graph не flatten files;
4. для Codex task в `services/payments/` строится root-to-cwd chain с nearest override;
5. для Claude task строится concat/lazy-load set по Claude semantics;
6. Dennett-native project instructions добавляются отдельным source;
7. conflict отображается с обоими origins;
8. original files не меняются.

## 25.20. Upstream commit изменил AGENTS.md

1. source adapter получает Git diff;
2. changed section reparse;
3. change event связывается с commit;
4. Effective Instruction Set cache invalidates;
5. если section human-owned, new source state принимается;
6. если section Dennett-managed и diverged от sync base, создаётся Instruction Conflict;
7. Dennett не откатывает commit и не переписывает файл;
8. пользователь/оркестратор выбирает merge, transfer ownership или import-only.

## 25.21. Dennett хочет обновить provider file

Система вывела новое project rule из repeated failures.

1. rule сначала остаётся Procedure/Instruction candidate;
2. replay подтверждает пользу;
3. native Instruction Item создаётся в project memory;
4. adapter проверяет, нужен ли export;
5. для existing AGENTS создаётся suggested patch либо managed block;
6. size/smell/conflict checks проходят до записи;
7. source ownership сохраняется;
8. если patch отклонён, native rule всё равно может использовать Dennett runtime, но не выдаётся как факт принятия командой.

3. последующие observations связываются facets/theme similarity;
4. sleep замечает recurring pattern;
5. Semantic Structure Proposal предлагает dimension `interaction_motion_tactility` только в design scope;
6. eval проверяет, помогает ли она поднимать правильные references;
7. если нет — observations остаются freeform;
8. если да — schema принимается и backfills relevant items.

## 25.23. Malicious imported memory pack

1. manifest содержит procedure, предлагающую отправлять secrets;
2. import scan маркирует source external-shared;
3. procedure quarantined;
4. instruction influence ограничено project scope;
5. no permission expansion;
6. suspicious artifact не исполняется;
7. project evidence остаётся доступным после redaction/scan;
8. attack case добавляется в security replay corpus.

## 26.6. Acceptance rules для компонентов

### Event Ledger

**Принять**, если обеспечивает durable write, replay, provenance и conflict-safe sync.  
**Отказаться/упростить**, только если другая схема обеспечивает те же инварианты с меньшей сложностью. Вероятность отказа низкая: это core.

### Raw Evidence Preservation

**Принять** для consequential/project/sensor events с policy.  
**Ограничить**, если storage/privacy cost превышает пользу; использовать retention/tiering, а не выбрасывать grounding глобально.

### Topic Dossiers

**Принять**, если улучшают navigation/context quality.  
**Перевести в on-demand view**, если maintenance cost велик или summary drift высок.

### Temporal Graph

**Принять selectively**, если multi-hop/causal/temporal tests выигрывают.  
**Не graphify всё**, если extraction cost и errors выше gains.

### Dense Retrieval

**Принять как lane**.  
**Никогда не использовать единственным**.

### Late Interaction

**Принять** для code/visual/doc domains при measurable gain.  
**Отключить** для дешёвых/simple scopes.

### Reranker

**Принять per-domain**, если улучшает end-task quality без unacceptable latency/recall loss.  
**Отключить** при candidate collapse.

### Learned Memory Manager

**Принять позднее**, если shadow/replay превосходит heuristic baseline и не нарушает invariants.  
**Не давать** authority над permissions, canonical deletion и raw evidence.

### Neural/Parametric Memory

**Принять как cache/advisory**, если reproducible gain.  
**Отклонить как canonical**, пока нет provenance, deletion и correction guarantees.

### Automatic Sleep

**Принять incremental/priority-driven**.  
**Отклонить full nightly scan** при отсутствии utility.

### Automatic Deletion

**Принять** только для explicit ephemeral policy/duplicates/corrupt telemetry.  
**Отклонить** utility-based physical deletion без user/policy mandate.

### Human-readable Notes

**Принять как UI/export/evidence**.  
**Отклонить как единственный canonical layer**.

### Selective Semantic Structuring

Принять, если сохраняется 100% labelled unseen insights на evidence/open layer, а accepted schemas статистически улучшают хотя бы одну end-task/retrieval metric без неприемлемого ontology growth.

### Preference Trajectory

Принять, если лучше flat profile на drift/cross-context cases и не ухудшает OP-Bench-like over-personalization.

### Communication Model

Принять, если растут human acceptance/style scores, но factual accuracy и disclosure safety не снижаются.

### Action Deliberation

Принять, если превосходит trigger-only/binary policy по utility с учётом false interventions, annoyance и authorization violations.

### Safe Executable Projection

Принять только для bounded DSL/materialized queries с deterministic tests. Arbitrary generated code как baseline отклонён.

### World Intelligence Profile

Принять, если снижает stale recommendations и повышает project tool-selection success относительно обычного web search + notes.

### Federated Memory Spaces

Принять, если mount/promotion не создают cross-space leakage и provenance сохраняется end-to-end.

### Portable Project Memory Pack

Принять, если clean clone восстанавливает project memory, parallel Git work не создаёт неприемлемых conflicts, а privacy audit не пропускает private fixtures.

### RO-Crate/PROV projection

Сохранять как optional generated interoperability layer. Если maintenance cost не окупается реальным exchange, native manifest остаётся единственным обязательным.

### Instruction Interoperability

Import-only обязателен. Bidirectional mode принимается per-provider только после no-overwrite/no-loop/provider-parity tests.

### Gate 2: Temporal

# 27. Переход от предыдущей модели и роль заметок, связей и тегов

## 27.1. Что именно меняется

Предыдущая модель рассматривала человекочитаемые заметки как основной формат памяти. Новая модель разделяет два вопроса:

1. **в каком виде система надёжно хранит историю и доказательства;**
2. **в каком виде человек и агент читают, редактируют и осмысляют память.**

Ответы больше не обязаны совпадать.

Каноническое основание:

- Memory Events;
- Evidence Objects;
- policy/identity/version records;
- операции correction, revision, forgetting и authorization.

Человекочитаемое основание интерфейса:

- Markdown notes;
- topic dossiers;
- project timelines;
- current-state cards;
- графические представления;
- galleries;
- history/diff views.

Это не отказ от заметок. Это отказ делать свободный Markdown ответственным одновременно за историю, temporal semantics, ACL, дедупликацию, conflicts, encrypted fields, multimodal lineage и distributed sync.

## 27.2. Заметка как evidence

Если пользователь вручную пишет заметку, она является полноценным evidence object:

- сохраняется исходный текст;
- фиксируется автор;
- фиксируется время;
- фиксируется project/scope;
- сохраняется история редакций;
- из текста могут выводиться claims, intents и relations;
- удаление или правка не стирает прежнюю версию без policy.

То есть пользовательские заметки не считаются «вторичным мусором». Они являются прямым пользовательским свидетельством, но их смысловые projections всё равно отделены от оригинала.

## 27.3. Заметка как projection

Если Dennett автоматически формирует сводку темы, она является projection:

- содержит `generated_from`;
- указывает watermark исходных events;
- имеет generator/model/prompt version;
- содержит stable handles на claims/episodes/evidence;
- может быть пересобрана;
- не повышает доверие исходным сведениям;
- при ручной правке создаёт correction event.

Projection может иметь editable и generated sections. Система не должна молча перезаписывать пользовательский текст новой генерацией.

## 27.4. Связи сохраняются, но становятся типизированными

Backlinks и свободные wiki-links остаются удобными. Но для системной логики relation должна иметь:

- source и target;
- relation type;
- valid time;
- provenance;
- confidence;
- scope;
- sensitivity;
- extraction/version;
- status: proposed, active, contested, superseded, retracted.

Свободная ссылка `[[Проект Dennett]]` может быть navigation cue. Она не обязана автоматически означать сильную semantic relation.

## 27.5. Теги сохраняются, но не заменяют policy

Теги полезны для:

- навигации;
- пользовательской организации;
- soft retrieval priors;
- UI filters;
- topic discovery;
- project workflow;
- ручной пометки важного.

Теги не должны определять в одиночку:

- ACL;
- encryption;
- principal identity;
- deletion obligations;
- current-state semantics;
- instruction authority;
- trusted/untrusted origin;
- valid time;
- canonical project ownership.

Причина проста: тег можно забыть, опечатать, удалить или получить из ошибочного LLM extraction. Security и correctness не должны зависеть от свободной строки.

## 27.6. Импорт существующей памяти

Миграция старой памяти выполняется без уничтожения исходной структуры.

### Шаг 1. Snapshot

- сделать read-only snapshot старой памяти;
- посчитать hashes;
- сохранить пути, timestamps, links, tags и attachments;
- назначить import batch ID.

### Шаг 2. Import as evidence

Каждая исходная заметка импортируется как evidence object и `legacy_note_imported` event.

### Шаг 3. Preserve semantics

- wiki-links импортируются как proposed relations;
- tags — как user annotations;
- frontmatter — как evidence metadata;
- file path — как legacy navigation path;
- attachments — как evidence objects;
- дневниковые заметки — как historical episodes/events;
- осознанные заметки — как user/agent-authored claims с исходным provenance.

### Шаг 4. Semantic extraction

Из imported evidence создаются proposals, но они не становятся автоматически подтверждённой current truth.

### Шаг 5. Conflict review

Система ищет:

- дубли;
- несовместимые актуальные утверждения;
- notes без evidence;
- сломанные links;
- secrets;
- prompts, случайно доступные обычному retrieval;
- project scope ambiguity.

### Шаг 6. Build projections

Создаются initial episodes, dossiers, current-state candidates и indexes.

### Шаг 7. Compare

Старый notes-only retrieval и новый Memory Fabric проходят одинаковый Dennett-specific benchmark. Миграция принимается только при сохранении recall и улучшении temporal/provenance/security properties.

## 27.7. Обратимый экспорт

Dennett должен уметь экспортировать память в человекочитаемый пакет:

- Markdown dossiers;
- timeline files;
- selected raw transcripts;
- media manifest;
- project index;
- source/provenance footnotes;
- relations/backlinks;
- redacted export profile;
- machine-readable manifest для повторного импорта.

Пользователь не должен оказаться заложником внутренней базы Memory Fabric.

# 28. Референсные алгоритмы бизнес-логики

Этот раздел не задаёт язык программирования или конкретную технологию. Он задаёт порядок операций и обязательные гарантии.

## 28.1. Запись события

```text
function RECORD_EVENT(request):
    authenticate actor and device
    validate scope, principal and policy hints
    derive or validate idempotency_key

    if canonical event with idempotency_key exists:
        return existing event handle

    evidence_refs = []
    for each attached payload:
        apply local privacy/redaction policy
        store immutable evidence object
        evidence_refs.append(object handle)

    event = create canonical event envelope(
        actor,
        source,
        occurred_at,
        observed_at,
        committed_at,
        scope,
        causality,
        evidence_refs,
        trust origin,
        sensitivity,
        retention,
        schema version
    )

    atomically append event to durable ledger
    expose event through recent-event overlay
    enqueue deterministic and semantic projection jobs
    return event handle and processing state
```

### Обязательные свойства

- acknowledgement не ждёт LLM extraction;
- повтор не создаёт дубль;
- evidence сохраняется до interpretation;
- scope и sensitivity существуют до индексации;
- recent read видит событие сразу;
- failed enrichment можно повторить.

## 28.2. Создание semantic proposals

```text
function DERIVE_PROPOSALS(event, projector_version):
    load event and admissible evidence renditions
    reconstruct minimal local context
    segment into observations/episode candidates

    run deterministic extractors first
    optionally run model-based extraction

    emit typed proposals:
        claim
        entity
        relation
        episode boundary
        prospective intent
        causal trace update
        procedure candidate
        dossier update suggestion

    attach provenance, confidence components, model version and scope
    do not mutate current state directly
```

## 28.3. Admission semantic proposal

```text
function ADMIT_PROPOSAL(proposal):
    verify provenance references exist
    inherit and restrict access/sensitivity
    classify origin and trust domain
    run schema/domain validation
    check duplication and identity ambiguity
    check contradiction candidates
    check whether proposal attempts to create instruction authority

    if security or provenance invalid:
        reject or quarantine
    else if ambiguity is material:
        admit as contested/hypothesis
    else:
        admit as active semantic object

    emit admission decision event
    enqueue affected state/index/dossier updates
```

## 28.4. Разрешение текущего состояния

```text
function RESOLVE_CURRENT_STATE(state_key, as_of=now):
    candidates = collect admissible claims for key and scope
    candidates = filter by transaction time <= query time
    candidates = evaluate valid-time overlap

    resolver = resolver_registry.get(predicate/domain)

    if resolver is deterministic and preconditions hold:
        result = resolver(candidates)
    else if authoritative source exists:
        result = select authoritative candidate(s)
    else if state machine contract exists:
        result = replay domain transitions
    else:
        result = preserve candidates and classify conflict

    if material ambiguity remains:
        return unresolved/conflicted state with candidates

    persist rebuildable current-state projection
    return result with provenance and freshness
```

### Важно

`latest timestamp wins` используется только тогда, когда predicate contract действительно означает «последнее легитимное обновление является текущим». Для предпочтения, цены, статуса branch и медицинского диагноза могут требоваться разные resolvers.

## 28.5. Поиск и реконструкция контекста

```text
function MEMORY_QUERY(query_request):
    authorize requested scope before search
    classify query intent and consequence level
    choose retrieval plan and latency class

    candidate_sets = parallel_execute(selected lanes):
        exact/structured
        lexical
        dense
        late interaction
        temporal/current-state
        episode hierarchy
        graph/causal
        code/repository
        multimodal
        procedural
        recent overlay
        optional direct/full-context scan

    candidates = normalize and fuse(candidate_sets)
    candidates = deduplicate without losing independent support
    candidates = apply ACL, taint, trust and influence gate

    if reranker is enabled for this domain:
        rerank while preserving protected candidates

    bundle = assemble claims + episodes + evidence handles
    assess sufficiency, conflict and freshness

    while insufficient and budget remains:
        formulate follow-up retrieval step
        open selected handles or widen scope intentionally
        update bundle

    if consequential and unsupported:
        abstain or request verification

    return indexed context bundle + explanation trace
```

## 28.6. Защита от candidate collapse

При fusion/reranking некоторые кандидаты помечаются `protected`:

- exact identifier match;
- explicit user correction;
- authoritative current-state candidate;
- directly referenced project instruction;
- evidence required to show conflict;
- high-risk negative evidence;
- newest unindexed event from overlay.

Reranker не может выбросить все protected candidates. Для каждого lane измеряется candidate survival.

## 28.7. Семантический trigger

```text
function EVALUATE_PROSPECTIVE_INTENTS(observed_event):
    deterministic_matches = trigger_index.match_structured(observed_event)
    semantic_candidates = retrieve_intents(observed_event, scoped=true)

    for intent in union(deterministic_matches, semantic_candidates):
        if expired, paused, cooled_down or unauthorized:
            continue

        match = evaluate condition using current context and evidence
        estimate false_positive and false_negative cost

        if confidence below intent-specific threshold:
            continue or request cheap additional evidence

        emit trigger_detected event

        if action is consequential:
            run separate authorization decision

        execute, ask, notify or launch agent according to policy
        record outcome and user feedback
        update cooldown/status
```

Trigger matching никогда не является разрешением на действие.

## 28.8. Consolidation

```text
function CONSOLIDATE(priority_queue, budget):
    while budget remains and queue not empty:
        target = pop highest expected-utility target
        load only affected events/projections

        detect:
            episode boundaries
            repeated claims
            contradictions
            stale current state
            dossier drift
            procedure candidates
            orphaned evidence
            index lag

        propose projection changes
        validate against source evidence
        compare with current projection

        if no new evidence and change is semantic churn:
            reject
        else:
            commit projection-version events

        record cost, utility hypothesis and next review condition
```

Full archive scan не является обычным nightly action.

## 28.9. Коррекция

```text
function CORRECT(target_handle, correction, actor):
    authorize actor
    preserve target and referenced evidence
    append correction event

    create replacement/supersession/retraction proposal
    identify affected current-state keys, dossiers, graph edges,
        procedures, triggers, indexes and answers/actions

    recompute affected projections
    invalidate caches
    flag consequential past actions for review if needed
    return correction lineage
```

Коррекция не переписывает bytes старого события, кроме специальных privacy/legal erasure operations.

## 28.10. Forgetting и удаление

```text
function FORGET(request):
    authenticate and authorize
    resolve target set using stable handles and lineage graph
    classify operation:
        attention suppression
        supersession
        archive
        access revocation
        crypto-shred
        physical erase

    create deletion plan including:
        canonical objects
        projections
        indexes
        caches
        replicas
        derived procedures

```

## 28.11. Offline sync

```text
function SYNC_DEVICE(device_log):
    authenticate device and event signatures
    deduplicate by event/idempotency/content identity
    append valid unseen events to canonical ledger

    do not resolve with blind last-write-wins
```


## 28.13. Формирование context для оркестратора

Оркестратор получает не всю память, а четыре набора:

1. обязательные system/current-focus projections;
2. recent event overlay;
3. active intents/processes/decisions;
4. query-specific reconstructed context.

Постоянный «главный чат» не должен означать постоянное накопление всей истории в prompt. Его continuity обеспечивается через Memory Fabric и indexed working context.

## 28.14. Selective Semantic Structuring

```text
function STRUCTURE_SEMANTICS(event_or_observation, intended_use=None):
    preserve original evidence
    create or update open observation / claim / topic dossier

    if intended_use is absent:
        return freeform semantic memory

    if existing operational record is required by system contract:
        validate and update that record from evidence
        return record handle

    if request can be answered by direct retrieval or on-demand extraction:
        return evidence and topic context

    if the same structured query recurs and measurements show material benefit:
        create experimental Semantic View Definition
        evaluate against freeform baseline in shadow mode

        if view improves task quality, latency or cost without false certainty:
            validate the view
        else:
            disable it

    never migrate raw evidence merely to satisfy a semantic view
    never infer missing fields as facts
    never grant new influence or permissions through structuring
```

Главное отличие от версии 1.1: новая смысловая ось не обязана становиться schema. В большинстве случаев достаточно хорошо написанного dossier, evidence handles и on-demand extraction.


## 28.15. User-Aligned Action Deliberation

```text
function DELIBERATE_ACTION(opportunity, actor, environment):
    decision_context = reconstruct:
        observed event and uncertainty
        active goals and anti-goals
        object/person/project relationship
        relevant preference trajectories
        decision precedents and counterexamples
        current authoritative state
        delegation and communication policies
        user attention/workflow state
        action permissions and external side effects

    alternatives = generate at least:
        do_nothing
        remember_only
        monitor
        defer_to_boundary_or_digest
        prepare_draft
        run_reversible_simulation
        notify
        ask
        act_within_existing_authority

    for alternative in alternatives:
        estimate separately:
            goal alignment
            expected benefit
            urgency and decay of value
            reversibility
            downside/regret if wrong
            regret of inaction
            evidence sufficiency
            precedent similarity and applicability
            interruption/annoyance cost
            privacy/disclosure cost
            authorization requirement

    run over-personalization and cross-domain influence gate
    remove alternatives that exceed authority or safety policy

    choose the least intrusive alternative that preserves
        expected value and satisfies risk constraints

    if uncertainty is decision-relevant and clarification value
       exceeds interruption cost:
        ask the smallest useful question

    emit Decision Record with considered alternatives,
        evidence handles, chosen intervention level and review condition
    execute only through normal permission system
    learn from outcome and user feedback
```

`do_nothing` является нормальным результатом. Память помогает понять волю пользователя, но не подменяет текущую авторизацию.

# 29. Поэтапное построение системы с критериями продолжения и отказа

Порядок важен. Сначала строятся свойства, потеря которых необратима; затем — качество retrieval; только потом — сложная агентность и learned optimization.

## 29.1. Phase 0 — benchmark и corpus до архитектуры

### Сделать

- Dennett-specific test corpus;
- replay harness;
- synthetic scale generator;
- adversarial memory corpus;
- evaluation of previous v0.2;
- baseline full-context/BM25/dense/hybrid;
- project, code, multimodal и prospective scenarios.

### Принять этап, если

- каждый главный requirement имеет тест;
- есть frozen validation set;
- есть end-task, security и cost metrics;
- можно воспроизвести ошибку после изменения memory policy.

### Не переходить дальше, если

- качество измеряется только subjective demo;
- test data не содержит обновлений, конфликтов, ложных совпадений и отрицательных примеров;
- retrieval score не связан с действием агента.

## 29.2. Phase 1 — Canonical Core

### Сделать

- Event Ledger;
- Evidence Objects;
- IDs, timestamps, scopes, provenance;
- idempotent write;
- recent overlay;
- exact retrieval;
- correction events;
- basic retention/deletion;
- audit and restore test.

### Принять, если

- write survives derived failures;
- replay recreates identical canonical history;
- duplicate delivery safe;
- restore from backup proven;
- access policy attached before indexing;
- correction lineage visible.

### Отменить конкретную реализацию, если

- event append зависит от LLM;

### Принять, если

- exact identifiers не теряются;
- context size bounded;
- top-k false positives не управляют consequential action;
- latency укладывается в interactive class.

### Упростить, если

- contextualization cost не окупается;

# 30. Критическая самопроверка новой системы

Новая схема существенно сильнее v0.2, но она не является магически идеальной. Ниже перечислены её основные риски и обязательные способы проверки.

## 30.1. Риск переусложнения

Event sourcing, bitemporality, graph, multiple indexes, policies и multimodal lineage создают большую систему.

### Почему сложность всё же оправдана

Она появляется не ради абстракции, а из требований:

- параллельные agents;
- offline sync;
- correction;
- exact deletion;
- multi-year history;
- current/historical distinction;
- secrets;
- rebuilding after index failure.

### Ограничитель

Не все projections внедряются сразу. Core остаётся минимальным; каждый optional lane проходит ablation.

## 30.2. Event Ledger может расти бесконечно

Append-only история не устраняет физические пределы.

### Ответ

- logical unlimitedness, а не infinite storage;
- media tiering;
- compression/dedup;
- retention policies;
- archival;
- crypto-shred/erase;
- summaries не заменяют evidence до policy decision;
- cost forecasting.

### Критерий провала

Если ordinary queries или writes начинают зависеть от размера полного ledger, partitioning/indexing спроектированы неправильно.

## 30.3. Extraction всё равно ошибается

Typed claims не делают LLM extraction истинным.

### Ответ

- proposals, а не direct truth;
- confidence components;
- evidence links;
- reversible entity resolution;
- conflict preservation;
- domain validators;
- current-state resolvers;
- human correction;
- extractor versioning and replay.

## 30.4. Causal Trace не доказывает философскую причинность

Последовательность действий может создавать ложный post-hoc вывод.

### Ответ

Causal Trace хранит operational causality и dependencies, а не заявляет научную причинность. Уроки получают статус hypothesis до повторной проверки, comparative trace или controlled replay.

## 30.5. Current-state resolver может ошибиться

Нет одного универсального правила актуальности.

### Ответ

- resolver registry per predicate/domain;
- explicit assumptions;
- unresolved state как нормальный результат;
- authoritative-source adapters;
- tests per resolver;
- user override with provenance.

## 30.6. Graph способен разрастись в шум

Автоматическое извлечение всех relations создаёт edge explosion и ложные связи.

### Ответ

- selective graph admission;
- relation utility/usage telemetry;
- typed evidence;
- weak relations не участвуют в consequential reasoning;
- periodic orphan/noise review;
- graph remains optional projection.

## 30.7. Hybrid retrieval может быть нестабильным

Разные lanes возвращают incomparable scores, reranker может потерять важный candidate.

### Ответ

- score normalization;
- rank fusion;
- protected candidates;
- per-lane recall metrics;
- query-class policies;
- direct evidence opening;
- no universal reranker;
- fallback paths.

## 30.8. Active retrieval может стать дорогим

Iterative gatherer улучшает сложные ответы, но повышает latency и model usage.

### Ответ

- latency classes;
- budgeted sufficiency loop;
- direct answer for easy query;
- cache reusable query plans/dossiers;
- use active reconstruction only for uncertainty/high value;
- show delayed/deep mode explicitly.

## 30.9. Проектные scopes могут скрыть полезную cross-project память

Жёсткая изоляция ухудшит перенос опыта; слишком слабая создаст leakage.

### Ответ

- project as strong prior, not absolute wall;
- explicit cross-project widening;
- trust/access gate;
- reusable lessons separated from project secrets;
- explain which external project memory influenced result.

## 30.10. Сенсорная память создаёт privacy risk

Даже зашифрованный архив может содержать чужие разговоры, пароли и частную жизнь.

### Ответ

- local first-stage processing;
- privacy modes;
- app/window exclusions;
- speaker/identity confidence;
- minimization by renditions;
- sensitive regions;
- access after strong confirmation;
- retention and deletion UI;
- no ordinary indexing of secrets.

Окончательное решение о допустимости 24/7 capture зависит не только от retrieval quality, но и от законодательства, consent и реального threat model; это потребует отдельного юридического исследования перед эксплуатацией.

## 30.11. Prospective memory может раздражать

Высокий recall будущих намерений создаёт false alarms.

### Ответ

- per-intent costs;
- attention budget;
- cooldown;
- user feedback;
- silent evidence accumulation before notify;
- tiered trigger confidence;
- explicit expiry.

## 30.12. Automatic consolidation может создавать «самоуверенную биографию»

Модель способна превратить несколько эпизодов в слишком общий вывод о пользователе.

### Ответ

- claims retain scope and evidence;
- no stronger epistemic status through summary;
- counterexamples retrieved;
- hypothesis status;
- confidence decomposition;
- update only from new evidence;
- user can inspect and correct.

## 30.13. Learned policies могут оптимизировать benchmark, а не пользователя

### Ответ

- shadow mode;
- user-specific outcome metrics;
- holdout tasks;
- security invariant outside policy;
- gradual canary;
- rollback;
- no canonical deletion authority.

## 30.14. Физическое удаление из backups не мгновенно

### Ответ

Deletion contract должен честно указывать:

- что исчезает немедленно;
- что становится криптографически недоступно;
- какие backups истекут позже;
- какие content-free tombstones останутся;
- как проверяется non-retrievability.

Нельзя обещать мгновенное уничтожение всех копий, если backup policy этого не обеспечивает.

## 30.15. Человекочитаемый интерфейс может отставать от канонического состояния

### Ответ

Каждая projection показывает watermark и freshness. При lag UI должен показывать recent events отдельно, а не притворяться актуальным.

## 30.16. Benchmark-и не покрывают личную жизнь полностью

Public benchmarks ограничены и могут не отражать русский язык, голос, конкретный стиль пользователя, подписочные SDK и реальные Dennett projects.

### Ответ

Public suite используется как floor, но окончательное решение принимает Dennett-specific corpus и replay реальных исправлений пользователя.

## 30.17. «Лучшая память в принципе» не может быть доказана раз и навсегда

Исследования быстро меняются, модели и инструменты обновляются, а качество зависит от workload.

Поэтому сильнейшее свойство новой схемы — не конкретный graph algorithm или embedding model, а **заменяемость производных компонентов при сохранении канонических доказательств и тестов**. Memory Fabric должна позволять улучшить retrieval через год без миграции человеческой истории в очередной закрытый формат.

## 30.18. Semantic-view explosion и taxonomy churn

Даже мягкое автоматическое формирование facets и views способно создать отдельную структуру почти под каждый эпизод.

### Ответ

- freeform Open Observation остаётся нормальным конечным состоянием, а не «необработанным долгом»;
- reusable view предлагается только после повторения или доказанной operational utility;
- minimum support/examples threshold;
- on-demand extraction и расширение существующей view проверяются раньше создания новой;
- shadow mode and ablation;
- budget числа materialized views per domain;
- merge/deprecate lifecycle;
- residual open observations допустимы бессрочно.

Не всякое знание обязано получить собственную схему, view или фиксированный набор полей.

## 30.19. Жёсткая структура может заставить модель выдумывать поля

### Ответ

- semantic freeform является default;
- partial assimilation;
- `unknown`, `not_applicable` и residual различаются;
- required fields допустимы только в operational records и минимальны;
- evidence support проверяется по каждому извлечённому полю;
- незаполненность не понижает ценность исходного наблюдения;
- view/extraction confidence не подменяет claim confidence.

## 30.20. User Decision Model может превратиться в карикатуру на пользователя

Опасность: система несколько раз увидела выбор и решила, что пользователь «всегда такой».

### Ответ

- Preference Trajectory вместо вечного fact;
- explicit scope, context and exceptions;
- counterexamples и rejected generalizations;
- local/domain-specific prior прежде global rule;
- uncertainty decomposition;
- текущий запрос пользователя имеет приоритет;
- consequential decisions открывают evidence;
- direct feedback быстро создаёт correction event.

## 30.21. Персонализация может ухудшить объективность

Память может склонить систему соглашаться с пользователем или вставлять личные сведения не к месту.

### Ответ

- objective mode отделяет factual judgment от personalized fit;
- over-personalization gate;
- user preference разрешается только в подходящем influence channel;
- факт, риск и авторизация не меняются ради стиля пользователя;
- Communication Model влияет на форму ответа, а не на истинность.

## 30.22. Proactive behavior может стать скрытым микроменеджером

### Ответ

- интервенционная лестница от `do_nothing` до `act`;
- workflow-boundary preference;
- attention budget;
- digest/defer как полноценные варианты;
- измерение acceptance, regret и annoyance;
- снижение автономности после отклонений;
- отдельные policies для разных объектов и каналов.

Система должна оптимизировать не число инициатив, а ожидаемую полезность после стоимости вмешательства.

## 30.23. Portable Pack может раздуть Git

### Ответ

- в Git не хранятся embeddings, indexes, caches и telemetry;
- large raw evidence хранится по external content-addressed references или optional artifact store;
- immutable events сегментируются и уплотняются release snapshots без удаления provenance;
- deterministic views отделены от canonical segments;
- pack size budget и export profiles;
- binary artifacts только если переносимость оправдывает размер.

## 30.24. Portable Pack может утечь вместе с репозиторием

### Ответ

- default shareable profile excludes personal/global scopes;
- privacy lint до commit/export;
- secret scanning;
- explicit ownership and disclosure class;
- encrypted/private local overlay вне tracked pack;
- round-trip тестируется на чистой установке без private dependencies;
- export audit показывает точный included set.

## 30.25. Федерация усложняет поиск и исправления

Один claim может быть promoted, forked и переинтерпретирован в нескольких spaces.

### Ответ

- stable lineage handles;
- source and target versions remain separate;
- promotion is a new derivation event;
- revocation/refresh propagation policy;
- query result labels space origin;
- cross-space dedup не стирает независимое подтверждение;
- correction impact analyzer показывает downstream copies.

## 30.26. Imported Pack может отравить глобальную память

### Ответ

- отдельный trust domain;
- read-only/fork-on-write default;
- no automatic global promotion;
- imported procedures quarantined;
- новые facets, Semantic Views и operational type changes не принимаются из package authority alone;
- external instruction files остаются instruction only в project scope;
- promotion требует evidence, policy и purpose.

## 30.27. Двусторонняя синхронизация AGENTS/CLAUDE может зациклиться

### Ответ

- operation IDs/content hashes;
- three-way merge с base snapshot;
- manager ownership;
- provider-specific adapters;
- re-import verification;
- echo suppression;
- conflict вместо overwrite;
- import-only по умолчанию для чужих файлов.

## 30.28. Инструкции разных providers семантически неэквивалентны

Codex precedence, Claude concatenation/lazy loading и другие механизмы нельзя свести к одному текстовому merge без потери смысла.

### Ответ

Instruction Graph хранит provider-native applicability, precedence и load semantics. Effective Instruction Set строится отдельно для provider, working path и task. Синхронизация переносит только явно совместимые intent blocks; остальное остаётся provider-specific.

## 30.29. Стандарты переносимости могут ограничить внутреннюю модель

### Ответ

RO-Crate/PROV-подобные представления используются на export/interchange boundary. Canonical hot path Dennett не обязан храниться в JSON-LD или внешней ontology. Adapter должен быть заменяемым и round-trip tested; если standard не выражает объект полностью, сохраняется Dennett extension namespace и original canonical payload reference.

## 30.30. Forked project memory может разойтись навсегда

### Ответ

- fork identity and common ancestor;
- explicit upstream/downstream relation;
- selective rebase/cherry-pick of memory events;
- conflict-preserving merge;
- no assumption that both forks should converge;
- procedures/world knowledge могут продвигаться отдельно от project decisions.

## 30.31. Safe Executable Projection всё равно может быть опасной

Даже restricted DSL может закрепить неверную модель пользователя.

### Ответ

- shadow execution;
- evidence links and counterexamples;
- no direct permission/safety effects;
- bounded scope;
- versioning and rollback;
- differential tests against text/claim baseline;
- automatic disable on user correction or unexplained divergence.

# 31. Решения: обязательное, условное и отклонённое

## 31.1. Обязательное ядро

Без этих компонентов система не соответствует требованиям Dennett:

- immutable/replayable Event Ledger;
- Evidence Objects;
- typed scope, provenance, trust, sensitivity и retention;
- idempotent write and read-your-writes;
- observations/episodes;
- claims separated from evidence;
- current state separated from history;
- correction/supersession/retraction;
- project-aware context;
- hybrid exact/lexical/dense baseline;
- trust/influence gate;
- bounded working context with evidence handles;
- causality/action outcome capture;
- prospective intent entity;
- security/deletion lifecycle;
- independently rebuildable indexes;
- observability and continuous eval;
- human-readable projections/export;
- typed control envelope plus open-world semantic payload;
- Open Observation и preservation of residual meaning;
- versioned Schema Registry and governed Semantic Structure Proposals;
- federated Memory Spaces with explicit mount/promotion/fork semantics;
- imported-space trust boundary;
- project-local portable memory manifest and round-trip export/import;
- User Decision Model, Communication Model and Decision Precedents for user-aligned actions;
- Action Deliberation with `do_nothing`, defer and least-intrusive alternatives;
- separation of message content, style, disclosure and delivery authorization;
- World Intelligence profile with source snapshots, volatility and live revalidation;
- Instruction Graph and Effective Instruction Set preserving provider-native semantics;
- instruction ownership, base hashes and conflict-safe synchronization;
- privacy/export audit for portable project memory.

## 31.2. Сильные, но conditional components

Добавляются после measured gain:

- temporal knowledge graph;
- topic dossiers;
- event hierarchy beyond basic episodes;
- late-interaction retrieval;
- multimodal embeddings;
- visual relation graph;
- active iterative gatherer;
- model-generated query expansion;
- procedure learning;
- learned query router;
- neural/activation memory;
- billion-scale ANN;
- complex community/global graph summaries;
- automatic schema accommodation without human review;
- Safe Executable Projection of the user model;
- learned proactive intervention policy;
- managed-block write-back into provider instruction files;
- automatic generation of root `AGENTS.md` compatibility bridge;
- RO-Crate/PROV export adapter;
- content-addressed external artifact store for large project evidence;
- automatic promotion from project/research spaces into global World Intelligence;
- full semantic instruction translation between providers.

## 31.3. Осознанно отклонённые основы

### Notes-only

Недостаточны для provenance, time, conflicts, ACL, concurrency и media lineage.

### Vector-only

Плохо решает exact identifiers, current state, causality, updates и permissions.

### Graph-only

Дорог, noisy, уязвим к extraction errors и неудобен для raw evidence.

### Summary-only

Необратимо теряет детали и делает verification невозможной.

### Full-context always

Не масштабируется по latency/cost и может ухудшать reasoning.

### Last-write-wins semantic truth

Теряет конфликт и может выбрать неверную ветку.

### Агрессивное удаление по relevance score

Relevance зависит от будущего неизвестного запроса и не является основанием для физического стирания.

### Flat text memory без origin types

Создаёт provenance-role collapse и injection risk.

### Хранение private chain-of-thought

Не нужно для полезной памяти, создаёт шум, риск и зависимость от модели. Сохраняются решения, evidence и краткое rationale.

### Полностью автономный learned memory manager с первого дня

Не обеспечивает объяснимость и безопасность без данных, benchmark и invariant enforcement.

### Fixed-schema-only memory

Гарантирует удобные запросы только для заранее угаданных понятий, заставляет искажать неожиданные сведения и требует постоянного ручного расширения ontology.

### Freeform-only memory

Не даёт надёжных current-state resolvers, aggregate queries, permissions, temporal semantics, action policies и безопасной автоматизации.

### Хранение всей проектной Memory Fabric в Git

В репозиторий не должны попадать private/global memory, secrets, runtime telemetry, embeddings, indexes, caches и весь raw sensory archive. Git содержит только переносимый shareable pack.

### `AGENTS.md` как каноническая проектная память

`AGENTS.md` остаётся provider-compatible instruction source/bridge. Он не заменяет evidence, history, research claims, causal traces и portable memory pack.

### Удаление или переписывание чужих instruction files при импорте

Existing `AGENTS.md`, `CLAUDE.md`, `MEMORY.md` и rules сохраняются. Dennett импортирует их с provider-native semantics и пишет назад только при явном ownership.

### Слепой full bidirectional sync

Без field ownership, base snapshot и conflict handling он неизбежно перезаписывает изменения пользователя или другого инструмента.

### Сведение всех provider instructions в один плоский текст

Разные системы имеют несовместимые precedence и loading semantics; плоский merge теряет область действия и создаёт конфликты.

### Произвольный исполняемый код как модель пользователя

Свободно генерируемый Python слишком силён, трудно проверяем и способен превратить ошибочную память в действие. Допустима только restricted, evidence-linked, shadow-tested materialized policy.

### Автоматическое глобальное слияние импортированной памяти

Imported pack монтируется как отдельный trust domain. Знание продвигается через явную provenance-preserving derivation.

### Память как authorization token

Ни preference, ни прошлое разрешение, ни instruction file не расширяют capabilities. Авторизация остаётся отдельной текущей системой.

# 32. Передача документа в проектирование программной архитектуры

Следующий документ должен преобразовать эту бизнес-логику в компоненты, схемы данных и протоколы. Ниже перечислено, что архитектура обязана сохранить.

## 32.1. Архитектурные границы

Минимально должны быть различимы контракты:

- canonical event service;
- evidence/object service;
- policy and identity service;
- projection pipeline;
- current-state resolver;
- retrieval planner;
- lexical/dense/graph/code/multimodal adapters;
- context composer;
- prospective trigger engine;
- consolidation scheduler;
- deletion coordinator;
- sync/replication layer;
- observability/eval system;
- human memory UI/export.

Это не означает обязательные микросервисы. Несколько контрактов могут жить в одном процессе, но не должны смешивать ответственности так, чтобы потерять rebuild, audit или access boundary.

## 32.2. Обязательные стабильные идентификаторы

Архитектура должна определить стабильные IDs минимум для:

- principal;
- device;
- project;
- task/chat/session;
- event;
- evidence object;
- observation;
- episode;
- entity;
- claim;
- relation;
- current-state key;
- causal trace;
- prospective intent;
- procedure;
- projection version;
- policy;
- deletion request;
- index build/version;
- external source object;
- correction/supersession lineage.

## 32.3. Обязательные временные поля

Нельзя оставить только один `created_at`. Нужны, где применимо:

- occurred/valid time;
- observed time;
- committed/transaction time;
- processed time;
- superseded/retracted time;
- expiry;
- last verified;
- source fetch time;
- index watermark.

## 32.4. State machines

Архитектура должна формально описать state machines для:

- event processing;
- semantic proposal;
- claim lifecycle;
- current-state conflict;
- prospective intent;
- procedure validation;
- evidence retention/deletion;
- projection rebuild;
- index migration;
- offline sync;
- security quarantine.

## 32.5. Consistency contracts

Необходимо решить:

- durability/atomicity canonical append;
- ordering and causality;
- eventual consistency projections;
- read-your-writes overlay;
- index watermarks;
- idempotent consumers;
- conflict-preserving offline merge;
- transaction boundary for event + object references;
- policy changes and retroactive enforcement.

## 32.6. Query contracts

Каждый query должен принимать или выводить:

- principal/actor;
- purpose;
- project/domain scope;
- time semantics;
- consequence level;
- latency class;
- modalities;
- evidence requirement;
- trust/access constraints;
- token/context budget;
- requested current vs historical mode;
- explanation level.

## 32.7. Index contracts

Каждый derived index обязан сообщать:

- index type/version;
- model/tokenizer/version;
- covered partitions;
- latest event watermark;
- lag;
- health;
- rebuild method;
- deletion propagation state;
- supported filters;
- score semantics;
- fallback.

## 32.8. Security contracts

Архитектура должна гарантировать:

- authorization before retrieval/prompt assembly;
- no plain secret in ordinary index;
- origin/instruction separation;
- taint propagation;
- tool/action authorization independent from memory match;
- auditable memory influence;
- signed/verified device writes where required;
- quarantine for external procedures;
- exact deletion workflow.

## 32.9. Evaluation integration

Eval не должен быть внешним скриптом, который запускается раз в год. Архитектура должна сохранять:

- query traces;
- candidate sets;
- ranking decisions;
- context manifests;
- model/index/prompt versions;
- task outcomes;
- user corrections;
- costs/latency;
- protected privacy-safe replay artifacts.

## 32.10. Решения, которые сознательно откладываются до архитектуры

Этот документ не выбирает:

- конкретную database;
- конкретный event log;
- graph store;
- vector database;
- object storage;
- embedding model;
- reranker;
- ASR/OCR/VLM;
- CRDT implementation;
- encryption/key manager;
- message queue;
- deployment topology;
- programming language;
- local vs cloud split по каждому component.

Их нужно выбирать benchmark-ом под workload Dennett, а не по популярности.

## 32.11. Memory Space и federation contracts

Архитектура должна определить:

- identity и owner каждого space;
- parent/fork/common-ancestor relations;
- read/write/mount policy;
- trust domain;
- default retrieval prior;
- promotion and refresh semantics;
- cross-space correction impact;
- local/global stable handle format;
- unmount behavior;
- deletion obligations across promoted derivatives.

Нужно формально показать, почему mount не равен merge.

## 32.12. Portable Project Memory Pack contract

Нужно зафиксировать:

- package discovery rule;
- manifest versioning;
- directory layout;
- immutable event segment format;
- object/content-addressing rules;
- human projection format;
- schema/profile declarations;
- provenance mapping;
- integrity/signature options;
- tracked vs ignored data;
- large artifact references;
- privacy/export profiles;
- import health report;
- round-trip invariants;
- Git merge and compaction strategy.

Pack должен быть полезен как обычная папка, но не обязан содержать rebuildable indexes.

## 32.13. Open semantic and view contract

Нужно определить:

- minimal typed control envelope;
- Open Observation, Claim и Topic Dossier representation;
- flexible facet namespace rules;
- query-time extraction contract;
- Semantic View Definition;
- evidence and explicit-unknown requirements;
- materialization modes;
- usage telemetry and ablation;
- deprecation/disable without evidence migration;
- отдельную границу с operational schema contracts;
- security boundary preventing any view from gaining authority.


## 32.14. User Alignment and Deliberation contracts

Архитектура должна различать:

- Preference Trajectory;
- Communication Model;
- Decision Precedent;
- User Decision Model;
- Delegation Policy;
- Opportunity Candidate;
- Action Deliberation;
- intervention decision;
- authorization decision;
- post-action feedback.

Для каждого autonomous/proactive action должны логироваться alternatives, evidence, interruption cost, reversibility, authority и outcome. Telegram pipeline должен отдельно моделировать content, style, disclosure и delivery.

## 32.15. Instruction interoperability contract

Нужно определить provider adapter interface:

- discovery;
- precedence/order;
- path applicability;
- eager/lazy loading;
- file/section provenance;
- Effective Instruction Set;
- conflict diagnostics;
- ownership and base snapshots;
- import-only/suggested-patch/managed-block/managed-file modes;
- three-way merge;
- echo suppression;

# 33. Финальная формула памяти Dennett

Память Dennett не является одной базой, одним графом, одной папкой заметок, одним user profile или одним RAG pipeline.

Она является **адаптивной федеративной системой памяти и принятия решений**, где:

1. реальность фиксируется как события и доказательства;
2. control envelope типизирован всегда, а смысл допускает открытые наблюдения;
3. formal schemas появляются постепенно через assimilation, residual preservation и governed accommodation;
4. текущее состояние вычисляется отдельно от истории;
5. предпочтения хранятся как траектории с контекстом, изменениями и контрпримерами;
6. модель общения, решения и делегирования выводится из evidence, но не становится разрешением;
7. перед инициативой система сравнивает действие, бездействие, подготовку, отсрочку, вопрос и другие уровни вмешательства;
8. опыт действий хранится причинно и проверяемо;
9. будущие намерения имеют отдельный lifecycle;
10. память мира хранит источники, утверждения, версии, volatility и требует live-revalidation;
11. личная, мировая, проектная, исследовательская и импортированная память образуют федерацию пространств, а не одну свалку;
12. знания перемещаются между spaces через provenance-preserving promotion, а не скрытый merge;
13. проект может нести переносимый Memory Pack в репозитории без утечки всей глобальной памяти;
14. `AGENTS.md`, `CLAUDE.md`, `MEMORY.md` и rules подключаются через provider-aware Instruction Graph;
15. двусторонняя синхронизация использует ownership, base snapshots, three-way merge и conflicts;
16. поиск планируется под вопрос, проект, space, время, риск, цель использования и модальность;
17. найденное проходит access/trust/influence/over-personalization gate;
18. рабочий контекст остаётся компактным и содержит handles к полным evidence;
19. заметки, связи и теги остаются удобным человеческим слоем, но не ограничивают внутреннюю систему;
20. сенсорные данные объединяются общей временной шкалой;
21. сон обслуживает только полезные и затронутые области;
22. все derived structures, schemas, indexes и materialized policies заменяемы и откатываемы;
23. безопасность, forgetting, provenance и export privacy встроены в lifecycle;
24. сложность каждого механизма оправдывается benchmark-ом, replay и реальными пользовательскими outcomes;
25. система может становиться лучше, не переписывая собственную историю и не замыкая пользователя в одном формате.

Короткое имя этой модели:

> **Dennett Adaptive Federated Memory Fabric: evidence-grounded, event-sourced, selectively structured, user-aligned, project-portable, temporally correct, causally useful, prospectively active and continuously evaluated memory.**

# 35. Краткий чек-лист итогового решения

Перед началом архитектуры следует принять следующие положения как текущий design baseline:

- фундамент — evidence + event ledger, не notes/vector/graph;
- notes, links и tags сохраняются как human layer/projections;
- raw evidence, retrieval cue, claim и instruction разделены;
- current state не равен history;
- valid time не равен transaction time;
- conflicts сохраняются;
- project state и causal traces first-class;
- future intents first-class;
- retrieval query-planned и multi-lane;
- similarity проходит access/trust/influence gate;
- context compact, evidence открывается по handles;
- graph selective и rebuildable;
- multimodal memory синхронизирована по событиям;
- consolidation incremental и utility-driven;
- deletion — управляемый lifecycle;
- learned policy только после deterministic baseline;
- каждый сложный компонент обязан выиграть ablation;
- архитектура должна позволять заменить любой derived component без потери истории;
- control envelope типизирован, semantic payload допускает open-world observations;
- смысл не насильно втискивается в schema: Open Observation и Topic Dossier являются полноценным конечным состоянием;
- semantic structuring выбирается по операционной потребности, а не считается обязательной стадией созревания знания;
- flexible Semantic Views создаются on demand и могут быть удалены без миграции evidence;
- strict schema evolution применяется только к настоящим operational contracts;
- предпочтения являются траекториями с scope и counterexamples, а не вечными фактами;
- User Decision Model не является permission system;
- proactive deliberation всегда рассматривает `do_nothing`, defer, draft, ask и act;
- персонализация проходит relevance/over-personalization gate;
- для сообщений отдельно решаются content, style, disclosure и delivery;
- World Intelligence хранит snapshots/claims/freshness и перепроверяет volatile сведения;
- personal, world, project, research, task и imported memory существуют как federated spaces;
- mount не равен merge, promotion создаёт provenance-preserving event;
- проект имеет portable Memory Pack, но private/global data не обязаны входить в Git;
- hidden versioned directory в обычном project tree является default transport; Git Notes, отдельные refs и external stores остаются optional adapters;
- portable pack может иметь content-addressed/Merkle integrity root, selective disclosure и injection-resistant rehydration;
- portable pack не хранит rebuildable embeddings/indexes как canonical data;
- imported pack по умолчанию имеет отдельный trust domain и fork-on-write/read-only mode;
- `AGENTS.md` не является канонической памятью Dennett, но поддерживается как provider-native instruction source;
- `CLAUDE.md`, `AGENTS.md`, `MEMORY.md` и rules сохраняют собственные precedence/loading semantics;
- existing instruction files не удаляются и не переписываются без ownership;
- двусторонняя синхронизация использует base snapshot, manager ownership, three-way merge и conflict;
- root `AGENTS.md` compatibility bridge является optional, а не обязательным фундаментом;
- export/import проходит privacy audit, integrity validation и clean round-trip;
- все новые механизмы имеют Dennett-specific acceptance tests и могут быть отключены без потери evidence.
