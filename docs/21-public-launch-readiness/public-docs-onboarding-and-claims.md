## Stage 14 Decision

Stage 14 makes the public entrypoints truthful and usable for local checkout users. It does not expand the release target. The supported first-user path remains a local source checkout built with pnpm, plus controlled local tarball proof for distribution mechanics.

Public docs must prefer bounded, evidence-backed language over marketing language. If a capability needs runtime auth, model access, provider setup, a generated artifact, a local state database, or a specific proof command, the public doc must say so near the command that depends on it.

## Clean Checkout Onboarding

From a clean local checkout:

```powershell
pnpm install --frozen-lockfile
pnpm build
node .\dist\src\interfaces\cli.js --help
node .\dist\src\interfaces\cli.js support-bundle
```

Expected boundary:

- `dist` is generated by `pnpm build` and is not tracked source.
- `support-bundle` emits local-only redacted JSON to stdout and does not upload data.
- Stateful commands default to `.dennett/local-state.sqlite` under the checkout and usually accept `--state-db <path>`.
- Codex live runtime commands require local Codex/App Server authentication and account/model availability.

## Local Tarball Proof Onboarding

Use this command only for controlled local package proof:

```powershell
pnpm package:local-install:proof
```

This proves a local `.tgz` can be built, installed into a temporary npm consumer project, smoked through the installed `dennett-agent-orchestrator` bin, uninstalled, and verified as removed from that temporary consumer project.

It does not prove public npm publication, public registry install, package signing, provenance, retained SBOMs, installers, containers, hosted deployment, or general availability.

## Examples Validation Status

The current public examples are documented in [examples/agents/README.md](../../examples/agents/README.md).

Current validated paths:

- `examples/agents/valid/*.json` are loadable as Agent JSON and must declare top-level `params.*` references they use.
- `examples/agents/builder-drafts/valid-*.json` validate against `contracts/json-schema/builder-output.schema.json`.
- `examples/agents/builder-drafts/invalid-*.json` are negative examples and must document the expected schema violation.
- `phase5-codex-minimal.json` has an offline mocked CLI smoke test that resolves `--param topic`.

Validation command:

```powershell
pnpm test -- tests/unit/public-examples.test.ts
```

Live examples are not implied by the offline example test. Live Codex runs depend on authentication and availability of `gpt-5.3-codex` for the local account. The Mem0 runtime-memory example also requires a local memory-provider registration with `codex_ref` set to `primary_memory`.

## Public Claim Review Checklist

Before adding or changing public-facing docs, examples, issue templates, release notes, README text, package descriptions, or screenshots, verify:

1. The doc names whether the path is local checkout, local tarball proof, live runtime, or provider-backed.
2. Every command that requires `pnpm build`, generated `dist`, a local state DB, Codex auth, model access, or provider registration says so.
3. Example docs separate schema/offline validation from live runtime execution.
4. Public docs link to this document or another Stage 21 owner before making launch, package, runtime, memory, support, or production claims.
5. No claim relies on a task summary alone; it points to durable docs, tests, or proof commands.
6. Sensitive-data guidance is visible for issue reporting, support bundles, runtime diagnostics, logs, and examples.
7. Broken localized text or mojibake is removed from public entrypoints until reliable encoding and review exist.

## Allowed Public Claims

These claims are allowed when the referenced commands or docs remain accurate:

- Dennett can be built from a local source checkout with `pnpm install --frozen-lockfile` and `pnpm build`.
- The built local CLI can print help with `node .\dist\src\interfaces\cli.js --help`.
- `support-bundle` emits local-only redacted diagnostics for user-reviewed sharing.
- Controlled local tarball proof exists through `pnpm package:local-install:proof`.
- Public examples include schema-validated Agent JSON and builder wrapper examples.
- The Phase 5 example has an offline mocked CLI smoke test that validates parameter resolution without requiring live Codex access.
- Live Codex and Mem0 example runs are possible only when the local environment satisfies the documented auth, model, and provider requirements.

## Forbidden Public Claims

Do not claim:

- public npm publication, public registry install, public package namespace ownership, installer distribution, container distribution, signed artifacts, provenance, retained SBOMs, or public package rollback;
- hosted service operation, managed deployment, SaaS readiness, uptime, SLA, status-page monitoring, automatic telemetry, hosted audit readiness, managed incident response, cloud deployment, or production-load readiness;
- full Codex App Server certification, all model/options support, account/rate-limit guarantees, or broad runtime-provider reliability;
- `gpt-5.3-codex` availability for every user or account;
- native App Server memory, broad memory-provider support, provider reliability, provider-wide cleanup, true restore, graph-store cleanup, or durable external provider cleanup beyond the documented scoped Mem0 behavior;
- full user interaction readiness, complete managed-subagent orchestration, durable background subagent execution, live subagent cancellation delivery, complete public Builder 2.0 readiness, stable compatibility for experimental CLI commands, or any stable JS/TS API;
- support-bundle or redaction output makes it safe to share private prompts, memory records, credentials, tokens, cookies, unredacted account data, full transcripts, provider configs, or proprietary outputs.

## Completion Criteria For Future Public Docs Expansion

Future public-doc expansion requires:

- a named owner document for the new public claim;
- exact user-visible behavior and non-goals;
- validation commands or live/artifact evidence where the claim depends on a real runtime, provider, package, or hosted surface;
- explicit unsupported cases;
- rollback, cleanup, or recovery evidence where operational responsibility is implied;
- updated examples and issue templates when the user-facing path changes.
