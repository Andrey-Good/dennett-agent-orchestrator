# Denet Owner Playbook

**Для кого:** для владельца Denet, который определяет продукт, но не обязан быть профессиональным программистом.  
**Цель:** объяснить, что именно нужно делать человеку, что можно безопасно отдать агентам и как принимать работу без чтения каждой строки кода.

---

# 0. Ваша роль

Вы не обязаны быть главным разработчиком Denet. Ваша основная роль:

> **Product Owner + Acceptance Authority + Keeper of Intent.**

Вы отвечаете за:

- что должен уметь продукт;
- каким он должен ощущаться;
- какие компромиссы приемлемы;
- где проходит граница приватности и автономности;
- какие внешние расходы допустимы;
- когда milestone действительно полезен;
- какие ошибки или неудобства нельзя считать «нормальными для первой версии».

Вы не обязаны самостоятельно решать:

- какой private Rust type использовать;
- как назвать helper;
- какой connection pool выбрать внутри уже утверждённого adapter;
- как организовать локальный тест;
- как исправить lint;
- как разбить функцию на файлы, если module boundary уже определена.

---

# 1. Три уровня решений

## Green — агент решает сам

Примеры:

- private implementation;
- локальный refactor;
- naming внутри модуля;
- выбор test fixture;
- error type в рамках существующей convention;
- оптимизация без изменения behavior;
- документация уже утверждённого contract.

Вы не должны получать такие вопросы.

## Yellow — агент предлагает решение

Примеры:

- новая dependency;
- изменение public trait без breaking semantics;
- новый feature flag;
- заметный UX-компромисс;
- provider choice;
- performance vs storage trade-off;
- изменение default, которое обратимо.

Вы получаете короткую карточку:

- проблема;
- варианты;
- рекомендация;
- последствия;
- стоимость/риск;
- что произойдёт при бездействии.

## Red — решаете вы

- изменение бизнес-логики;
- privacy/ambient defaults;
- новая модель автономности;
- необратимое удаление/миграция;
- major architecture change;
- новый платный hosted dependency;
- изменение recoverability;
- новый внешний канал отправки;
- security weakening;
- release stable;
- существенный визуальный/UX выбор.

# 2. Что делать в начале milestone

Вам нужно подтвердить только пять вещей.

## 2.1. Пользовательский результат

Ответьте:

> Что я смогу сделать после milestone, чего не могу сделать сейчас?

Плохой ответ:

> Будет реализован Event Dispatcher.

Хороший ответ:

> Я смогу запустить фоновую задачу, закрыть приложение, открыть его снова и увидеть, что задача продолжилась или понятным образом восстановилась.

## 2.2. Не-цели

Согласуйте, что milestone **не** пытается сделать.

Это защищает от бесконечного расширения.

## 2.3. Demo path

Попросите один конкретный сценарий демонстрации.

## 2.4. Решения, требующие вас

Пусть агент заранее перечислит Yellow/Red gates.

## 2.5. Exit criteria

Вы должны видеть:

- required tests;
- demo;
- known limitations;
- что будет следующим.

# 3. Что делать во время разработки

## 3.1. Не следить за каждым commit

В нормальном режиме вы получаете сообщения только когда:

- Work Package завершён;
- нужен выбор;
- обнаружен blocker;
- CI красный и требует product decision;
- milestone готов к demo;
- найден риск безопасности/данных.

## 3.2. Отвечать на Decision Requests

Хороший ответ часто занимает одну строку:

> Выбираю вариант B. Цена допустима, но сохрани локальный fallback.

Не нужно объяснять техническую реализацию.

## 3.3. Проверять живой продукт

Наиболее ценная ваша работа:

- открыть сборку;
- пройти demo path;
- сказать, что неудобно;
- проверить, соответствует ли результат изначальной идее;
- заметить, что система задаёт слишком много вопросов;
- проверить скорость и ощущение контроля;
- определить, можно ли реально пользоваться этим ежедневно.

## 3.4. Не принимать зелёный CI за идеальный продукт

CI доказывает контрактную корректность. Вы доказываете пригодность.

# 4. Как принять Work Package без знания программирования

Попросите Completion Packet и ответьте на восемь вопросов.

1. Какое видимое или контрактное поведение изменилось?
2. Какие тесты это доказывают?
3. Что происходит при ошибке?
4. Можно ли остановить или откатить?
5. Изменились ли данные, права или внешние эффекты?
6. Есть ли известное ограничение?
7. Не появилось ли новое платное/внешнее требование?
8. Можно ли продолжить следующий пакет без скрытой ручной работы?

Для UI дополнительно:

- понятно ли, куда нажать без инструкции;
- всегда ли виден Stop/Back/Cancel;
- не путается ли proposal с уже выполненным действием;
- честно ли показан offline/stale state;
- не мешает ли interface обычной работе.

# 5. Можно ли просто сказать агенту «иди по плану»

## 5.1. Короткий ответ

**Да, но не на весь проект одним поручением.**

Безопасная команда выглядит так:

> Выполни Autonomous Batch AB-001 по готовым Work Packages и остановись на первом decision gate.

## 5.2. Когда можно оставить агента на часы

Можно, если:

- Work Packages READY;
- code roots определены;
- CI работает;
- worktree изолированы;
- нет реальных внешних эффектов;
- нет R3/R4 решений;
- stop conditions заданы;
- агент не может auto-merge критический код;
- есть checkpoints;
- тесты выполняются автоматически.

## 5.3. Когда можно оставить на ночь или день

После появления:

- стабильного fast CI;
- merge queue;
- test catalogue;
- automatic repository checks;
- batch runner;
- decision queue;
- detached review;
- reliable backups;
- ограниченного sandbox.

Даже тогда batch должен быть конечным: например, 3–8 подготовленных Work Packages, а не «делай что считаешь полезным».

## 5.4. Когда нельзя уходить надолго

- первые архитектурные vertical slices;
- UX foundation;
- privacy/ambient behavior;
- identity/recovery;
- migrations;
- permissions;
- external communication;
- data deletion;
- release process;
- выбор дорогой зависимости.

Здесь ваше решение имеет высокую стоимость изменения.

## 5.5. Почему агент не должен работать двое суток без ворот

Даже сильная модель может последовательно построить систему на одной неверной предпосылке. Через два дня получится большой логичный diff, который решает не ту задачу.

Gate нужен не потому, что агент слабый, а потому, что обратная связь должна приходить раньше, чем ошибка становится фундаментом.

# 6. Рекомендуемая частота вашего участия

## Самое начало проекта

- 20–40 минут на kickoff каждого vertical slice;
- короткий review после каждого R2+ package;
- demo 1–2 раза в неделю активной разработки;
- решения обычно пакетами, а не по одному сообщению.

## После появления рабочего skeleton

- kickoff milestone;
- ответы на Yellow/Red decision cards;
- demo в конце milestone;
- weekly product review;
- monthly architecture/usage review.

## После beta

- release acceptance;
- review incidents;
- product priorities;
- privacy/cost decisions;
- реальное использование продукта и feedback.

Практическая нагрузка может быть небольшой: несколько содержательных решений в неделю важнее ежедневного чтения кода.

---

# 7. Ваши обязательные артефакты

## 7.1. Product intent

Вы поддерживаете актуальными:

- Functional Concept;
- приоритеты roadmap;
- открытые product questions;
- rejected ideas, которые не нужно возвращать;
- personal preferences, важные для UX.

## 7.2. Decision log

Для Red decisions достаточно короткой записи:

```text
Decision: ambient microphone default remains opt-in during alpha.
Reason: need battery/privacy field data.
Revisit: after R-02 spike and 30-day pilot.
```

Агент оформит ADR или spec delta.

## 7.3. Acceptance notes

После demo:

- accepted;
- accepted with follow-up;
- rejected;
- UX correction;
- priority change.

---

# 8. Что вы не должны делать

- вручную придумывать архитектуру каждого модуля;
- соглашаться с непонятным решением только потому, что агент уверен;
- разрешать giant PR «потому что всё связано»;
- принимать «тесты проходят» без demo ключевого пути;
- разрешать обход тестов ради скорости;
- принимать external dependency без объяснения exit plan;
- разрешать агенту менять бизнес-логику вслед за удобством реализации;
- пытаться читать весь код, если есть хорошие tests/review/evidence;
- держать важные решения только в чате.

# 9. Готовые шаблоны ваших команд

## 9.1. Начать milestone

```text
Подготовь kickoff для Milestone <ID> по implementation strategy.
Покажи пользовательский результат, demo path, non-goals, Work Packages,
риски, мои Yellow/Red решения и exit criteria. Код пока не меняй.
```

## 9.2. Запустить пакет

```text
Выполни Work Package <ID> по Agent Execution Protocol.
Работай автономно внутри указанного scope.
Остановись на decision gate. Верни Completion Packet и demo instructions.
```

## 9.3. Запустить batch

```text
Запусти Autonomous Batch <ID>.
Продолжай только после green gates. Не принимай R3/R4 решения,
не добавляй production dependencies и не меняй public contracts без остановки.
```

## 9.4. Принять milestone

```text
Подготовь Milestone Acceptance Pack:
- что пользователь теперь может делать;
- demo;
- test/review evidence;
- known limitations;
- unresolved risks;
- rollback;
- следующий milestone.
```

## 9.5. Попросить объяснение без жаргона

```text
Объясни решение как владельцу продукта:
что изменится для пользователя, какой риск, почему выбран этот вариант,
что будет, если отказаться, и насколько дорого изменить решение позднее.
```

# 10. Минимальная owner checklist перед merge R3/R4

- [ ] Я понимаю пользовательское изменение.
- [ ] Я понимаю, какие данные/права затронуты.
- [ ] Есть rollback/recovery.
- [ ] Есть независимый review.
- [ ] Есть failure test.
- [ ] Нет скрытого нового внешнего сервиса или расхода.
- [ ] Spec/ADR обновлены.
- [ ] Demo соответствует исходной цели.
- [ ] Известные ограничения приемлемы.
- [ ] Следующий шаг не требует переделывать это решение немедленно.

# 11. Реалистичный ответ на главный вопрос

Вы можете организовать работу так, что говорите:

> «Иди по готовому batch-плану до ближайшего моего решения»

и агент работает автономно много часов, а со временем — ночь или день.

Но вы не должны говорить:

> «Реализуй весь Denet, вернусь через несколько дней»

пока не существуют:

- полный Work Package backlog;
- CI и test catalogue;
- границы автономности;
- decision gates;
- merge/review automation;
- recovery.

Ваша работа не в том, чтобы постоянно управлять кодом. Ваша работа — **не позволить технически красивой системе незаметно уйти от вашего продукта**.

---

# Финальная формула

> **Владелец Denet не является ручным диспетчером каждого агента. Он задаёт смысл, принимает дорогие решения и проверяет реальную полезность. Всё, что можно доказать компиляцией, тестами, контрактами, sandbox и review, должно решаться без его участия. Всё, что определяет волю пользователя, приватность, необратимость и ощущение продукта, остаётся под его контролем.**
