# Agent Orchestrator
## Финальное высокоуровневое описание проекта и канонический контракт agent JSON

Статус документа: базовый канонический документ для дальнейшего разбиения на отдельные спецификации.

Цель документа: зафиксировать общую модель системы так, чтобы дальнейшая детализация по отдельным файлам не меняла смысл проекта, не плодила двусмысленности и не размывала границу ответственности оркестратора.

---

## 1. Короткая формула проекта

Этот проект — это **Codex-first оркестратор агентных графов**.

Он позволяет:

- хранить агентов как переносимые JSON-файлы;
- собирать графы того, как запускаются агенты;
- задавать, что получает агент на вход;
- задавать, какие права, skills, MCP и plugins доступны агенту;
- запускать агентов через Codex и аналогичные runtime-системы;
- сохранять чаты только настолько подробно, насколько это нужно для resume;
- отделять сам агент от чатов, событий и интерфейсов.

Проект не является новой агентной системой поверх Codex. Он является **системой оркестрации запусков агентов**.

---

## 2. Главный принцип границы ответственности

Оркестратор отвечает только за:

- порядок запуска агентов;
- граф исполнения;
- выбор запускаемого агента или runtime-конфигурации;
- входы нод;
- доступные skills, MCP, plugins и права;
- обработку финального выхода ноды;
- сохранение чатов и данных для resume;
- отдельное управление внешними событиями и триггерами.

Оркестратор не отвечает за внутреннюю работу агента внутри runtime.

Оркестратор не должен:

- знать внутренний reasoning агента;
- моделировать его tool calls;
- анализировать его внутренние side effects;
- считать своим делом любые изменения файлов, которые агент сделал внутри runtime;
- вмешиваться в поведение агента сверх того, что сам передал на вход и задал в настройках запуска.

Для оркестратора агент внутри ноды — это черный ящик с четкой границей:

- на вход подается сообщение и конфигурация запуска;
- на выходе возвращается финальный результат;
- все внутреннее остается делом runtime.

---

## 3. Архитектурная форма системы

Система состоит из двух уровней:

### 3.1. Core

Core — это ядро, в котором живут:

- оркестрация графов;
- выполнение логики агентов;
- обработка событий;
- хранение чатов и состояния resume;
- работа с agent files;
- работа с runtime adapters.

### 3.2. Interfaces

Interfaces — это пользовательские интерфейсы поверх Core:

- CLI;
- будущий UI;
- возможные другие интерфейсы позже.

Интерфейс не является местом, где живет логика проекта. Он только подключается к Core.

---

## 4. Политика закрытия интерфейса

Пользователь должен иметь явную глобальную настройку поведения при закрытии интерфейса.

Поддерживаются два режима:

- `keep_core_running` — при закрытии CLI или UI Core продолжает работать, а уже запущенные в нем агенты продолжают выполнение;
- `stop_core` — при закрытии интерфейса Core тоже завершает работу.

Если выбран режим `stop_core`, то:

- все активные запуски прерываются штатным завершением Core;
- при следующем запуске системы никакого автоматического resume не происходит;
- пользователь может вручную открыть чат и явно продолжить работу, если это возможно.

Режим по умолчанию: `keep_core_running`.

---

## 5. Codex-first и работа через внешние runtime

Проект строится по принципу `codex-first`, но не `codex-only`.

Это означает следующее:

1. На первом этапе основной опорный runtime — Codex.
2. При реализации оркестратора нужно по возможности использовать уже готовые функции Codex App Server и аналогов:
   - resume;
   - live-комментарии во время работы;
   - подключение MCP;
   - подключение skills;
   - подключение plugins;
   - иные доступные нативные механизмы runtime.
3. Проект не должен заново изобретать то, что уже корректно реализовано в Codex App Server или аналогичном runtime.
4. При этом внутренняя модель оркестратора не должна быть жестко привязана только к OpenAI.

---

## 6. Поправка терминов: не SDK, а MCP

В этой системе используются следующие термины:

- **Skill** — текстовая инструкция или набор инструкций для агента.
- **MCP** — интерфейс взаимодействия агента с внешней системой, сервисом или возможностью.
- **Plugin** — готовый набор skills и MCP.
- **Runtime adapter** — слой интеграции с конкретным внешним runtime, например Codex.

Важно:

- у этой утилиты **нет собственной сущности SDK** как части модели агента;
- там, где раньше использовалось слово `SDK`, в действительности имеется в виду `MCP`;
- работа с MCP выполняется через нативные механизмы Codex App Server или его аналогов.

---

## 7. Кто определяет контракт skills, MCP и plugins

Контракт, внутренняя форма и способ хранения skills, MCP и plugins определяются **не этой утилитой**, а тем, как они определены в Codex App Server и совместимых runtime-средах.

Эта утилита:

- не придумывает новый внутренний формат skills;
- не придумывает новый внутренний формат MCP;
- не придумывает новый внутренний формат plugins;
- не интерпретирует их содержимое глубже необходимого минимума.

Эта утилита только:

- ссылается на уже существующие Codex-совместимые сущности;
- хранит минимальные ссылки и pass-through-конфигурацию там, где это нужно;
- определяет, какие из них доступны конкретной ноде;
- передает их runtime adapter-у.

Единственное исключение — встроенный MCP-сервер самой утилиты для общения агента с пользователем во время работы. Его контракт определяется этой утилитой, потому что это собственный системный MCP этой утилиты.

## 8. Словарь сущностей

### 8.1. Agent file

Переносимый JSON-файл, описывающий граф агента.

### 8.2. Node

Единица исполнения внутри графа. Нода запускает либо runtime-агента, либо другой orchestrator-agent.

### 8.3. Edge

Связь между двумя нодами, задающая порядок и условие перехода.

### 8.4. Chat

Локально сохраненное состояние общения с агентом, предназначенное прежде всего для resume.

### 8.5. Resume state

Минимальный набор данных, достаточный для продолжения чата или запуска.

### 8.6. Runtime adapter

Адаптер к конкретному runtime, например Codex.

### 8.7. Skill

Текстовая инструкция.

### 8.8. MCP

Интерфейс взаимодействия агента с внешней возможностью.

### 8.9. Plugin

Готовый набор skills и MCP.

### 8.10. Params

Параметры агента, настраиваемые пользователем без клонирования agent file.

### 8.11. Vars

Глобальные runtime-переменные графа.

### 8.12. Node output

Финальный результат конкретной ноды.

### 8.13. Event

Внешняя сущность, которая запускает агента по триггеру.

### 8.14. Trigger

Источник события.

### 8.15. Permission set

Набор прав, с которыми runtime запускает агента в ноде.

---

## 9. Что является источником истины

### 9.1. Агент

Источник истины для агента — **agent JSON file**.

Агент не должен считаться записью БД.

### 9.2. Чаты и resume

Источник истины для chat/resume state — локальное хранилище Core.

Базовая форма хранения:

- локальная БД;
- текст и JSON, достаточные для resume;
- без попытки хранить внутреннюю полную историю runtime.

### 9.3. События

События и триггеры не живут внутри agent file. Они хранятся отдельно.

---

## 10. Что именно хранится в чатах

Чаты нужны не для форензики и не для полного технического архива.

Чат хранит только то, что нужно для resume и нормального пользовательского опыта:

- сообщения пользователя;
- видимые ответы агента;
- текущее эффективное контекстное окно чата;
- сведения о выбранном runtime;
- параметры запуска, если они влияют на resume;
- native session handle, если runtime его поддерживает;
- сведения, нужные для локального fallback-resume.

Чат не хранит по умолчанию:

- chain-of-thought;
- внутренние tool traces;
- историю каждого внутреннего действия агента;
- историю изменений файлов;
- скрытые промежуточные рассуждения.

---

## 11. Resume

Resume всегда выполняется только по явному действию пользователя.

Никакого auto-resume после перезапуска Core или интерфейса быть не должно.

Поддерживаются два режима resume:

### 11.1. Native resume

Если runtime умеет продолжать собственную сессию, Core должен предпочитать именно этот путь.

### 11.2. Local resume

Если runtime не умеет native resume, Core использует локально сохраненный chat/resume state.

---

## 12. Параметры, переменные и outputs

Внутри выполнения есть три разных слоя данных:

### 12.1. Params

Пользовательские параметры агента.

### 12.2. Vars

Глобальные runtime-переменные графа.

### 12.3. Node outputs

Журнал финальных outputs конкретных нод.

Эти сущности не должны сливаться в один словарь.

---

## 13. Граф исполнения

### 13.1. Базовый режим графа

В рамках одного run граф исполняется **последовательно**.

Это означает:

- в один момент времени активна только одна нода;
- переход идет от текущей ноды к следующей;
- параллельное исполнение нескольких нод в базовой модели не поддерживается.

### 13.2. Старт графа

Run всегда начинается с `entry_node_id`.

### 13.3. Завершение графа

Граф завершается, если:

- у текущей ноды нет подходящего следующего edge;
- текущая нода завершилась ошибкой без продолжения;
- run был отменен;
- run был прерван внешне.

---

## 14. Закон разделения control flow и data flow

Связи (`edges`) отвечают только за:

- порядок исполнения;
- выбор следующей ноды;
- условия перехода.

Связи не отвечают за передачу данных.

Передача данных выполняется только через:

- `params`;
- `vars`;
- `node outputs`;
- `event payload`, если запуск пришел от события.

---

## 15. Как строится сообщение на вход ноды

Сообщение, которое подается ноде на вход, определяется **не строкой**, а JSON-структурой.

Эта структура состоит из частей.

Каждая часть может быть:

- обычным текстом;
- ссылкой на доступную переменную.

Это означает, что вход ноды строится как последовательность блоков, где можно смешивать текст и переменные.

Оркестратор не должен подставлять скрытые неописанные данные на вход ноды.

---

## 16. Разрешенные ссылки во входе ноды

Во входе ноды разрешены следующие ссылки:

- `params.<name>`
- `vars.<name>`
- `node.<node_id>.text`
- `node.<node_id>.json.<path>`
- `event.<path>`

Если ссылка неразрешима, это является ошибкой конфигурации или исполнения.

Скрытых fallback-поисков по другим пространствам имен нет.

---

## 17. Нода как граница вызова

Нода описывает только границу вызова агента.

Нода должна явно описывать:

- что именно запускается;
- через какой runtime adapter это запускается;
- какой prompt используется;
- какое входное сообщение строится;
- какие skills, MCP и plugins доступны;
- какие права действуют;
- как интерпретируется финальный output.

---

## 18. Два вида нод

Поддерживаются два вида нод.

### 18.1. `runtime_agent`

Нода напрямую запускает агент через конкретный runtime adapter.

Такая нода сама содержит:

- `runtime_adapter`;
- `prompt`;
- ссылки на `skills`, `mcps`, `plugins`;
- `permissions`;
- `runtime_options`.

### 18.2. `orchestrator_agent`

Нода запускает другой agent file, управляемый этим же оркестратором.

Такая нода содержит:

- `agent_ref`;
- входное сообщение;
- ожидаемый режим output.

Для `orchestrator_agent` не задаются собственные `skills`, `mcps`, `plugins`, `prompt` и `permissions`, если иное не будет добавлено отдельной будущей спецификацией.

---

## 19. Output ноды

Нода всегда возвращает только **финальный output агента**, а не внутреннюю историю его действий.

Поддерживаются два режима output:

- `text`
- `json`

### 19.1. Режим `text`

Output трактуется как текст.

Он сохраняется как:

- текстовый output ноды;
- возможный пользовательский финальный ответ;
- возможный input следующих нод.

### 19.2. Режим `json`

Output трактуется как JSON-словарь.

Правила:

- ожидается именно JSON-словарь;
- массив, строка, число или иной JSON-корень считаются некорректным output для этого режима;
- если output корректен, его поля переносятся в `vars`.

Оркестратор не добавляет скрытых инструкций в prompt, чтобы заставить агента вернуть JSON.

Если автор ноды хочет JSON-выход, он сам должен обеспечить это prompt-ом, skill-ами или иными явными средствами.

---

## 20. Политика перезаписи переменных

Если разные ноды возвращают JSON с одинаковыми именами переменных, это разрешено.

Базовое правило:

- **последняя запись побеждает**.

Это сделано специально, чтобы разные ноды могли актуализировать состояние одной и той же переменной по мере продвижения графа.

---

## 21. Финальный ответ агента

По умолчанию финальным ответом всего агента считается output **последней успешно отработавшей ноды**, после которой граф завершился.

Это регулируется полем `final_output.mode`.

Поддерживаются режимы:

- `last_node_output` — финальный ответ агента равен output последней завершившейся ноды;
- `none` — у агента нет автоматического финального пользовательского ответа.

Если установлен режим `none`, это не означает, что run не имеет результата. Это означает только, что Core не создает автоматический финальный ответ в чате.

---

## 22. Live-общение во время работы агента

Во время работы агента поддерживаются два разных механизма взаимодействия с пользователем:

- комментарии пользователя в активный run;
- встроенный MCP-канал от агента к пользователю.

Они независимы и могут быть включены вместе.

---

## 23. Комментарии пользователя во время работы

Комментарий — это пользовательское сообщение, которое отправляется агенту во время уже идущего run, но не является ответом на MCP-вопрос.

### 23.1. Настройка

В agent file можно указать:

- что комментарии во время работы запрещены;
- что комментарии во время работы разрешены.

Если комментарии разрешены, обязательно должен быть указан список `target_node_ids`.

### 23.2. Доставка комментария

Так как run последовательный и в один момент времени активна только одна нода, комментарий может быть доставлен только если текущая активная нода входит в список `target_node_ids`.

Если текущая активная нода не входит в этот список, комментарий отклоняется.

### 23.3. Способ доставки

Core не придумывает собственный нестандартный механизм внедрения комментариев в runtime, если runtime уже умеет live-комментарии нативно.

Runtime adapter должен использовать нативный механизм runtime, если он существует.

Если активный runtime не поддерживает live-комментарии, то для такой ноды комментарии считаются недоступными.

---

## 24. Встроенный MCP-сервер для общения агента с пользователем

Для гибкости система вводит собственный встроенный MCP-сервер, который позволяет агенту обращаться к пользователю во время работы.

Этот MCP-сервер является **системным MCP самой утилиты**.

Он нужен для того, чтобы агент мог:

- отправить пользователю промежуточное сообщение во время работы;
- задать вопрос;
- запросить обязательный ответ;
- предложить готовые варианты ответа.

### 24.1. Включение

В agent file можно указать, что агенту разрешено общение с пользователем через встроенный MCP.

Если это включено, Core делает встроенный MCP-сервер доступным активной ноде.

### 24.2. Что считается сообщением через встроенный MCP

Сообщение через встроенный MCP — это не финальный ответ агента. Это промежуточное сообщение внутри текущего run.

Оно отображается в том же окне, где пользователь пишет обычные сообщения и получает финальный ответ, но оформляется как **нефинальное системное сообщение агента**.

### 24.3. Варианты поведения

Сообщение через встроенный MCP может быть:

- без ожидания ответа;
- с обязательным ожиданием ответа.

Если `require_response = true`, активная нода ждет ответ пользователя.

Если `require_response = false`, активная нода не ждет ответ и продолжает выполнение.

### 24.4. Варианты формы

Сообщение через встроенный MCP может содержать:

- обычный текст;
- JSON-структуру с готовыми вариантами ответа.

---

## 25. Правило маршрутизации пользовательских сообщений во время run

Если одновременно включены:

- комментарии во время работы;
- встроенный MCP-канал;

то действует следующее правило:

1. Если у run есть активный ожидающий MCP-вопрос и пользователь отвечает именно на него через явное действие интерфейса, ответ идет в MCP-канал.
2. Любое другое пользовательское сообщение во время run считается комментарием.

Следовательно, свободный пользовательский текст не должен автоматически считаться MCP-ответом без явного связывания с конкретным MCP-вопросом.

---

## 26. Отображение сообщений во время работы

Промежуточные сообщения во время работы агента должны отображаться так, чтобы не засорять окно.

Обязательные требования к интерфейсу:

- такие сообщения должны визуально отличаться от финального ответа;
- они должны отображаться компактно;
- они не должны маскироваться под финальный output ноды;
- они должны быть доступны в том же окне, где происходит основное общение.

---

## 27. Права

Права относятся к runtime-вызову агента, а не к plugins.

Plugins в этой модели не имеют отдельной permission-модели.

У системы должны быть инструменты ограничения прав, но проект не должен по умолчанию строиться на предположении, что пользователя надо защищать от его осознанных действий.

Базовый принцип:

- утилита дает инструменты самоограничения;
- пользователь сам решает, использовать их или нет.

---

## 28. События и триггеры

Триггеры не входят в agent file.

Событие — это отдельная сущность, которая связывает:

- trigger;
- agent reference;
- optional payload;
- optional стартовый prompt.

Базовое правило запуска события:

- событие вызывает один запуск;
- если запуск не удался, автоматический повтор не выполняется.

Никакой базовой системы auto-retry для triggers в этой версии модели нет.

---

## 29. Что не входит в базовую архитектуру

В базовую модель проекта сознательно **не входят**:

- глобальная очередь задач;
- система приоритетов;
- scheduler;
- worker-архитектура как обязательная основа проекта;
- автоматические ретраи run-ов;
- авто-перезапуск неудачных trigger-run-ов;
- собственный внутренний контракт skill/plugin/MCP вместо Codex-совместимого;
- хранение внутренней полной истории runtime;
- система пользовательских тест-кейсов как обязательная часть продукта;
- разделение на «мозги» и «мышцы»;
- сложное artifact-хранилище как обязательная часть ядра.

Если что-то из этого понадобится позже, оно должно появляться как отдельное расширение модели, а не задним числом превращать это в базовую философию проекта.

---

## 30. Версионирование и совместимость

### 30.1. `graph_contract_version`

Каждый agent file обязан иметь поле `graph_contract_version`.

Это поле:

- не является версией самого агента;
- не является версией утилиты;
- не является пользовательской меткой релиза.

Оно означает только одно:

**какая версия логики обработки графа требуется для корректного запуска этого agent file.**

### 30.2. Назначение

Это нужно затем, чтобы:

- agent files, созданные в новых версиях утилиты, могли использовать новые возможности;
- старая версия утилиты могла корректно отказать в запуске, если она не умеет правильно интерпретировать такой graph contract;
- ошибка несовместимости была явной и контролируемой.

### 30.3. Поведение при несовместимости

Если Core не поддерживает `graph_contract_version`, указанный в agent file, он обязан:

- не запускать агент;
- вернуть явную ошибку несовместимости формата;
- не пытаться частично «догадаться», как обработать неизвестные поля.

---

## 31. Runtime adapter

Любой runtime adapter должен уметь:

- запустить ноду;
- принять prompt и input message в канонической форме этой утилиты;
- передать в runtime skills, MCP, plugins и permissions;
- вернуть финальный output в режиме `text` или `json`;
- сообщить, поддерживает ли runtime native resume;
- использовать нативный механизм live-комментариев, если он существует;
- использовать встроенный MCP-канал утилиты, если runtime поддерживает MCP.

Runtime adapter не обязан раскрывать внутреннюю инструментальную историю runtime.

---

## 32. Исходы выполнения ноды

На уровне оркестратора различаются следующие исходы:

- `success`
- `invalid_output`
- `runtime_error`
- `interrupted`
- `cancelled`

### 32.1. `success`

Нода завершилась корректно и вернула валидный output.

### 32.2. `invalid_output`

Runtime завершил вызов, но output не соответствует ожидаемому режиму ноды.

### 32.3. `runtime_error`

Внешний runtime не смог корректно завершить вызов.

### 32.4. `interrupted`

Выполнение было прервано внешним завершением Core или интерфейса.

### 32.5. `cancelled`

Выполнение было явно отменено пользователем.

---

## 33. Политика secret markers

Секреты не являются частью базового результата ноды.

Поля `secrets` в базовом контракте output отсутствуют.

Допускается отдельный **опциональный** механизм secret markers.

Если он включен в agent file или chat policy:

- агенту сообщается специальный текстовый способ маркировать чувствительные фрагменты ответа;
- при сохранении такие фрагменты вырезаются;
- шифруются;
- сохраняются отдельно от основного текста;
- при explicit resume могут быть восстановлены.

По умолчанию этот механизм выключен, потому что требует дополнительной инструкции агенту и неявно меняет его поведение.

## 34. Канонический контракт agent JSON

Ниже описан фиксированный high-level контракт agent file.

### 34.1. Общие правила

1. Все обязательные поля действительно обязательны.
2. Неописанные top-level поля считаются недопустимыми.
3. Pass-through runtime-специфичные поля допускаются только в специально разрешенных местах.
4. Все идентификаторы внутри agent file должны быть уникальны в своей области видимости.

## 35. Top-level структура agent JSON

Каноническая структура agent JSON:

```json
{
  "graph_contract_version": "1.0",
  "meta": {},
  "entry_node_id": "start",
  "params": {},
  "initial_vars": {},
  "skills": [],
  "mcps": [],
  "plugins": [],
  "permissions": {},
  "interaction": {},
  "chat": {},
  "final_output": {},
  "nodes": [],
  "edges": []
}
```

### 35.1. `graph_contract_version`

- Тип: `string`
- Обязательное: да
- Назначение: версия логики обработки графа

### 35.2. `meta`

- Тип: `object`
- Обязательное: да
- Назначение: метаданные агента

Поля `meta`:

- `id`
  - Тип: `string`
  - Обязательное: да
  - Назначение: уникальный идентификатор агента
- `name`
  - Тип: `string`
  - Обязательное: да
  - Назначение: человекочитаемое имя
- `description`
  - Тип: `string`
  - Обязательное: нет
  - Назначение: описание агента
- `agent_version`
  - Тип: `string`
  - Обязательное: нет
  - Назначение: логическая версия самого агента, независимая от `graph_contract_version`

### 35.3. `entry_node_id`

- Тип: `string`
- Обязательное: да
- Назначение: id стартовой ноды

### 35.4. `params`

- Тип: `object`
- Обязательное: нет
- Назначение: определение параметров агента

Каждое поле внутри `params` — это имя параметра.

Значение параметра имеет вид:

```json
{
  "type": "string",
  "required": true,
  "default": null,
  "description": "Описание параметра",
  "mutable_in_ui": true
}
```

Поля описания параметра:

- `type`
  - Тип: `string`
  - Обязательное: да
  - Допустимые значения: `string`, `number`, `boolean`, `object`, `array`, `null`
- `required`
  - Тип: `boolean`
  - Обязательное: да
- `default`
  - Тип: любой валидный JSON literal
  - Обязательное: нет
- `description`
  - Тип: `string`
  - Обязательное: нет
- `mutable_in_ui`
  - Тип: `boolean`
  - Обязательное: нет
  - Значение по умолчанию: `true`

### 35.5. `initial_vars`

- Тип: `object`
- Обязательное: нет
- Назначение: начальные значения глобальных runtime-переменных

### 35.6. `skills`

- Тип: `array`
- Обязательное: нет
- Назначение: список skill bindings, доступных агенту

Элемент `skills[]` имеет вид:

```json
{
  "id": "planner_skill",
  "codex_ref": "planner_skill",
  "inline_text": "...",
  "frozen": false
}
```

Поля:

- `id`
  - Тип: `string`
  - Обязательное: да
  - Назначение: локальный id skill внутри agent file
- `codex_ref`
  - Тип: `string`
  - Обязательное: нет
  - Назначение: ссылка на существующий Codex-совместимый skill
- `inline_text`
  - Тип: `string`
  - Обязательное: нет
  - Назначение: встроенный текст skill в Codex-совместимом виде
- `frozen`
  - Тип: `boolean`
  - Обязательное: нет
  - Значение по умолчанию: `false`
  - Назначение: если `true`, встроенный текст имеет приоритет перед локальной одноименной сущностью

Правило:

- хотя бы одно из полей `codex_ref` или `inline_text` должно быть задано.

### 35.7. `mcps`

- Тип: `array`
- Обязательное: нет
- Назначение: список MCP bindings, доступных агенту

Элемент `mcps[]` имеет вид:

```json
{
  "id": "filesystem",
  "codex_ref": "filesystem",
  "config": {}
}
```

Поля:

- `id`
  - Тип: `string`
  - Обязательное: да
- `codex_ref`
  - Тип: `string`
  - Обязательное: да
- `config`
  - Тип: `object`
  - Обязательное: нет
  - Назначение: pass-through конфигурация для runtime

### 35.8. `plugins`

- Тип: `array`
- Обязательное: нет
- Назначение: список plugin bindings, доступных агенту

Элемент `plugins[]` имеет вид:

```json
{
  "id": "review_pack",
  "codex_ref": "review_pack",
  "config": {}
}
```

Поля:

- `id`
  - Тип: `string`
  - Обязательное: да
- `codex_ref`
  - Тип: `string`
  - Обязательное: да
- `config`
  - Тип: `object`
  - Обязательное: нет
  - Назначение: pass-through конфигурация для runtime

### 35.9. `permissions`

- Тип: `object`
- Обязательное: нет
- Назначение: глобальный permission profile агента

Поля:

- `profile`
  - Тип: `string`
  - Обязательное: нет
  - Назначение: имя runtime-native профиля прав
- `allow`
  - Тип: `array[string]`
  - Обязательное: нет
  - Назначение: явные разрешения
- `deny`
  - Тип: `array[string]`
  - Обязательное: нет
  - Назначение: явные запреты
- `extra`
  - Тип: `object`
  - Обязательное: нет
  - Назначение: pass-through данные для runtime adapter

Если `permissions` отсутствует, используется runtime default.

### 35.10. `interaction`

- Тип: `object`
- Обязательное: нет
- Назначение: правила общения с пользователем во время run

Поля `interaction`:

#### `comments`

```json
{
  "enabled": false,
  "target_node_ids": []
}
```

- `enabled`
  - Тип: `boolean`
  - Обязательное: да
- `target_node_ids`
  - Тип: `array[string]`
  - Обязательное: да, если `enabled = true`
  - Назначение: список нод, которым можно прокидывать пользовательские комментарии во время работы

#### `user_mcp`

```json
{
  "enabled": false,
  "server_name": "orchestrator.user_chat"
}
```

- `enabled`
  - Тип: `boolean`
  - Обязательное: да
- `server_name`
  - Тип: `string`
  - Обязательное: нет
  - Значение по умолчанию: `orchestrator.user_chat`
  - Назначение: имя встроенного MCP-сервера утилиты

### 35.11. `chat`

- Тип: `object`
- Обязательное: нет
- Назначение: правила хранения чата и resume

Поля `chat`:

```json
{
  "prefer_native_resume": true,
  "store_visible_messages": true,
  "store_context_window": true,
  "allow_fresh_start": true,
  "secret_markers": {
    "enabled": false,
    "open_marker": "[[SECRET]]",
    "close_marker": "[[/SECRET]]"
  }
}
```

Поля:

- `prefer_native_resume`
  - Тип: `boolean`
  - Обязательное: нет
  - Значение по умолчанию: `true`
- `store_visible_messages`
  - Тип: `boolean`
  - Обязательное: нет
  - Значение по умолчанию: `true`
- `store_context_window`
  - Тип: `boolean`
  - Обязательное: нет
  - Значение по умолчанию: `true`
- `allow_fresh_start`
  - Тип: `boolean`
  - Обязательное: нет
  - Значение по умолчанию: `true`
- `secret_markers`
  - Тип: `object`
  - Обязательное: нет

Поля `secret_markers`:

- `enabled`
  - Тип: `boolean`
  - Обязательное: да
- `open_marker`
  - Тип: `string`
  - Обязательное: нет
  - Значение по умолчанию: `[[SECRET]]`
- `close_marker`
  - Тип: `string`
  - Обязательное: нет
  - Значение по умолчанию: `[[/SECRET]]`

### 35.12. `final_output`

- Тип: `object`
- Обязательное: нет
- Назначение: правило финального ответа агента

Вид:

```json
{
  "mode": "last_node_output"
}
```

Поля:

- `mode`
  - Тип: `string`
  - Обязательное: да
  - Допустимые значения: `last_node_output`, `none`

### 35.13. `nodes`

- Тип: `array`
- Обязательное: да
- Назначение: список нод агента

Элемент `nodes[]` имеет общий вид:

```json
{
  "id": "plan",
  "title": "Планирование",
  "kind": "runtime_agent",
  "runtime_adapter": "codex",
  "agent_ref": null,
  "prompt": "...",
  "input": {
    "parts": []
  },
  "skill_ids": [],
  "mcp_ids": [],
  "plugin_ids": [],
  "permissions": {},
  "runtime_options": {},
  "output": {
    "mode": "text"
  }
}
```

Поля:

- `id`
  - Тип: `string`
  - Обязательное: да
  - Назначение: уникальный id ноды
- `title`
  - Тип: `string`
  - Обязательное: нет
  - Назначение: человекочитаемое имя ноды
- `kind`
  - Тип: `string`
  - Обязательное: да
  - Допустимые значения: `runtime_agent`, `orchestrator_agent`
- `runtime_adapter`
  - Тип: `string`
  - Обязательное: да для `runtime_agent`, нет для `orchestrator_agent`
- `agent_ref`
  - Тип: `string`
  - Обязательное: да для `orchestrator_agent`, нет для `runtime_agent`
  - Назначение: ссылка на другой orchestrator-agent
- `prompt`
  - Тип: `string`
  - Обязательное: да для `runtime_agent`, запрещено для `orchestrator_agent`
- `input`
  - Тип: `object`
  - Обязательное: да
- `skill_ids`
  - Тип: `array[string]`
  - Обязательное: нет
  - Допустимо только для `runtime_agent`
- `mcp_ids`
  - Тип: `array[string]`
  - Обязательное: нет
  - Допустимо только для `runtime_agent`
- `plugin_ids`
  - Тип: `array[string]`
  - Обязательное: нет
  - Допустимо только для `runtime_agent`
- `permissions`
  - Тип: `object`
  - Обязательное: нет
  - Допустимо только для `runtime_agent`
  - Назначение: override глобального permission profile
- `runtime_options`
  - Тип: `object`
  - Обязательное: нет
  - Допустимо только для `runtime_agent`
  - Назначение: pass-through runtime-specific options
- `output`
  - Type: `object`
  - Required: yes
  - Purpose: explicit node output contract
- `output.mode`
  - Type: `string`
  - Required: yes
  - Allowed values: `text`, `json`
- `output.schema`
  - Forbidden when `output.mode = text`
  - Required when `output.mode = json`
  - Must be a JSON Schema 2020-12 object whose root describes a JSON object

Если поле `permissions` у ноды отсутствует, используется top-level `permissions`.

### 35.14. `input`

Поле `input` имеет вид:

```json
{
  "parts": [
    {"type": "text", "text": "Задача:\n"},
    {"type": "ref", "ref": "params.task"}
  ]
}
```

Поля:

- `parts`
  - Тип: `array`
  - Обязательное: да
  - Назначение: список частей входного сообщения

Элемент `parts[]` может быть только одного из двух видов.

#### Вариант 1: текст

```json
{
  "type": "text",
  "text": "..."
}
```

- `type` = `text`
- `text`
  - Тип: `string`
  - Обязательное: да

#### Вариант 2: ссылка

```json
{
  "type": "ref",
  "ref": "vars.plan"
}
```

- `type` = `ref`
- `ref`
  - Тип: `string`
  - Обязательное: да
  - Допустимые формы:
    - `params.<name>`
    - `vars.<name>`
    - `node.<node_id>.text`
    - `node.<node_id>.json.<path>`
    - `event.<path>`

### 35.15. `edges`

- Тип: `array`
- Обязательное: нет
- Назначение: список переходов между нодами

Элемент `edges[]` имеет вид:

```json
{
  "from": "plan",
  "to": "review",
  "condition": {
    "code": "vars.get('need_review', False)"
  }
}
```

Поля:

- `from`
  - Тип: `string`
  - Обязательное: да
- `to`
  - Тип: `string`
  - Обязательное: да
- `condition`
  - Тип: `object`
  - Обязательное: нет

Поля `condition`:

- `code`
  - Тип: `string`
  - Обязательное: да
  - Назначение: Python-код или Python-выражение условия

Правила условий:

1. Условие вычисляется в контексте:
   - `params`
   - `vars`
   - `node`
   - `event`
2. Значение последнего выражения считается результатом условия.
3. Outgoing edges одной ноды проверяются в порядке, в котором они записаны в `edges`.
4. Берется **первый** edge, условие которого вернуло `true`.
5. Если ни один edge не подошел, граф завершается.
6. Если `condition` отсутствует, edge считается безусловным.

## 36. Контракт встроенного MCP-сервера `orchestrator.user_chat`

Этот контракт определяется данной утилитой.

### 36.1. Сообщение от агента пользователю

Канонический payload:

```json
{
  "prompt_id": "optional-id",
  "kind": "text",
  "text": "Нужно ли продолжать?",
  "require_response": true,
  "options": [
    {"id": "yes", "label": "Да", "value": "yes"},
    {"id": "no", "label": "Нет", "value": "no"}
  ]
}
```

Поля:

- `prompt_id`
  - Тип: `string`
  - Обязательное: нет
  - Назначение: идентификатор вопроса
- `kind`
  - Тип: `string`
  - Обязательное: да
  - Допустимые значения: `text`, `options`
- `text`
  - Тип: `string`
  - Обязательное: да
- `require_response`
  - Тип: `boolean`
  - Обязательное: да
- `options`
  - Тип: `array`
  - Обязательное: нет
  - Используется, если агент хочет предложить готовые варианты ответа

Элемент `options[]`:

```json
{
  "id": "yes",
  "label": "Да",
  "value": "yes"
}
```

Поля:

- `id` — `string`, обязательно
- `label` — `string`, обязательно
- `value` — любой валидный JSON literal, обязательно

### 36.2. Ответ пользователя агенту

Канонический payload ответа:

```json
{
  "prompt_id": "optional-id",
  "kind": "text",
  "text": "Продолжай"
}
```

или

```json
{
  "prompt_id": "optional-id",
  "kind": "option",
  "option_id": "yes",
  "value": "yes"
}
```

Поля:

- `prompt_id`
  - Тип: `string`
  - Обязательное: нет
- `kind`
  - Тип: `string`
  - Обязательное: да
  - Допустимые значения: `text`, `option`
- `text`
  - Тип: `string`
  - Обязательное: да для `kind = text`
- `option_id`
  - Тип: `string`
  - Обязательное: да для `kind = option`
- `value`
  - Обязательное: нет для `kind = text`, да для `kind = option`

## 39. Нерушимые законы проекта

1. Агент хранится как JSON-файл и является переносимым артефактом.
2. Оркестратор управляет только границей вызова агента, а не его внутренней кухней.
3. Контракты skills, MCP и plugins определяются Codex-совместимыми runtime-средами, а не этой утилитой.
4. MCP — корректный термин; собственной сущности SDK в модели агента нет.
5. Связи отвечают за порядок исполнения, а не за передачу данных.
6. Вход ноды строится из JSON-частей, где смешиваются текст и ссылки на переменные.
7. Output ноды — это только финальный output агента.
8. Если output ноды имеет режим `json`, Core не меняет prompt автоматически.
9. Переменные с одинаковыми именами могут перезаписываться; побеждает последнее значение.
10. Чаты хранят только видимую историю и данные, нужные для resume.
11. Resume всегда явный.
12. Триггеры живут вне agent file и делают один запуск без авто-ретрая.
13. Пользователь может ограничивать себя, но система не обязана навязывать ограничения по умолчанию.
14. Встроенный MCP-сервер общения с пользователем является системным MCP этой утилиты.
15. `graph_contract_version` обязателен и независим от версии агента и утилиты.

## 41. Встроенный агент-конструктор

В системе должен существовать встроенный системный агент-конструктор.

Этот агент:

- предназначен для создания новых agent files;
- может изменять существующие agent files;
- может запускать созданных агентов для проверки;
- может сам быть вызван как агент из интерфейса;
- может сам быть нодой внутри другого orchestrator-agent.

Агент-конструктор не является отдельной магической подсистемой вне общей модели проекта.

Он остается агентом той же природы, что и остальные агенты, но имеет особую системную роль и может получать расширенные возможности там, где это явно разрешено пользователем или системной конфигурацией.

### 41.1. Назначение агента-конструктора

Агент-конструктор нужен затем, чтобы пользователь мог создавать, изменять и улучшать агентов не только вручную через редактирование JSON, но и через обычное общение с агентом.

Результатом его работы являются:

- новые agent files;
- изменения в drafts;
- новые live-версии после deploy;
- вспомогательные решения по структуре графа.

### 41.2. Builder не заменяет ручное редактирование

Наличие встроенного агента-конструктора не отменяет того, что agent file остается обычным JSON-файлом, который пользователь может редактировать напрямую.

Builder — это системный способ ускорить создание и развитие агентов, а не единственный допустимый путь.

## 42. Подключаемая память как отдельная ось системы

Чат и resume не равны памяти.

Помимо chat/resume state, система должна поддерживать отдельные подключаемые memory sources.

Память в этой модели:

- не является обязательной;
- может отсутствовать вовсе;
- может быть одной;
- может быть множественной;
- может подключаться на уровне всего агента;
- может подключаться на уровне отдельной ноды.

Оркестратор не должен интерпретировать внутреннюю структуру такой памяти глубже необходимого минимума.

Он отвечает только за то, какие memory sources доступны агенту и конкретной ноде.

### 42.1. Каноническая роль памяти

Память нужна для того, чтобы агент или нода могли использовать долговременный внешний контекст, который не равен:

- параметрам агента;
- vars графа;
- истории чата;
- resume state.

### 42.2. Связь памяти с Codex-first моделью

Если конкретный runtime умеет работать с памятью нативно, оркестратор должен по возможности использовать его естественный путь.

Если память в проекте представляется через MCP или иной runtime-совместимый механизм, оркестратор должен лишь настраивать доступность такого memory source, а не придумывать собственный внутренний формат данных памяти.

### 42.3. Расширение канонического контракта agent JSON

Каноническая top-level структура agent JSON дополнительно поддерживает поле:

- `memory_bindings`
  - Тип: `array`
  - Обязательное: нет
  - Назначение: список подключаемых источников памяти, доступных агенту

Элемент `memory_bindings[]` имеет высокоуровневый вид:

```json
{
  "id": "project_memory",
  "kind": "runtime_memory",
  "codex_ref": "project_memory",
  "config": {},
  "scope": "agent"
}
```

Допустимые значения `scope`:

- `agent`
- `node`

Если `scope = node`, доступ к памяти должен задаваться на уровне конкретной ноды.

Также нода дополнительно поддерживает поле:

- `memory_ids`
  - Тип: `array[string]`
  - Обязательное: нет
  - Назначение: список memory bindings, доступных именно этой ноде

Если `memory_ids` не указано, действуют глобальные правила доступности памяти для агента.

## 43. Источники исполнения, аккаунты и лимиты

Проект должен поддерживать не только разные runtime adapters, но и разные реальные источники исполнения внутри одного и того же runtime.

Это означает, что пользователь может подключать:

- несколько аккаунтов;
- несколько API-ключей;
- несколько сессий;
- несколько подписочных профилей;
- разные источники моделей и исполнения внутри одного семейства runtime.

### 43.1. Что такое источник исполнения

Источник исполнения — это конкретная подключенная сущность, через которую runtime adapter реально запускает агента.

Примеры:

- конкретная Codex-сессия;
- конкретный API-ключ;
- конкретный аккаунт;
- конкретный подписочный профиль;
- иная runtime-совместимая точка запуска.

### 43.2. Что делает оркестратор

Оркестратор должен уметь:

- хранить информацию о доступных источниках исполнения;
- связывать ноды с допустимыми источниками;
- позволять пользователю выбирать и ограничивать, какие источники можно использовать;
- показывать лимиты и статус тех источников, которые реально доступны в системе и используются агентом.

### 43.3. Что оркестратор не делает

Оркестратор не должен заново реализовывать всю внутреннюю логику работы источника исполнения, если она уже существует внутри Codex App Server или аналогичного runtime.

### 43.4. Расширение канонического контракта agent JSON

Каноническая top-level структура agent JSON дополнительно поддерживает поле:

- `runtime_sources`
  - Тип: `array`
  - Обязательное: нет
  - Назначение: список доступных источников исполнения, на которые может ссылаться агент

Элемент `runtime_sources[]` имеет высокоуровневый вид:

```json
{
  "id": "primary_codex_session",
  "runtime_adapter": "codex",
  "source_ref": "codex_session_main",
  "description": "Основная рабочая сессия Codex"
}
```

Нода дополнительно поддерживает поля:

- `runtime_source_ids`
  - Тип: `array[string]`
  - Обязательное: нет
  - Назначение: список допустимых источников исполнения для этой ноды
- `runtime_source_policy`
  - Тип: `string`
  - Обязательное: нет
  - Допустимые значения: `inherit`, `restrict`, `prefer_first`

Значение по умолчанию: `inherit`.

### 43.5. Лимиты

Если runtime adapter умеет получать информацию о лимитах, квотах или доступности источника, эта информация должна быть видима пользователю через интерфейсы утилиты.

Если runtime такой информации не предоставляет, утилита не обязана имитировать ее искусственно.

## 44. Исследование файлов пользователя агентом-конструктором

Агент-конструктор может исследовать файлы пользователя, если это явно разрешено пользователем и его текущими правами.

Это нужно затем, чтобы builder мог:

- понимать задачу глубже, чем из одного короткого промта;
- анализировать существующие agent files;
- читать рабочие документы, примеры и контекст проекта;
- строить или изменять агентов с учетом реального окружения пользователя.

Это не означает, что builder автоматически получает безусловный доступ ко всем файлам.

Как и любой другой агент в этой системе, он получает только те права, которые реально были ему выданы.

## 45. Итеративная сборка и проверка кандидатов агентом-конструктором

Создание агента builder-агентом не должно мыслиться как одноразовая генерация одного JSON-файла.

Нормальная модель работы builder-agent выглядит так:

1. построить одного или нескольких кандидатов;
2. запустить их на проверку;
3. посмотреть результаты;
4. сравнить варианты;
5. внести изменения;
6. выбрать лучший итоговый вариант;
7. сохранить его как draft или deploy в live.

### 45.1. Что именно сравнивает builder

Builder может сравнивать кандидатов по:

- корректности структуры;
- успешности выполнения;
- качеству финального результата;
- удобству дальнейшего использования;
- соответствию задаче пользователя.

### 45.2. Это не пользовательская система тест-кейсов

Эта итеративная проверка не делает обязательной отдельную пользовательскую систему формальных test cases.

Это внутренняя инженерная способность builder-agent, а не отдельный обязательный публичный модуль продукта.

## 46. Реестр агентов как рабочая поверхность Core

Несмотря на то что agent file является источником истины для определения агента, Core должен поддерживать локальный реестр агентов как рабочую поверхность системы.

Реестр нужен затем, чтобы:

- получать список агентов;
- быстро открывать их из CLI и UI;
- связывать agent files с чатами, drafts и событиями;
- хранить служебные метаданные локальной системы;
- отличать известные Core агенты от просто лежащих на диске файлов.

### 46.1. Источник истины и индекс

Важно различать:

- **agent file** — источник истины описания агента;
- **agent registry** — локальный индекс и рабочая поверхность Core.

Реестр не должен переписывать смысл agent file и не должен становиться его заменой.

### 46.2. Минимальная роль реестра

Реестр должен хотя бы позволять:

- перечислять известных системе агентов;
- знать путь к их файлам;
- знать их локальный статус в Core;
- связывать их с drafts, чатами и событиями.

## 47. Черновики, live и deploy

В системе должен существовать явный жизненный цикл редактирования агента.

Минимально различаются:

- `draft` — редактируемый черновик;
- `live` — текущая рабочая версия агента, которую система считает основной;
- `deploy` — явное действие, переводящее выбранный draft в новую live-версию.

### 47.1. Зачем нужен draft/live/deploy

Это нужно затем, чтобы:

- пользователь не ломал рабочий агент каждым промежуточным изменением;
- builder-agent мог спокойно экспериментировать и улучшать граф;
- существовал понятный момент публикации новой рабочей версии;
- resume и события не зависели от случайного незавершенного редактирования.

### 47.2. Draft не обязан быть переносимым артефактом

Переносимым артефактом остается agent file как таковой.

Draft — это рабочее состояние редактирования внутри Core и локальной среды пользователя.

### 47.3. Live не отменяет `agent_version` и `graph_contract_version`

`live` — это не версия формата и не логическая версия агента.

Это только обозначение того, какая рабочая версия агента сейчас считается основной внутри локальной системы пользователя.

## 48. Атомарная запись agent files

Запись agent files должна выполняться атомарно.

Это означает, что сохранение не должно происходить как небезопасная прямая перезапись файла.

Минимально требуемая логика:

1. запись во временный файл в той же директории;
2. flush буфера;
3. fsync файла;
4. атомарная замена целевого файла;
5. при возможности — fsync директории.

### 48.1. Зачем это нужно

Это нужно затем, чтобы:

- не повреждать agent files при сбоях;
- не получать полу-записанные JSON-файлы;
- обеспечивать безопасную работу builder-agent и deploy-процесса.

### 48.2. Область применения

Этот закон относится как минимум к:

- live agent files;
- draft agent files;
- иным критически важным JSON-файлам, если такие появятся в рамках этой модели.

## 49. Рабочая версия агента и независимость разных осей версионирования

В проекте должны различаться несколько независимых осей версионирования.

### 49.1. `graph_contract_version`

Показывает, какая версия логики обработки графа нужна для корректного запуска agent file.

### 49.2. `meta.agent_version`

Показывает логическую версию самого агента как артефакта и идеи.

### 49.3. `live revision`

Показывает, какая рабочая версия агента сейчас считается основной внутри локальной системы пользователя.

### 49.4. Версия утилиты

Показывает версию самой программы и ее возможностей.

### 49.5. Почему они не должны смешиваться

Эти оси решают разные задачи:

- `graph_contract_version` — совместимость формата;
- `meta.agent_version` — логическая эволюция агента;
- `live revision` — текущее рабочее состояние в локальной системе;
- версия утилиты — возможности самого продукта.

Смешивать их в одно число или одно поле запрещено, потому что это делает дальнейшую эволюцию системы двусмысленной.
